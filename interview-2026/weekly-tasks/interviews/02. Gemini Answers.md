Handling a 10× traffic spike on a single shard without immediate re-sharding requires a, "move the workload off the hot node" approach, focusing on caching, read-write separation, and application-level load shedding to maintain the P99 SLA. [1]  
Here is the step-by-step strategy to handle a 10x hot shard: 
1. Immediate Read Offloading (Cache & Replicas) [2]  
The goal is to stop read traffic from hitting the hot primary node. 

• In-Memory Caching (Redis/Memcached): Move frequently accessed hot keys to a distributed cache. Ensure cache TTLs are short enough to maintain consistency but long enough to significantly reduce load. 
• Scale Read Replicas: Spin up multiple read replicas specifically for the overloaded shard and direct all read traffic to them. 
• Cache Aside Pattern: Update the application to check the cache before hitting the database. [3, 4, 5, 6, 7]  

2. Mitigate Write Overload 
If the hotspot is caused by writes (e.g., a "hot key" like a celebrity profile or a single active partition), you must reduce the immediate write volume. 

• Batch Processing: Instead of writing every request to the database immediately, aggregate writes in the application layer or a queue (e.g., Kafka) and write in bulk. 
• Write-Back Caching: Write the update to Redis and update the database asynchronously in the background. Note: This introduces the risk of data loss if the cache fails before persisting to the DB. [3, 10, 11, 12, 13]  

3. Application-Level Traffic Shaping 
Control the load before it hits the database. 

• Rate Limiting/Throttling: Implement rate limiting (token bucket or leaky bucket algorithm) specifically on API endpoints that query the hot shard. 
• Load Shedding: If the P99 exceeds the SLA, drop non-critical requests or serve cached data for slightly stale data to protect the system's overall health. 
• Circuit Breakers: Use tools like Resilience4j or Hystrix to stop calling the hot database if latency exceeds the SLA, preventing "retry storms". [3, 4, 16]  

4. Database-Level Optimization 

• Vertical Scaling: Immediately increase CPU and RAM for the hot shard to handle higher throughput. 
• Index Optimization: Analyze and remove unnecessary indexes that might be slowing down write operations on the hot shard. [4, 17]  

5. Tactical Data Reorganization

Under a spike, when short TTLs cause cache misses to hammer the database (a scenario known as a "cache stampede" or "thundering herd"), the best approach is to serve stale data (specifically, stale-while-revalidate). [1, 2]  
Serving slightly stale data is preferred because it maintains high performance and protects the backend database, while the data is refreshed asynchronously in the background. [1]  
Key Strategies to Fix This 

• Stale-While-Revalidate (Stale Data): Configure the cache to continue serving expired data for a short period while a single backend process refreshes the cache asynchronously. 
• Request Coalescing (Singleflight/Locking): When a cache miss occurs, only the first request is sent to the database to rebuild the cache, while subsequent requests wait for that result. 
• Add Jitter to TTLs (Randomness): If multiple keys expire at the same time, it causes a "cache avalanche." Adding random jitter (randomized expiry time) to the TTL ensures that cache items expire at different times. 
• Pre-fetching/Cache Warming: For predictable spikes (e.g., a scheduled marketing campaign), pre-load the cache with data before the spike occurs. [1, 2, 3, 4, 5, 6, 7]  

Why Other Options Are Less Ideal 

• Increasing TTL: This only delays the problem; the stampede will still happen, just later. 
• Blocking Requests: This results in high error rates or latency for users during the spike. 
• Violating Consistency: Serving stale data is a form of, but it is controlled and temporary (bounded staleness), rather than arbitrary inconsistency. [8, 9, 10, 11, 12]  

Summary 
The most robust solution is to serve stale data combined with request coalescing. This approach solves the immediate database load problem without sacrificing user experience. [1, 3]

When the database becomes the bottleneck due to a write-heavy spike (likes, counters, inventory updates), the primary goal is to decouple the incoming request from the database write and reduce the number of I/O operations. [1, 2]  
Here are the strategies to address write-heavy DB bottlenecks: 
1. Architectural Changes: Offload & Queue 

• Asynchronous Writes (Message Queues): Do not write to the DB synchronously. Instead, push writes to a message queue (Kafka, RabbitMQ, SQS) and use background workers to consume messages and update the database at a manageable rate. 
• Write-Back (Write-Behind) Caching: Use Redis/Memcached to store hot data (e.g., current inventory, like counts). Applications update the cache, and a background process asynchronously persists the data to the database. 
• CQRS (Command Query Responsibility Segregation): Separate the read database from the write database. Use a high-performance write store (like Cassandra or ScyllaDB) for ingestion, and asynchronous sync to your SQL database. [1, 2, 3, 4, 5]  

2. Database-Level Optimizations 

• Batch Updates/Bulk Inserts: Instead of updating a counter or inventory 100 times, batch 100 updates into a single transaction (e.g., ). This reduces lock contention significantly. 
• Reduce Indexing: Every index must be updated on every write. Temporarily drop non-essential indexes on the table being heavily updated. 
• Partitioning/Sharding: Partition data by a high-cardinality key (e.g.,  or ) to distribute write load across multiple database nodes.

3. Application-Level Mitigations 

• Rate Limiting & Throttling: If a specific product or user is causing a hotspot, use rate limiting to restrict the number of incoming writes for that specific key.

During a write spike with the database down, Kafka acts as a durable buffer, allowing producers to continue writing. While Kafka and Redis are active, consumers will encounter errors, resulting in high lag. The system must be designed to handle this backpressure, ensuring no data loss by delaying database writes until the DB recovers. [1, 2, 3, 4, 5]  
Immediate Actions & Impact 

• Kafka as a Buffer: The producer keeps writing to Kafka, allowing the application to avoid immediate downtime. 
• Consumer Lag: Consumers trying to write to the down database will fail, increasing consumer_lag. 
• Database Recovery: Once the database is back up, consumers will work through the backlog. 
• Risk: If the spike lasts longer than the disk space available on the Kafka brokers, the producers will fail. [1, 3, 6, 7]  

Key Recommendations 

1. Monitor Lag: Monitor the consumer lag and set alerts for when it exceeds a predefined threshold. 
2. Ensure Durability: Configure producers with acks=all and appropriate replication factors to ensure data is not lost while stored in Kafka. 
3. Implement Circuit Breakers: Use circuit breakers on consumers to prevent overwhelming the database once it comes back online. 
4. Increase Retention: Temporarily increase Kafka's message retention time if the database outage is expected to exceed the default retention period. [2, 7, 8, 9, 10]  

Potential Failures 

• Consumer Failures: If consumers do not handle the database downtime, they might crash. 
• Kafka Disk Full: A prolonged spike might fill up the Kafka logs. 
• Resource Exhaustion: Redis or the application might run out of memory or connections if not scaled for the spike. [2, 6, 11, 12]

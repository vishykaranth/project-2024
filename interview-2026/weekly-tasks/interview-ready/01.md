# 1️⃣ Cache Outage / Cache Stampede (Redis Down)

**Question:**
*At 30K TPS, Redis goes down for 90 seconds. What happens and how do you prevent a full outage?*

### Interview-Ready Script

> “At 30K TPS, a Redis outage behaves like a massive cache miss, pushing all traffic to the database and causing connection pool exhaustion and cascading failures.
>
> To prevent a full outage, I immediately shift the system from a high-performance mode to a functional-minimal mode.
>
> I use circuit breakers around Redis to fail fast, serve stale data using stale-while-revalidate, and rely on an L1 in-memory cache for hot keys.
> Database access is strictly rate-limited, and non-critical features are disabled.
>
> Once Redis recovers, I reopen traffic gradually to avoid a second stampede.
>
> The goal isn’t heroics — it’s graceful degradation while keeping the system alive.”

## **Signal:** resilience, incident thinking, maturity

# 2️⃣ Hot Shard / 10× Traffic Spike on a Single Shard

**Question:**
*One shard gets 10× traffic. You can’t reshard immediately. What do you do?*

### Interview-Ready Script

> “When a shard becomes hot, the priority is to move load off the primary without changing data boundaries.
>
> I immediately offload reads using aggressive caching and read replicas, often adding an L1 cache to eliminate network calls for the hottest keys.
>
> If writes are the problem, I batch them or queue them to smooth the spike, and apply per-key rate limiting to prevent a single hotspot from taking down the system.
>
> As a tactical fix, I may split hot keys or vertically scale the shard, while planning long-term redistribution once traffic stabilizes.
>
> The key is protecting P99 latency without introducing emergency sharding.”

## **Signal:** pragmatic scalability, ops realism

# 3️⃣ Cache TTLs Causing DB Stampede

**Question:**
*Short TTLs cause cache misses that hammer the DB. What’s the best fix?*

### Interview-Ready Script

> “This is a classic cache stampede. The correct fix is not increasing TTLs, but serving stale data safely.
>
> I implement stale-while-revalidate so expired entries are served briefly while one background refresh repopulates the cache.
>
> I also use request coalescing so only one request rebuilds a missing key, and add TTL jitter to avoid synchronized expirations.
>
> This protects the database, maintains low latency, and provides bounded, controlled staleness instead of unpredictable failures.”

## **Signal:** cache maturity, correctness tradeoffs

# 4️⃣ Write-Heavy Spike (Likes, Counters, Inventory)

**Question:**
*The database becomes the bottleneck during write-heavy spikes. What do you do?*

### Interview-Ready Script

> “For write-heavy spikes, the key principle is decoupling user requests from database writes.
>
> I move writes to a durable queue like Kafka and process them asynchronously in batches, dramatically reducing lock contention and I/O.
>
> All writes are idempotent to handle retries and replays safely.
>
> For soft-state data like likes or views, async propagation is fine.
> For hard-state data like inventory, I use reservation semantics instead of counters to preserve correctness.
>
> This protects the database while maintaining user-visible reliability.”

## **Signal:** correctness + scalability awareness

# 5️⃣ Exactly-Once Semantics Across Regions

**Question:**
*A region goes down during payment authorization. How do you guarantee exactly-once under 150ms P99?*

### Interview-Ready Script

> “Exactly-once semantics across regions requires avoiding synchronous cross-region locking.
>
> I rely on idempotent APIs, request IDs, and a distributed state machine backed by a strongly consistent geo-replicated database.
>
> Writes are committed locally within a quorum to stay under 150ms, while other regions replicate asynchronously.
>
> On retries or failover, the system checks transaction state and returns the existing result instead of re-executing.
>
> This guarantees exactly-once behavior without violating latency SLOs.”

## **Signal:** distributed systems depth

# 6️⃣ DB Down During Write Spike (Kafka Still Up) — **Bar Raiser Question**

**Question:**
*Kafka is up, Redis is up, DB is down for 5 minutes. Writes keep coming. What happens?*

### Interview-Ready Script

> “Kafka acts as a durable buffer, but it’s not infinite.
>
> Producers continue writing with `acks=all`, while consumers stop committing offsets due to DB failures, causing controlled lag growth.
>
> As lag or broker disk thresholds are crossed, producers apply backpressure or reject requests — we fail fast rather than corrupt the system.
>
> For hard writes, users receive clear rejections.
> For soft writes, we return 202 Accepted with delayed processing.
>
> Once the DB recovers, consumers replay the backlog gradually with rate limits to avoid a second outage.”

## **Signal:** principal-level ownership, operational realism

# 7️⃣ Architectural Philosophy (Microservices vs Simplicity)

**Question:**
*Why not shard or go microservices early?*

### Interview-Ready Script

> “I optimize for simplicity first. A modular monolith with clear boundaries is easier to scale and debug than premature distribution.
>
> I avoid major architectural refactors until data proves the current system is the bottleneck.
>
> Complexity is a tax — and distributed systems require matching operational maturity.
>
> My goal is to scale throughput, cost efficiency, and team velocity together, not just raw performance.”

## **Signal:** principal mindset, leadership maturity

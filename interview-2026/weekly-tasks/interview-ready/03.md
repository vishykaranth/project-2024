# 🧠 PRINCIPAL ENGINEER CHEAT SHEET

**(Systems, Scalability, Reliability, Leadership, Trade-offs)**

> **Prime Directive:**
> *Design systems that scale traffic, teams, failures, and time.*

---

## 0️⃣ Principal Mindset (Say This Implicitly)

* **Scale is multidimensional**: traffic, data, teams, cost, operations
* **Complexity is a liability**, not an achievement
* **Correctness beats cleverness**
* **Latency, availability, and cost are business decisions**
* **You design for failure first, growth second**

---

## 1️⃣ Requirements Before Architecture (Always Start Here)

Ask these *out loud*:

| Dimension   | Question                               |
| ----------- | -------------------------------------- |
| Users       | DAU? growth curve? global vs regional? |
| Traffic     | steady vs spiky? fan-out patterns?     |
| Latency     | P95 / P99 targets?                     |
| Data        | read/write ratio? size growth?         |
| Consistency | strong vs eventual?                    |
| Failure     | acceptable blast radius?               |
| Team        | who owns this at 2AM?                  |

> ❌ Designing before this = mid-level mistake

---

## 2️⃣ Scalability First Principles

* **Vertical scaling buys time**
* **Horizontal scaling buys headroom**
* **Async buys survivability**
* **Caching buys cost + latency**
* **Sharding buys pain**

---

## 3️⃣ Performance vs Scalability vs Reliability

| Concept     | Meaning                            |
| ----------- | ---------------------------------- |
| Performance | Fast at current load               |
| Scalability | Predictable behavior under growth  |
| Reliability | Predictable behavior under failure |

> A fast system that collapses under load is **not scalable**

---

## 4️⃣ Scaling Levers (Use in This Order)

1. **Algorithm & query optimization**
2. **Caching (L1 → L2 → CDN)**
3. **Batching**
4. **Async / queues**
5. **Read replicas**
6. **Vertical scaling**
7. **Sharding**
8. **Multi-region active-active**

---

## 5️⃣ Caching Cheat Sheet

### Patterns

* Cache-aside (default)
* Read-through (simplicity)
* Write-through (strong consistency)
* Write-behind (soft state only)

### Failure Fixes

* Stale-while-revalidate
* Request coalescing (singleflight)
* TTL jitter
* L1 in-memory cache
* Circuit breakers

> 🧠 *Serve stale > serve errors*

---

## 6️⃣ Hot Keys / Hot Shards

**Immediate**

* Cache aggressively
* Rate-limit by key
* Add read replicas
* Vertical scale

**Tactical**

* Hot key splitting
* Batched writes
* Drop non-essential indexes

**Strategic**

* Reshard only with proof

---

## 7️⃣ Database Scaling

### Reads

* Indexing
* Read replicas
* Caching
* Materialized views

### Writes

* Batch
* Queue
* Partition
* Reduce indexes

### Sharding

* Last resort
* Requires:

  * stable shard key
  * resharding strategy
  * operational maturity

---

## 8️⃣ Write-Heavy Systems

### Golden Rule

> *Decouple user request from DB write*

### Tools

* Kafka / SQS
* Batch processors
* Write-behind cache (carefully)
* CQRS (when justified)

### Data Types

| Data              | Strategy                        |
| ----------------- | ------------------------------- |
| Likes / Views     | Async, eventual                 |
| Inventory / Money | Reservation, strong consistency |

---

## 9️⃣ Queues (Kafka Reality)

* Kafka smooths spikes — it doesn’t increase throughput
* At-least-once delivery
* Exactly-once = idempotency + state
* Lag is acceptable
* Disk exhaustion is not

### When DB Is Down

* Consumers stop
* Lag grows
* Producers apply backpressure
* Fail fast before meltdown

---

## 🔟 Exactly-Once Semantics

Never trust:

* retries
* consumers
* networks

Always use:

* idempotency keys
* state machines
* safe retries
* deduplication

> Exactly-once is **designed**, not configured

---

## 1️⃣1️⃣ Consistency Model Cheat Sheet

| Use Case  | Model    |
| --------- | -------- |
| Payments  | Strong   |
| Inventory | Strong   |
| Auth      | Strong   |
| Feeds     | Eventual |
| Likes     | Eventual |

> CAP theorem is **not optional**

---

## 1️⃣2️⃣ Multi-Region Systems

* Latency is physics
* Cross-region sync kills P99
* Use:

  * local quorum writes
  * async replication
  * region-local reads

**Active-active** = resilience, not performance

---

## 1️⃣3️⃣ Microservices vs Monolith

### Start With

* Modular monolith
* Clear boundaries
* Strong tests

### Move When

* Team scaling breaks velocity
* Ownership boundaries stabilize
* Operational maturity exists

> Premature microservices = distributed monolith

---

## 1️⃣4️⃣ Observability = Scalability

Must have:

* P95 / P99 latency
* Error budgets
* Saturation metrics
* Queue lag
* Per-shard visibility

If you can’t see it, you can’t scale it

---

## 1️⃣5️⃣ Reliability Patterns

* Circuit breakers
* Timeouts
* Retries (bounded)
* Bulkheads
* Load shedding
* Graceful degradation

> Systems fail — cascades are optional

---

## 1️⃣6️⃣ Cost Awareness (Often Missed)

* Cost per request must go **down** at scale
* Caching saves DB + money
* Over-sharding explodes infra costs
* Latency optimizations often reduce cost

---

## 1️⃣7️⃣ Incident Response Philosophy

* Protect core state
* Fail fast
* Degrade gracefully
* Recover slowly
* Avoid retry storms

---

## 1️⃣8️⃣ People Scaling (Principal-Level Signal)

* Optimize for debugging speed
* Optimize for onboarding
* Optimize for ownership
* Fewer moving parts > clever abstractions

> Systems that humans can’t operate do not scale

---

## 🎯 Universal Principal Closing Line

> “I optimize for systems that scale traffic, failures, cost, and teams — not just throughput.”

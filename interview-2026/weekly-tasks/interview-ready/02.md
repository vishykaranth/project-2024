# 🧠 Principal Engineer Scalability Cheat Sheet

> **Mantra:** *Scale systems, scale people, scale operations — not just throughput.*

---

## 1️⃣ First Principles (Say These Out Loud)

* **Scalability ≠ Performance**
  Performance is speed at one point in time. Scalability is behavior under growth and failure.

* **Complexity is a Tax**
  Every distributed component adds operational cost, debugging time, and failure modes.

* **Protect the Database at All Costs**
  Databases fail last and recover slowest — everything else exists to shield them.

---

## 2️⃣ The Principal Decision Framework

Ask **before** designing:

| Dimension      | Question                              |
| -------------- | ------------------------------------- |
| Business Phase | Growth vs velocity vs reliability?    |
| User Impact    | Who is affected at P99, not averages? |
| Correctness    | Strong vs eventual consistency?       |
| People         | Can the team operate this at 2AM?     |
| Cost           | Does cost/unit go **down** at scale?  |

> ❗ *If you can’t explain why complexity is justified, don’t add it.*

---

## 3️⃣ Scaling Levers (In Order of Preference)

1. **Caching** (L1 → L2 → CDN)
2. **Batching** (reduce I/O)
3. **Async / Queues** (time-shift load)
4. **Read Replicas**
5. **Vertical Scaling**
6. **Sharding** (last resort)
7. **Multi-Region Active-Active** (resilience, not throughput)

---

## 4️⃣ Cache Failure Playbook

### Problems

* Cache stampede
* Thundering herd
* Hot keys

### Solutions (in order)

* **Stale-While-Revalidate**
* **Request Coalescing (singleflight)**
* **TTL Jitter**
* **L1 In-memory Cache**
* **Circuit Breakers**

> 🧠 *Serving stale data is better than serving errors.*

---

## 5️⃣ Hot Shard / Hot Key Handling

**Immediate (No Resharding):**

* Cache hot keys aggressively
* Add read replicas
* Rate-limit by key
* Vertical scale the shard

**Tactical:**

* Hot key splitting
* Batched writes
* Drop non-essential indexes

**Strategic:**

* Re-shard only with proof

---

## 6️⃣ Write-Heavy Spike Strategy

### Golden Rule

> *Decouple user requests from DB writes.*

### Tools

* Kafka / SQS (durable queues)
* Batch writes
* Write-behind cache (soft state only)
* CQRS (when justified)

### Critical Distinction

| Data Type         | Strategy                         |
| ----------------- | -------------------------------- |
| Likes / Views     | Async, eventual                  |
| Inventory / Money | Reservations, strong consistency |

---

## 7️⃣ Kafka Is Not Infinite

### When DB Is Down

* Consumers stop committing offsets
* Lag grows (controlled)
* Producers apply backpressure
* Fail fast before disk exhaustion

### User Semantics

* **Hard writes:** reject (503/429)
* **Soft writes:** accept (202)

> ❗ *Backpressure is a feature, not a failure.*

---

## 8️⃣ Exactly-Once Semantics

You **never** rely on:

* Kafka exactly-once
* DB uniqueness alone

You **always** use:

* Idempotency keys
* State machines
* Safe retries
* De-duplication

> *Exactly-once is an illusion — you approximate it with discipline.*

---

## 9️⃣ Multi-Region Reality Check

| Requirement        | Solution                 |
| ------------------ | ------------------------ |
| <150ms latency     | Local quorum writes      |
| Global failover    | Active-active services   |
| Strong consistency | Spanner / Cockroach      |
| Availability       | AP for non-critical data |

> 🌍 *Latency is physics. Don’t fight it.*

---

## 🔟 Microservices vs Monolith

### Start With

* Modular monolith
* Clear boundaries
* Independent deployability (within reason)

### Move When

* Team size explodes
* Release velocity stalls
* Data boundaries are proven

> 🚫 *Premature microservices = distributed monolith.*

---

## 1️⃣1️⃣ Observability = Scalability

You cannot scale what you can’t see.

**Mandatory:**

* P95 / P99 latency
* Error budgets
* Queue lag
* Saturation metrics
* Per-shard visibility

---

## 1️⃣2️⃣ Incident Response Philosophy

* Degrade gracefully
* Fail fast
* Protect core state
* Recover slowly

> 🧠 *A system that degrades predictably is a scalable system.*

---

## 🎯 Interview Closing Line (Memorize This)

> “As a Principal Engineer, I don’t optimize for maximum throughput — I optimize for sustainable growth across technology, cost, and people.”

## Scalability 

- Scalability is the ability of a system to handle increased load by adding resources, 
    - while maintaining acceptable performance, reliability, and cost efficiency. 
    - For me, scalability is not just about throughput
        — it’s about predictable behavior under stress, graceful degradation, and operational simplicity.
- In Java systems, vertical scaling quickly runs into GC pause times, heap fragmentation, and diminishing returns beyond certain core counts. That’s why most modern Java platforms favor horizontal scaling—stateless services behind a load balancer, with state externalized to caches or data stores.
- Horizontal scaling is easy at low scale; it gets hard when you introduce stateful workflows, ordering guarantees, or exactly-once semantics.
- Stateless services scale linearly and simplify autoscaling. Any required state should be externalized—Redis for ephemeral state, databases for durable state.
- Caching is a correctness problem before it’s a performance problem—TTL strategy, cache invalidation, and staleness tolerance matter more than hit ratio.
- Asynchronous processing decouples request rate from processing rate. But without backpressure—timeouts, queues with limits, circuit breakers—you just move the bottleneck.

# Design Pattern Decision Tree – "Which Pattern Would You Use and Why?"

> **Interview intent**: This is not about naming patterns. It’s about *diagnosing the problem correctly* and *justifying trade-offs*. Use the questions verbatim in interviews.

---

## STEP 1: What is actually changing?

### Q1: Is the change primarily about **behavior / logic**?

* ✅ Yes → go to **Behavioral Patterns**
* ❌ No → go to Q2

### Q2: Is the change about **object creation or lifecycle**?

* ✅ Yes → go to **Creational Patterns**
* ❌ No → go to Q3

### Q3: Is the change about **structure / integration / composition**?

* ✅ Yes → go to **Structural Patterns**

> **Principal signal**: I first classify the axis of change before naming a pattern.

---

## BEHAVIORAL PATTERNS – Decision Path

### Q4: Does behavior vary based on **runtime choice or configuration**?

* ✅ Yes → **Strategy**

  * *Why*: Algorithms vary independently of callers
  * *Avoid when*: Too many tiny strategies with no real volatility

---

### Q5: Does behavior vary based on **lifecycle state**?

* ✅ Yes → **State**

  * *Why*: Transitions must be explicit and illegal states impossible
  * *Escalate to*: Workflow / FSM / Saga if distributed or long-running

---

### Q6: Do you need to **trigger multiple reactions** to one event?

* ✅ Yes → **Observer / Pub-Sub**

  * *Why*: Decouples producers from consumers
  * *Watch for*: Memory leaks, backpressure, event storms

---

### Q7: Do you need to **queue, retry, audit, or undo operations**?

* ✅ Yes → **Command**

  * *Why*: Makes actions first-class objects
  * *Often paired with*: Saga, Event Sourcing

---

### Q8: Is interaction logic becoming **too coupled or tangled**?

* ✅ Yes → **Mediator**

  * *Why*: Centralizes orchestration
  * *Trade-off*: Mediator can become a god object

---

### Q9: Do you need to **add operations frequently but object structure is stable**?

* ✅ Yes → **Visitor**

  * *Why*: Optimizes for new behavior, not new types
  * *Avoid when*: Object model is volatile

---

### Q10: Do you need **controlled rollback or snapshots**?

* ✅ Yes → **Memento**

  * *Why*: Preserves encapsulation while enabling restore
  * *Avoid when*: Persistence or event sourcing is a better fit

---

## CREATIONAL PATTERNS – Decision Path

### Q11: Do you need to **control how objects are created**?

* ✅ Yes → continue

---

### Q12: Does creation logic vary by **subtype or environment**?

* ✅ Yes → **Factory Method**

  * *Why*: Pushes creation volatility to subclasses

---

### Q13: Do you need **families of related objects** to be consistent?

* ✅ Yes → **Abstract Factory**

  * *Why*: Prevents incompatible combinations

---

### Q14: Is object construction **complex with many invariants**?

* ✅ Yes → **Builder**

  * *Why*: Separates construction from representation

---

### Q15: Is object creation **expensive and repetitive**?

* ✅ Yes → **Prototype** or **Object Pool**

  * *Prototype*: Clone configuration-heavy objects
  * *Pool*: Reuse scarce resources (with caution)

---

### Q16: Do you need **exactly one shared instance**?

* ⚠️ Rarely justified → **Singleton**

  * *Principal rule*: Treat as infrastructure, not business logic

---

## STRUCTURAL PATTERNS – Decision Path

### Q17: Are you integrating with **incompatible interfaces**?

* ✅ Yes → **Adapter**

  * *Why*: Shields core domain from external change

---

### Q18: Do you need a **simplified interface over complexity**?

* ✅ Yes → **Facade**

  * *Why*: Defines clean system boundaries

---

### Q19: Do you want to **add behavior dynamically** without inheritance?

* ✅ Yes → **Decorator**

  * *Why*: Composable behavior
  * *Watch for*: Debuggability issues

---

### Q20: Do you need to treat **individual objects and compositions uniformly**?

* ✅ Yes → **Composite**

  * *Why*: Simplifies tree-like structures

---

### Q21: Are abstraction and implementation changing along **independent dimensions**?

* ✅ Yes → **Bridge**

  * *Why*: Avoids inheritance matrix explosion

---

### Q22: Do you need to **control access, latency, or lifecycle**?

* ✅ Yes → **Proxy**

  * *Why*: Adds cross-cutting concerns transparently

---

### Q23: Is memory usage exploding due to **many similar objects**?

* ✅ Yes → **Flyweight**

  * *Why*: Shares immutable intrinsic state

---

## DISTRIBUTED / PRINCIPAL-LEVEL ESCALATIONS

### Q24: Is the problem **long-running and distributed**?

* ✅ Yes → **Saga / Workflow Engine**

### Q25: Do you need **auditability and replay**?

* ✅ Yes → **Event Sourcing**

### Q26: Are read and write concerns scaling differently?

* ✅ Yes → **CQRS**

---

## FINAL INTERVIEW CLOSER

> *I don’t start with patterns. I start with the axis of change, failure modes, and scale. The pattern is the consequence, not the goal.*

# Principal Engineer – Design Pattern Trap Questions & How to Avoid Them

> **Purpose**: These are not knowledge checks. They are *judgment checks*. Interviewers use them to detect over-engineering, pattern cargo-culting, and lack of scale awareness.

---

## TRAP CATEGORY 1: "Why not just use X?"

### Trap 1: "Why not just use an enum with switch statements?"

**What they’re testing**:

* Do you understand when *simplicity beats purity*?
* Can you articulate the tipping point?

**Bad answer**:

> Because State Pattern is cleaner and object-oriented.

**Principal answer**:

> Enums are fine when behavior is simple and stable. I switch to State when behavior diverges per lifecycle and illegal transitions must be prevented by design.

**Avoidance rule**:

> Always acknowledge the simpler alternative first.

---

### Trap 2: "Why not Strategy instead of State?"

**What they’re testing**:

* Do you understand intent, not just structure?

**Bad answer**:

> They’re similar but State is better here.

**Principal answer**:

> Strategy is chosen externally and remains stable; State transitions internally based on lifecycle. If transitions matter, it’s State.

**Avoidance rule**:

> Anchor on *who controls change*.

---

## TRAP CATEGORY 2: Over-Engineering

### Trap 3: "Isn’t this too many classes?"

**What they’re testing**:

* Can you resist pattern worship?

**Bad answer**:

> That’s the cost of clean design.

**Principal answer**:

> If states or strategies don’t have meaningful behavioral differences, I collapse them. Patterns are a tool, not a goal.

**Avoidance rule**:

> Explicitly talk about pruning and simplification.

---

### Trap 4: "What if there are 30+ states?"

**What they’re testing**:

* Can you manage state explosion?

**Principal answer**:

> That’s usually a smell. I separate lifecycle states from attributes, introduce hierarchical states, or move to a table-driven FSM or workflow.

**Avoidance rule**:

> Show escape hatches beyond GoF patterns.

---

## TRAP CATEGORY 3: Scale & Distribution

### Trap 5: "How does this work in a distributed system?"

**What they’re testing**:

* Do you think beyond a single JVM?

**Bad answer**:

> We’d just persist the state.

**Principal answer**:

> State becomes a contract. I add versioning, idempotent transitions, and optimistic locking, or coordinate via Saga for cross-service workflows.

**Avoidance rule**:

> Always mention failure, retries, and ordering.

---

### Trap 6: "What happens on partial failure?"

**What they’re testing**:

* Operational maturity

**Principal answer**:

> Transitions must be replayable and idempotent. If a transition can’t be retried safely, the design isn’t production-ready.

**Avoidance rule**:

> Treat retries as a first-class requirement.

---

## TRAP CATEGORY 4: Testing & Operability

### Trap 7: "How do you test this?"

**What they’re testing**:

* Can you test behavior, not implementation?

**Bad answer**:

> Unit tests for each class.

**Principal answer**:

> I test the transition matrix, invariants, and illegal transitions. The state graph is the test surface.

**Avoidance rule**:

> Talk about properties and invariants, not mocks.

---

### Trap 8: "How do you debug this in production?"

**What they’re testing**:

* Observability thinking

**Principal answer**:

> I log state transitions as events with correlation IDs and versions. If I can’t reconstruct history, I consider event sourcing.

**Avoidance rule**:

> Tie patterns to observability.

---

## TRAP CATEGORY 5: Wrong Pattern Choice

### Trap 9: "Why not just use a workflow engine?"

**What they’re testing**:

* Build vs buy judgment

**Principal answer**:

> For short-lived, in-process logic I prefer code. For long-running, human-in-the-loop, or cross-service flows, workflows win.

**Avoidance rule**:

> Show you know when code stops scaling organizationally.

---

### Trap 10: "When would you NOT use this pattern?"

**What they’re testing**:

* Intellectual honesty

**Bad answer**:

> It’s generally a good pattern.

**Principal answer**:

> When the cost of abstraction exceeds the volatility it manages, I don’t use it.

**Avoidance rule**:

> Every good answer includes a *non-use case*.

---

## TRAP CATEGORY 6: Meta-Traps (Very Senior)

### Trap 11: "Who decides which pattern to use on a team?"

**What they’re testing**:

* Leadership, not design

**Principal answer**:

> Patterns emerge from reviews and shared understanding. My role is to guide the decision, not mandate it.

---

### Trap 12: "What if the business changes its mind?"

**What they’re testing**:

* Change tolerance

**Principal answer**:

> I optimize for change in the most volatile dimension and keep the rest boring.

---

## FINAL PRINCIPAL CLOSER (MEMORIZE)

> *Patterns are hypotheses about change. If the change doesn’t happen, the pattern shouldn’t survive either.*

# Design Pattern Misapplication Horror Stories (Real‑World, Principal‑Level)

> **Interview gold**: These stories show judgment, scars, and restraint. Use them when interviewers ask *“Tell me about a time…”* or push back on your design.

---

## 1. State Pattern → State Explosion Meltdown

**Context**: Payments platform handling refunds, chargebacks, disputes, retries, manual ops.

**What went wrong**:

* Every business nuance became a new state
* 40+ states, hundreds of transitions
* Developers feared touching it

**Symptoms**:

* Bugs only reproducible in certain state paths
* Impossible to onboard new engineers
* Business couldn’t reason about flow

**Root cause**:

* Confused *lifecycle states* with *attributes and flags*

**Fix**:

* Reduced to 8 core lifecycle states
* Moved risk, compliance, and ops decisions to attributes
* Externalized transitions into a table‑driven FSM

**Principal lesson**:

> *If you’re modeling decisions instead of lifecycle, you’re already in trouble.*

---

## 2. Strategy Pattern → Configuration Hell

**Context**: Pricing engine with country, channel, partner‑specific logic

**What went wrong**:

* Hundreds of Strategy implementations
* Selection logic more complex than strategies themselves

**Symptoms**:

* New strategy broke existing behavior
* Impossible to know which strategy was active

**Root cause**:

* Strategy used for combinatorial rules

**Fix**:

* Replaced with a rule engine + data‑driven configuration
* Kept Strategy only for genuinely distinct algorithms

**Principal lesson**:

> *Strategy fails when the real problem is rules, not algorithms.*

---

## 3. Abstract Factory → Inheritance Matrix Explosion

**Context**: Multi‑cloud deployment (AWS, Azure, GCP) × environments × tenants

**What went wrong**:

* Factory per cloud, per environment
* Subclass explosion

**Symptoms**:

* Changes required edits in 10+ factories
* Subtle inconsistencies across clouds

**Root cause**:

* Tried to encode deployment policy in code

**Fix**:

* Switched to declarative configuration + composition
* Factories only for truly incompatible APIs

**Principal lesson**:

> *Abstract Factory collapses under multi‑dimensional variability.*

---

## 4. Singleton → Global Contention Time Bomb

**Context**: Central cache and metrics registry

**What went wrong**:

* Singleton held mutable state
* Lock contention under load

**Symptoms**:

* Latency spikes
* Non‑deterministic test failures

**Root cause**:

* Treated Singleton as a shortcut, not infrastructure

**Fix**:

* Made it immutable + injected dependencies
* Scoped lifecycle to process, not JVM

**Principal lesson**:

> *Singleton is a lifecycle decision, not a design convenience.*

---

## 5. Decorator → Debugging Nightmare

**Context**: Request processing pipeline (auth, rate‑limit, logging, tracing)

**What went wrong**:

* 8+ decorators stacked dynamically
* Order‑dependent behavior

**Symptoms**:

* Logs impossible to follow
* Production bugs only fixed by trial and error

**Root cause**:

* Decorator used where pipeline semantics mattered

**Fix**:

* Replaced with explicit chain / pipeline definition
* Named steps, fixed order

**Principal lesson**:

> *When order matters, explicit pipelines beat decorators.*

---

## 6. Observer → Event Storm & Memory Leaks

**Context**: In‑process event system for domain events

**What went wrong**:

* Listeners registered dynamically
* No lifecycle management

**Symptoms**:

* Memory leaks
* Cascading side effects

**Root cause**:

* Observer used without ownership or backpressure

**Fix**:

* Replaced with async messaging
* Explicit subscriptions and lifecycles

**Principal lesson**:

> *Events without ownership become noise.*

---

## 7. Command Pattern → Accidental Distributed System

**Context**: Commands persisted and replayed for retries

**What went wrong**:

* Commands weren’t idempotent
* Replay caused double charges

**Symptoms**:

* Data corruption
* Manual reconciliation

**Root cause**:

* Treated Command as transport instead of intent

**Fix**:

* Added idempotency keys
* Promoted commands to domain events

**Principal lesson**:

> *Replayability without idempotency is corruption.*

---

## 8. CQRS → Operational Overhead Collapse

**Context**: Medium‑scale CRUD system

**What went wrong**:

* Separate read/write models
* Eventual consistency surprised business

**Symptoms**:

* Stale reads complaints
* Complex ops for little gain

**Root cause**:

* CQRS chosen for elegance, not scale

**Fix**:

* Collapsed back to a single model
* Reintroduced CQRS only on hot paths

**Principal lesson**:

> *CQRS is a scaling tool, not a default architecture.*

---

## 9. Workflow Engine → Vendor Lock‑In Trap

**Context**: Business workflows encoded entirely in BPMN

**What went wrong**:

* Business logic scattered across diagrams
* Engineers couldn’t reason about behavior

**Symptoms**:

* Slow changes
* Knowledge siloed in tooling

**Root cause**:

* Used workflow engine as a programming language

**Fix**:

* Moved core logic back to code
* Used workflows only for orchestration

**Principal lesson**:

> *Workflows orchestrate; code decides.*

---

## FINAL PRINCIPAL TAKEAWAY (USE THIS VERBATIM)

> *Every pattern fails when it’s used to solve the wrong problem. Most failures aren’t technical — they’re conceptual mismatches.*

The State pattern organizes complex, state-dependent behavior into classes, making it easier to read and maintain. However, it does not implicitly provide atomicity for state transitions in a multi-threaded environment. [4]  
Here is why your assessment is correct, and how the State pattern is typically combined with other techniques to fix the race condition. 
Why You Are Correct: The "Check-Then-Act" Race 
If Thread A and Thread B both read the "Paid" state simultaneously, they can both proceed to execute their respective state transitions (e.g., Thread A changes to "Shipped," Thread B changes to "Refunded") before the other has finished. This causes a race condition. [1, 5]  
Why the State Pattern is Still Used 
The State pattern "moves the logic around" in a way that makes it easier to apply synchronization, even if it doesn't do the synchronization itself. [6]  
Instead of a giant  block where you need to lock the entire object, you can place synchronization (locks) inside specific  methods within each State class. [7]

To make the State pattern thread-safe, you must combine it with synchronization or atomic operations:

3. State Machine as a Service: Use a single-threaded executor or an Actor model to manage state, so only one thread can modify the State object at a time. [6, 8, 9, 10]

To enforce correct state transitions in a distributed, stateless environment where the database is the source of truth, you can employ several strategies: 
1. Optimistic Concurrency Control (OCC) 
This is a common and effective approach. It works by having each service request a specific version of a record, attempt to make a change, and then verify that the version hasn't changed before committing the update [1]. 

• Mechanism: Add a version number or a timestamp column to the state table in your database. 
• Process: 

	1. A service reads the current state and version (e.g., , ). 
	2. The service performs its logic to transition the state (e.g., to ). 
	3. The service attempts to update the record in the database using a conditional query:  [1]. 
	4. Enforcement: If the update affects zero rows, it means another service updated the record in the meantime, changing the version to something other than 1. The service must then retry the operation, starting from step 1 [1]. [6, 7, 8, 9, 10]  

2. Pessimistic Locking 
This strategy prevents conflicts by locking the record at the start of a transaction, ensuring no other service can read or write to it until the current operation is complete. 

• Mechanism: Use database-level locks, such as . 
• Process: 

	1. A service starts a database transaction. 
	2. It acquires a lock on the specific order record using a query like  [1]. 
	3. The service performs the state transition within the transaction. 
	4. The service commits the transaction, which automatically releases the lock. 
	5. Enforcement: Other services attempting to lock the same record will be forced to wait until the lock is released. [16, 17, 18, 19, 20]  

3. Conditional Writes (Check Constraints) 
This approach uses database constraints to enforce the validity of the target state based on the current state. 

• Mechanism: Define a  constraint in your database schema that explicitly allows only specific transitions (e.g., an order can only go from  to  if its current status is ) [1]. 
• Enforcement: If a service tries to perform an invalid state transition, the database will raise an error, preventing the update and maintaining data integrity. [25, 26, 27, 28, 29]  

4. Event Sourcing 
In this pattern, you don't store the current state directly, but rather a sequence of immutable events that led to the current state [1]. 

• Mechanism: The database stores a log of events (e.g., , , ). The current state is built by replaying these events in order. 
• Process: A service reads the events, determines the current state, adds a new event (e.g., ), and writes this new event to the database. 
• Enforcement: Database constraints can ensure that events are only appended in a valid sequence, and the immutable nature of events simplifies concurrency issues. [32, 33, 34, 35, 36]  

Summary of Best Practices 

| Strategy [37, 38, 39, 40, 41] | Pros | Cons  |
| --- | --- | --- |
| Optimistic Concurrency | High concurrency, scalable for high-traffic systems | Requires retry logic in application code  |
| Pessimistic Locking | Simpler logic, prevents race conditions effectively | Can reduce concurrency, potential for deadlocks  |
| Conditional Writes | Strong data integrity guaranteed at the DB level | Can make the database schema complex for many transitions  |

For most distributed systems, Optimistic Concurrency Control (using version numbers) offers a good balance of data integrity and performance, making it a widely recommended approach [1]. [42, 43, 44, 45]

However, the State pattern is rarely used for storing state; it is used to manage behavioral complexity that databases cannot handle. [3, 4]  
Here is why the State pattern persists, even in highly disciplined database environments: 
1. Behavior vs. Data (The "Active" Object) 
Your database dictates that an order cannot move from  to . That is data integrity.The State pattern dictates what happens when a  clicks "Edit" while in the  state.

• With Pattern: The  object implements  to throw an exception, while  allows it. The object behaves differently based on its state. [3, 8, 9]  

2. Encapsulation of Transitions (Not Just Checks) 
Database constraints are usually passive guards (blocking invalid actions). The State pattern actively manages the workflow transition and related side effects, such as: 

• Firing an email when a state changes to . 
• Updating a cache. 
• Triggering an downstream event in an audit log. [13, 14, 15]  

3. Testing Complex Logic 
Testing a  statement with 10 states and 20 transitions is nightmare-inducing. The State pattern allows you to write unit tests for each state class in isolation. You can test  to ensure it only moves to  or , without mocking the entire application state. [3, 14, 16]  
4. Avoiding "Fat" Services (Single Responsibility) [17]  
Without the State pattern, your  becomes a dumping ground for state-dependent logic, leading to "if-stuffed" code. The pattern encapsulates state-specific behavior, keeping the context class slim. [9, 13]

## 🎯 90-Second Executive Answer

**URL Shortener under Extreme Load**

> **Frame (10–15s)**
> “When a celebrity tweets a link, the problem isn’t raw scale — it’s *skew*. In distributed systems, one hot key at 10 million QPS is far more dangerous than evenly distributed traffic. I design for two distinct failure modes: **hot keys** and **high-cardinality random traffic**.”

---

### 🔥 Hot Key (Celebrity Traffic) — *Performance Risk* (30s)

“All requests for a single short URL can melt one cache shard or trigger a cache stampede if it expires.

I solve this by pushing resolution to the **edge**:

* CDN caching with **stale-while-revalidate**, so millions of users get instant redirects while only one request revalidates the backend
* A short-lived **in-process L1 cache** so app instances don’t hammer Redis
* Optional Redis key replication if needed

Result: hot traffic never reaches the database, and latency stays flat.”

---

### 💣 High-Cardinality Traffic — *Cost Risk* (30s)

“With Lambda and DynamoDB On-Demand, nothing crashes — you just get a massive bill.

So I block bad traffic **before** it becomes compute:

* A **Bloom filter at the edge** to instantly reject non-existent keys
* **AWS WAF** rate rules for IPs requesting thousands of unique URLs
* **Provisioned DynamoDB with ceilings** so spend is capped, even if we return controlled 503s

Result: the system degrades safely instead of bankrupting us.”

---

### 🧠 Close (10–15s)

> “In short:
> **Hot keys are a performance problem solved with edge caching.
> Random keys are a cost problem solved with negative filtering and hard limits.**
> I design for both from day one.”

# 🔗 URL Shortener — System Design (Principal Level)

---

## 1️⃣ Clarify Requirements (30–60 seconds)

### Functional

* Shorten a long URL → short URL
* Redirect short URL → original URL
* Custom aliases (optional)
* Expiration (optional)
* Analytics (click count, geo, referrer)

### Non-Functional

* **Low latency redirects** (<50 ms)
* **High read traffic** (read-heavy: ~100:1)
* **High availability**
* **Globally accessible**
* Strong consistency **not required** for analytics

📌 **Principal signal:** *Redirect path must be faster than write path.*

---

## 2️⃣ High-Level Architecture

```
Client
  |
CloudFront (CDN)
  |
API Gateway / ALB
  |
Redirect Service (Stateless)
  |
Cache (Redis / DAX)
  |
Database (DynamoDB / Cassandra)
```

Separate:

* **Write path** (Create short URL)
* **Read path** (Redirect — hot path)

---

## 3️⃣ Core Design Decisions (This is where interviews are won)

---

## 3.1 Short Code Generation

### Option A — Random Base62 (Most common)

```
Base62 = [a-zA-Z0-9]
Length = 7 → 62^7 ≈ 3.5 trillion URLs
```

**Flow**

1. Generate random string
2. Check collision
3. Retry if exists (very low probability)

✅ Simple
❌ Collision checks required

---

### Option B — Snowflake / ID + Base62 (Preferred at scale)

```
ID (64-bit) → Base62 encode
```

**Why this is better**

* No collisions
* Monotonic IDs (good for DB partitioning)
* Faster writes

📌 **Principal callout:** *Avoid UUID — too long, bad cache locality.*

---

## 4️⃣ Database Design

### Table: `short_urls`

| Field           | Purpose         |
| --------------- | --------------- |
| short_code (PK) | Lookup key      |
| long_url        | Redirect target |
| created_at      | TTL / analytics |
| expiry          | Optional        |
| user_id         | Optional        |

### DB Choice

| Option    | When                               |
| --------- | ---------------------------------- |
| DynamoDB  | Massive scale, predictable latency |
| Cassandra | Multi-region writes                |
| RDS       | ❌ Avoid (read-heavy, scale pain)   |

📌 **Partition key = short_code**
Uniform distribution → no hot partitions.

---

## 5️⃣ Redirect Flow (🔥 Critical Path)

### Step-by-step

1. Client hits `https://sho.rt/abc123`
2. CloudFront checks edge cache
3. Cache miss → Redirect Service
4. Redis/DAX lookup
5. DB lookup (only if cache miss)
6. HTTP **301/302 redirect**

### Latency Budget

* Edge cache hit → ~5–10 ms
* Cache hit → ~15 ms
* DB hit → ~30–40 ms

📌 **Principal signal:** *Redirect path has zero business logic.*

---

## 6️⃣ Caching Strategy (This is mandatory)

### Multi-level cache

1. **CDN** — hottest URLs
2. **Redis / DAX** — warm URLs
3. **DB** — cold storage

### Cache Key

```
short_code → long_url
```

### TTL Strategy

* Long TTL (URLs rarely change)
* Invalidate on delete/expiry only

📌 *Never cache negative lookups aggressively — prevents recovery.*

---

## 7️⃣ Write Flow (Create Short URL)

1. API Gateway
2. Auth (optional)
3. Generate short_code
4. Write to DB
5. Write-through cache
6. Return short URL

Latency not critical (user waits once).

---

## 8️⃣ Analytics (Decouple or die)

🚫 **Never update analytics synchronously during redirect**

### Correct way

1. Redirect happens immediately
2. Fire-and-forget event → Kafka / Kinesis
3. Analytics workers aggregate asynchronously

Metrics:

* Click count
* Geo
* Referrer
* Device

📌 **Principal quote:**

> “Redirects should not care about analytics.”

---

## 9️⃣ Expiration Handling

### Option A — Lazy Deletion (Preferred)

* Check expiry during redirect
* If expired → 404

### Option B — TTL Index

* DynamoDB TTL attribute
* Background cleanup

---

## 🔥 Failure Scenarios & Answers Interviewers Love

### ❓ What if Redis goes down?

→ Fall back to DB
→ Increased latency but system still works

### ❓ What if DB goes down?

→ CDN + cache still serve hot URLs
→ Writes disabled temporarily

### ❓ Hot key (celebrity link)?

→ CDN absorbs traffic
→ Cache protects DB

---

## ⚠️ Common Interview Traps

### Trap 1: “We’ll use UUIDs”

❌ Long URLs
❌ Bad cache behavior
❌ Ugly links

---

### Trap 2: “Strong consistency needed”

❌ Not required
Redirects tolerate eventual consistency.

---

### Trap 3: “Let’s shard by user”

❌ Redirects don’t know users
❌ Hot partitions

---

## 🧠 Principal-Level Summary (Say this)

> “URL shorteners are read-heavy systems where latency matters more than consistency. The design optimizes the redirect path with aggressive caching and uses simple, collision-safe ID generation. Writes and analytics are decoupled to protect the hot path.”

If you say that calmly — interviewers stop asking questions.

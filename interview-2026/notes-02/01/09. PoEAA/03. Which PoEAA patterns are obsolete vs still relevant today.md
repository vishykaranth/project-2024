# PoEAA Patterns: **Obsolete vs Still Relevant (2026 View)**

> **Rule of thumb:**
> If a pattern exists mainly to work around **framework limitations**, it’s likely obsolete.
> If it encodes a **fundamental trade-off**, it’s still relevant.

---

## 🟢 STILL VERY RELEVANT (You should actively use these)

These survive **microservices, cloud, ORMs, REST, and event-driven systems**.

### 🧠 Domain & Application Structure

### ✅ Domain Model

Still the gold standard for complex business logic.

> Now reinforced by DDD, not replaced.

### ✅ Transaction Script

Perfect for:

* CRUD-heavy services
* Serverless functions
* Internal tools

> “Simple logic deserves simple structure.”

---

### ✅ Service Layer

Still critical to:

* Define application boundaries
* Enforce transactional consistency
* Expose APIs cleanly

> In microservices, *the service layer is the service*.

---

### 🗄️ Data Access & Consistency

### ✅ Data Mapper

Core idea behind Hibernate/JPA, MyBatis, EF Core.

> ORM didn’t replace it—it industrialized it.

### ✅ Unit of Work

Still essential:

* Transactions
* Batched writes
* Consistency

> Every ORM implements this whether you know it or not.

---

### 🔄 Concurrency & Correctness

### ✅ Optimistic Offline Lock

More relevant than ever in:

* Distributed systems
* REST APIs
* Event-driven architectures

> Version checks scale better than locks.

---

### 🌐 Distribution

### ✅ Data Transfer Object (DTO)

Still mandatory for:

* API boundaries
* Performance isolation
* Security

> DTOs are the antidote to leaking domain models.

### ✅ Remote Facade

Lives on as:

* API Gateway
* Backend-for-Frontend (BFF)

---

### 🧱 Base Patterns

### ✅ Gateway

Still everywhere:

* External APIs
* Payment providers
* Messaging systems

### ✅ Value Object

Central to:

* DDD
* Immutability
* Thread safety

---

## 🟡 CONTEXTUALLY RELEVANT (Use sparingly)

These are **not obsolete**, but no longer default.

### ⚠️ Active Record

Good for:

* Rails-style apps
* Simple CRUD services

Bad for:

* Complex domains

> Breaks down once logic grows.

---

### ⚠️ Table Module

Still seen in:

* Legacy systems
* Reporting-heavy domains

> Usually a stepping stone to Domain Model.

---

### ⚠️ Registry

Works in:

* Small apps
* Framework internals

Dangerous in:

* Large systems

> Hidden dependencies hurt testability.

---

### ⚠️ Client Session State

Okay for:

* Stateless JWTs
* Short-lived workflows

> But watch security and size.

---

## 🔴 MOSTLY OBSOLETE (Know them, don’t build them)

These existed to compensate for **2000-era tech constraints**.

---

### ❌ Entity Beans–Style Patterns (Implied by PoEAA)

Patterns tightly coupled to:

* Stateful servers
* Heavy containers

> Cloud-native killed them.

---

### ❌ Pessimistic Offline Lock

Scales poorly in distributed systems.

> Locks don’t survive network partitions.

---

### ❌ Implicit Lock

Hidden locking is a debugging nightmare.

> Modern systems demand explicit behavior.

---

### ❌ Database Session State

Kills horizontal scalability.

> Shared DB session ≠ cloud friendly.

---

### ❌ Serialized LOB

Breaks:

* Queryability
* Partial updates
* Schema evolution

> Replaced by document stores or JSON columns.

---

### ❌ Transform View (XSLT-heavy)

Too rigid and complex.

> Modern templating or frontend rendering won.

---

### ❌ Two-Step View

Over-engineered for most modern UIs.

> SPAs and SSR made this unnecessary.

---

## 🧭 THE MOST IMPORTANT INTERVIEW ANSWER

If asked:

> “Are PoEAA patterns still relevant?”

Say this:

> “The patterns that manage **complexity and trade-offs** are timeless.
> The ones that compensate for missing frameworks are mostly obsolete.”

That answer is **Principal-level**.

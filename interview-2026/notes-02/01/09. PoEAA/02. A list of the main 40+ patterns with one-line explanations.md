# 📘 Patterns of Enterprise Application Architecture

## The Main 40+ Patterns — One-Line Each

---

## 1  **Domain Logic Patterns**

*(How you organize business logic)*

1. **Transaction Script**
   → Organizes business logic as procedural scripts per request.

2. **Domain Model**
   → Encapsulates complex business logic in rich object models.

3. **Table Module**
   → Centralizes domain logic per database table, not per object.

4. **Service Layer**
   → Defines a clear application boundary with business operations.

---

## 2  **Data Source Architectural Patterns**

*(How your app talks to the database)*

5. **Table Data Gateway**
   → One class acts as a gateway for all operations on a table.

6. **Row Data Gateway**
   → One object represents a single row in a database table.

7. **Active Record**
   → Domain object contains both data and persistence logic.

8. **Data Mapper**
   → Separates domain objects from database access logic.

---

## 3  **Object-Relational Behavioral Patterns**

*(How objects behave with relational data)*

9. **Unit of Work**
   → Tracks changes to objects and writes them as a single transaction.

10. **Identity Map**
    → Ensures each database row is loaded only once per session.

11. **Lazy Load**
    → Delays loading related data until it is actually needed.

---

## 4  **Object-Relational Structural Patterns**

*(How objects map to tables)*

12. **Identity Field**
    → Stores object identity using a database key.

13. **Foreign Key Mapping**
    → Represents relationships using foreign keys.

14. **Association Table Mapping**
    → Maps many-to-many relationships via a join table.

15. **Dependent Mapping**
    → Child objects are persisted as part of their parent.

16. **Embedded Value**
    → Stores value objects directly inside parent table rows.

17. **Serialized LOB**
    → Serializes object graphs into large binary/text fields.

18. **Single Table Inheritance**
    → Stores an inheritance hierarchy in one table.

19. **Class Table Inheritance**
    → One table per class in an inheritance hierarchy.

20. **Concrete Table Inheritance**
    → One table per concrete subclass.

---

## 5  **Web Presentation Patterns**

*(How web requests are handled)*

21. **Model View Controller (MVC)**
    → Separates domain logic, UI rendering, and request handling.

22. **Page Controller**
    → One controller per web page or screen.

23. **Front Controller**
    → A single entry point routes all requests.

24. **Template View**
    → Renders pages using templates filled with model data.

25. **Transform View**
    → Uses transformations (e.g., XSLT) to render views.

26. **Two-Step View**
    → First creates logical view, then renders it physically.

27. **Application Controller**
    → Centralizes navigation and workflow decisions.

---

## 6  **Distribution Patterns**

*(How systems communicate across process boundaries)*

28. **Remote Facade**
    → Coarse-grained API over fine-grained domain objects.

29. **Data Transfer Object (DTO)**
    → Bundles data to reduce remote call overhead.

30. **Value Object**
    → Immutable object defined only by its values.

31. **Remote Proxy**
    → Local object that represents a remote service.

---

## 7  **Offline Concurrency Control Patterns**

*(How multiple users safely update data)*

32. **Optimistic Offline Lock**
    → Detects conflicts at commit time using version checks.

33. **Pessimistic Offline Lock**
    → Locks data early to prevent concurrent updates.

34. **Coarse-Grained Lock**
    → Locks large chunks of related data together.

35. **Implicit Lock**
    → Locking is hidden inside data access logic.

---

## 8  **Session State Patterns**

*(Where user state lives)*

36. **Client Session State**
    → Stores session state on the client.

37. **Server Session State**
    → Stores session state in server memory.

38. **Database Session State**
    → Persists session state in a database.

---

## 9  **Base Patterns**

*(Foundational infrastructure patterns)*

39. **Gateway**
    → Encapsulates access to an external system or resource.

40. **Mapper**
    → Translates between different object representations.

41. **Layer Supertype**
    → Base class shared by all objects in a layer.

42. **Registry**
    → Global lookup for commonly used objects.

43. **Plugin**
    → Enables dynamic extension of system behavior.

44. **Service Stub**
    → Local stand-in for a remote service.

45. **Separated Interface**
    → Splits interface from implementation for flexibility.

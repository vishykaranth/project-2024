## Answer 1

- When you move from a steady 10k TPS to a **1k → 100k TPS burst** model, the primary challenge shifts from *capacity* to *elasticity velocity*.
- A static 100k TPS architecture is prohibitively expensive (you’re paying for 99k TPS of idle air 99% of the time), but a slow-scaling architecture will crash before the first server finishes booting.

### 1. What breaks first?

In a 100x burst (1k to 100k in seconds), the "Traditional Auto-scaling" is the first thing to fail.

- **The Scaling Lag:** Standard VM or Container auto-scaling takes 2–5 minutes to spin up, pull images, and pass health checks. By then, your 5-minute sale is over, and your existing 1k-capacity fleet has already spiraled into "Retry Storms" and 504 Gateway Timeouts.
- **The Connection Pool Collapse:** Even if your app scales, your Database won't. 100k TPS typically translates to 1M+ concurrent TCP connections. Your RDS/Postgres instance will hit `max_connections` or SNAT port exhaustion at the Load Balancer level before the CPU even spikes.
- **The "Thundering Herd" on Cache:** If your cache (Redis/Memcached) isn't partitioned for 100k TPS, you'll hit network bandwidth limits on the cache nodes, causing a "Cache Stampede" where all 100k requests hit the DB simultaneously.

---

### 2. The "Elastic" Architecture Change

To handle this without a massive bill, you must move from **Provisioned** to **Serverless/On-Demand** and **Asynchronous** patterns.

| Component | 10k Steady Approach (Expensive) | 1k → 100k Burst Approach (Efficient) |
| --- | --- | --- |
| **Compute** | Oversized EKS/EC2 Cluster | **AWS Lambda / Google Cloud Run** (Instant scaling with no idle cost). |
| **Database** | Massive Provisioned RDS Instance | **DynamoDB / CosmosDB (On-Demand mode)**. It scales based on request volume, not provisioned units. |
| **Ingestion** | Direct API to DB writes | **Message Queue (SQS/Kafka)**. Use the queue as a "Load Leveler" to buffer the 100k burst and drain it at a cheaper pace. |
| **Edge** | Standard Load Balancer | **Edge Functions + Global CDN**. Move "Sold Out" logic or static checks to the CDN edge to stop 90k of those requests from ever hitting your servers. |

---

### 3. Summary of Key Changes

- **Queue-Based Load Leveling:** Do not try to process 100k transactions in real-time if you don't have to. Accept the "Order" into a high-throughput queue (like SQS) instantly, tell the user "We're processing your request," and let a smaller worker fleet (scaled for ~10k TPS) drain that queue over 30–60 minutes.
- **Virtual Waiting Rooms:** Use a service like *Cloudflare Waiting Room*. When the burst hits 100k, it holds users in a queue outside your infrastructure, letting only your "Golden 10k" through at a time. This protects your origin and saves you from paying for the 100k peak.
- **Database "On-Demand" Mode:** Switch to serverless databases where you pay per request. While the *unit price* is higher, the *total monthly cost* for a 5-minute burst is pennies compared to keeping a 100k-ready cluster running 24/7.

When you move from a steady **10k TPS** to a **1k → 100k TPS burst**, the "expensive" part of the previous architecture is the **Idle Provisioning**. If you provision for the 100k peak, you are wasting 99% of your budget 99% of the time.

To handle a 100k TPS burst while guaranteeing "exactly-once" style results and immediate user feedback, we must decouple **Acceptance** from **Processing**.

Here is the high-scale, AWS-native request flow for "Place Order":

---

### The "Buffered Write" Flow

1. **API Gateway (REST/HTTP API):** Receives the `POST /orders` request. It performs basic JWT validation and schema validation at the edge.
2. **Lambda (The "Ingestor"):** A lightweight function that does two things:
* Generates a **Correlation ID** (if not provided by the client) to ensure idempotency.
* Validates that the request has an `idempotency_key` (to prevent double clicks).


3. **SQS (Standard Queue):** The Lambda publishes the order payload to SQS and immediately returns a `202 Accepted` status to the user with the `order_id`.
> **Why?** SQS is one of the few AWS services that can scale to effectively infinite TPS instantly without "warm-up" time.


4. **Lambda (The "Processor"):** This is a separate fleet of functions triggered by SQS. We set the **Reserved Concurrency** on this function to match our downstream bottlenecks (e.g., the Payment Gateway's rate limit).
5. **DynamoDB (Idempotency Table):** The Processor first checks an Idempotency table using the `idempotency_key`. If the key exists, it discards the duplicate. If not, it marks the status as `PENDING`.
6. **Payment Gateway (External):** The Processor calls the payment provider (e.g., Stripe/Braintree).
* *Note:* If the payment provider is slow, SQS handles the back-pressure automatically by keeping the message in the queue.


7. **DynamoDB (Orders Table):** Once payment is confirmed, the Processor updates the order record in DynamoDB (On-Demand mode) to `PAID`.
8. **SNS / EventBridge:** The Processor publishes an `Order.Completed` event to trigger downstream fulfillment, inventory, and email notifications.

---

### How this solves your constraints:

* **100k TPS Burst:** SQS acts as a "shock absorber." The API Gateway and SQS can take the 100k hit, while your backend "drains" the queue at a manageable 10k TPS.
* **No Double Charges:** Using a DynamoDB Idempotency check with **Conditional Writes** (`attribute_not_exists`) ensures that even if a message is retried, the payment logic only executes once.
* **Immediate Response:** The user receives a `202 Accepted` in <100ms because the system only waits for the SQS write, not the full payment/database cycle.
* **Cost Efficiency:** You only pay for the Lambda executions and SQS messages used during that 5-minute window.

---

**Would you like me to provide the DynamoDB schema design for the Idempotency and Orders tables?**

To implement this, we focus on a **distributed idempotency** strategy. The key is that the "Persistence" layer is actually two separate actions: an immediate "Record of Intent" (SQS) and a final "Record of Truth" (DynamoDB).

### Order Placement Sequence Flow

1. **API Gateway (Throttling & Validation):**
The Gateway uses a **Usage Plan** to protect the backend. It validates the presence of an `x-idempotency-key` in the header. If missing, it rejects the request (400) immediately to save downstream compute.
2. **Lambda A (The "Acceptor"):**
This function performs a **"Check-and-Set"** on a DynamoDB **Idempotency Table**.
* If the key exists: Return the previous response (cached).
* If not: Create a record with a `TTL` (e.g., 24 hours) and status `IN_PROGRESS`.


3. **SQS (High-Throughput Buffer):**
Lambda A sends the payload to **Amazon SQS**. Crucially, it returns a `202 Accepted` to the user with a `Location` header or `orderId` for polling.
> **Status:** User is now free. The system is "indebted" to process the order.


4. **Lambda B (The "Processor"):**
This Lambda is triggered by the SQS queue. It uses **Batching** (e.g., 10 messages at a time) to reduce the number of calls to the database and payment provider.
5. **Payment Gateway (External Call):**
Lambda B executes the payment. It passes the `idempotency_key` to the Payment Provider (like Stripe/Square) to ensure that even if Lambda B crashes and retries, the provider won't charge the card twice.
6. **DynamoDB (Order Store - The Persistence):**
After a successful payment, Lambda B performs a **TransactWriteItems** operation:
* **Put** the final order into the `Orders` table.
* **Update** the `Idempotency` table status to `COMPLETED`.


7. **EventBridge (Downstream Fan-out):**
A successful write triggers an event. This notifies the "Inventory Service" to decrement stock and the "Email Service" to send a receipt.

---

### Why this design survives 100k TPS

* **Atomic Transactions:** By using DynamoDB's `TransactWriteItems`, you ensure that you never have an order marked as "Paid" without a corresponding record in the Order Store.
* **The "Drain" Rate:** You can cap the concurrency of Lambda B. If your payment provider can only handle 5,000 requests per second, you set Lambda B's concurrency limit to match that. The 100k burst stays safely in SQS without crashing anything.

# 🟧 AWS Interview Simulation

**Prompt:**

> *Design a highly available, scalable payment/order system on AWS.*

---

## 1️⃣ Candidate Opens (This opening matters a LOT)

### 🔹 Principal-level opening (memorize this structure)

> “Before proposing an architecture, I’d like to clarify scale, consistency, latency, compliance, and failure tolerance, because those directly influence service choices.”

If interviewer says *“assume reasonable defaults”*, continue:

> “I’ll assume peak traffic of ~10k TPS, sub-200ms latency for order placement, strong consistency for payment state, PCI compliance, and regional availability with graceful degradation.”

That sentence alone signals **senior maturity**.

---

## 2️⃣ High-Level Architecture (Whiteboard Level)

### 🔹 Core Flow

**Client → API → Order Service → Payment Orchestration → Settlement**

### 🔹 AWS Components (What AWS expects)

### Ingress

* **Route 53** – DNS + health checks
* **ALB + AWS WAF**

  * Rate limiting
  * Bot protection

> *Why not API Gateway?*
> “I’d use API Gateway if the system were fully serverless. For long-running orchestration, ALB + ECS gives me more control.”

---

### Compute Layer

* **ECS on Fargate**

  * One service for Orders
  * One service for Payments
* **Auto Scaling based on request count / queue depth**

> AWS signal:
> “I avoid managing servers unless there’s a clear reason.”

---

## 3️⃣ State & Data (This is where Principal candidates win)

### Order State

* **DynamoDB**

  * `order_id` as PK
  * `state`, `version`
  * Conditional writes for transitions

> “DynamoDB gives predictable latency and enforces state transitions using conditional expressions.”

Example:

```sql
SET state = 'PAID'
WHERE order_id = :id AND state = 'AUTHORIZED'
```

---

### Payment Ledger

* **Aurora PostgreSQL (Multi-AZ)**

  * Strong consistency
  * Auditable ledger
  * ACID guarantees

> “I split hot operational state from financial truth.”

This sentence alone = **Principal signal**.

---

### Caching

* **ElastiCache (Redis)**

  * Order reads
  * Payment status polling

---

## 4️⃣ Asynchronous & Eventing (Critical)

### Payment Flow

* Order placed → **SQS**
* Payment processor consumes
* External PSP call (Stripe / bank)
* Result emitted

### Why SQS?

* Absorbs spikes
* Prevents DB overload
* Enables retries

> AWS phrase:
> “Queues are my shock absorbers.”

---

### Event Fan-out

* **EventBridge**

  * OrderPlaced
  * PaymentAuthorized
  * PaymentFailed

Consumers:

* Notification service
* Fraud detection
* Audit logging

---

## 5️⃣ Failure Handling (AWS WILL drill here)

### ❓ *What if payment provider times out?*

Expected answer:

* Timeout + retry with backoff
* Idempotency keys
* DLQ after N retries

> “All external calls are assumed unreliable.”

---

### ❓ *What if a duplicate request arrives?*

* Idempotency key stored in DynamoDB
* Conditional write prevents double charge

---

### ❓ *What if DynamoDB is throttled?*

* Adaptive capacity
* Partition key design
* Backpressure via queues

---

## 6️⃣ Consistency Model (Very important)

| Component      | Consistency |
| -------------- | ----------- |
| Order creation | Strong      |
| Payment state  | Strong      |
| Notifications  | Eventual    |
| Analytics      | Eventual    |

> “I am strongly consistent where money is involved, eventually consistent everywhere else.”

---

## 7️⃣ Security & Compliance (Do NOT skip)

* **IAM least privilege**
* **KMS for encryption**
* **Secrets Manager** for PSP keys
* **VPC endpoints** (no public DB access)
* **PCI isolation**

  * Tokenize card data
  * Never store PAN

AWS phrase:

> “Compliance boundaries are architectural, not procedural.”

---

## 8️⃣ Availability Strategy

### Default

* **Multi-AZ** everything
* No multi-region by default

> “Multi-region is a business decision, not an architecture default.”

### If required:

* Active/passive
* DynamoDB Global Tables (orders only)
* Ledger remains single-region with DR

---

## 9️⃣ Deployment & Ops

* Blue/green deploys
* Canary for payment flows
* CloudWatch alarms
* X-Ray tracing
* Correlation IDs

---

## 🔥 Interviewer Drill-Downs (Traps)

### ❓ *Why not just use Aurora for everything?*

Correct:

> “Aurora would bottleneck under spikes. DynamoDB absorbs write bursts without coordination.”

---

### ❓ *Why not Lambda?*

Correct:

> “Payment orchestration often exceeds Lambda limits and benefits from long-lived workflows.”

---

### ❓ *How do you roll back a bad deploy?*

Correct:

> “Traffic shift rollback, not data rollback.”

---

## 🧠 Final Principal Summary (Memorize)

> “This design separates operational state from financial truth, uses queues to absorb failure, relies on managed services to reduce operational load, and enforces correctness at both the application and data layers.”

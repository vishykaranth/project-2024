When you move from a steady 10k TPS to a **1k → 100k TPS burst** model, the primary challenge shifts from *capacity* to *elasticity velocity*.

A static 100k TPS architecture is prohibitively expensive (you’re paying for 99k TPS of idle air 99% of the time), but a slow-scaling architecture will crash before the first server finishes booting.

### 1. What breaks first?

In a 100x burst (1k to 100k in seconds), the "Traditional Auto-scaling" is the first thing to fail.

* **The Scaling Lag:** Standard VM or Container auto-scaling takes 2–5 minutes to spin up, pull images, and pass health checks. By then, your 5-minute sale is over, and your existing 1k-capacity fleet has already spiraled into "Retry Storms" and 504 Gateway Timeouts.
* **The Connection Pool Collapse:** Even if your app scales, your Database won't. 100k TPS typically translates to 1M+ concurrent TCP connections. Your RDS/Postgres instance will hit `max_connections` or SNAT port exhaustion at the Load Balancer level before the CPU even spikes.
* **The "Thundering Herd" on Cache:** If your cache (Redis/Memcached) isn't partitioned for 100k TPS, you'll hit network bandwidth limits on the cache nodes, causing a "Cache Stampede" where all 100k requests hit the DB simultaneously.

### 3. Summary of Key Changes

1. **Queue-Based Load Leveling:** Do not try to process 100k transactions in real-time if you don't have to. Accept the "Order" into a high-throughput queue (like SQS) instantly, tell the user "We're processing your request," and let a smaller worker fleet (scaled for ~10k TPS) drain that queue over 30–60 minutes.

3. **Database "On-Demand" Mode:** Switch to serverless databases where you pay per request. While the *unit price* is higher, the *total monthly cost* for a 5-minute burst is pennies compared to keeping a 100k-ready cluster running 24/7.

### The Corrected "Synchronous Auth" Flow

1. **API Gateway (Throttling & Validation):** Directly handles the incoming burst. If the burst exceeds your hard capacity (e.g., 100k+), it returns a `429 Too Many Requests`. This is your first line of defense.
2. **Lambda (The Orchestrator):** Execution begins synchronously.
* **Step A: Idempotency Check.** Hits DynamoDB (On-Demand) to ensure this `idempotency_key` isn't a double-click.
* **Step B: Inventory Reservation.** Performs a conditional update in DynamoDB: `SET stock = stock - 1 WHERE stock > 0`. If this fails, return `400 Out of Stock` immediately.


3. **The Synchronous Payment Call:** The Lambda calls the Payment Provider (e.g., Stripe) **directly**.
* **Success:** Continue to step 4.
* **Decline:** Return the error (e.g., `402 Payment Required`) to the user immediately.


> *Note: This is the critical "Success/Failure" moment the user sees on their screen.*


4. **The "Async Hand-off" (Post-Auth):** Once the card is authorized, the Lambda publishes an `Order.Authorized` message to **SQS/EventBridge** and returns a `200 OK` to the user.
5. **Downstream (The Rest):** Fulfillment, Shipping, and Final Invoice Generation happen asynchronously via the SQS workers. The user is happy because they have a "Success" screen and an Order ID.

In a high-scale environment, you must **flip the order of operations**. You never authorize a payment unless you have successfully reserved the inventory first. If you charge a user and then realize the item is gone, you face a "Compensating Transaction" (Refund) which is expensive, creates customer anger, and risks fee losses.

---

### The "Reservation-First" Flow

To avoid the "Charge without Product" scenario, we use **DynamoDB Transactions** as our "All-or-Nothing" gate before the external API call.

#### 1. State: PENDING (Atomic Gate)

* **Service:** Lambda + DynamoDB.
* **Action:** Execute a `TransactWriteItems`.
* **Condition:** `Inventory > 0` AND `IdempotencyKey` does not exist.
* **Success:** Inventory is decremented; the request moves to the Payment state.
* **Failure:** Return `409 Conflict` (Sold Out) or `400` (Duplicate). **No payment is attempted.**



#### 2. State: AUTHORIZING (The External Risk)

* **Service:** Lambda + External Payment Provider (Stripe/Adyen).
* **Action:** Synchronous HTTPS call.
* **Scenario A (Payment Success):** Move to **CONFIRMED**.
* **Scenario B (Payment Declined):** We must now **Compensate**. The Lambda immediately fires an asynchronous "Inventory Release" event and returns `402 Declined` to the user.
* **Scenario C (Timeout/Network Error):** This is the "Gray Zone." You don't know if the bank took the money.



#### 3. State: COMPENSATING (Handling Scenario B/C)

* **Service:** AWS Step Functions (Standard or Express).
* **Action:** If the Payment call fails or times out, Step Functions triggers a retry or a cleanup.
* **Logic:** `Update Inventory SET stock = stock + 1`. This puts the item back for the next user in the 100k burst.

### Failure Semantics: "What if the DB fails after Payment?"

If the bank says **"Success"** but your Lambda crashes before it can write the "Order Confirmed" record to DynamoDB, you have a **Zombies Order** (Charged but not recorded).

**The Solution: Post-Auth Recovery**

1. **Before** calling the Payment Provider, write an "Intent" record to DynamoDB with a status of `PAYMENT_STARTED`.
2. **After** the Payment Provider returns success, update it to `SUCCESS`.
3. **The Reaper:** Have a background **Lambda (via DynamoDB Streams or EventBridge Scheduler)** that looks for `PAYMENT_STARTED` records older than 5 minutes.
4. **Action:** The Reaper calls the Payment Provider’s `GET /transaction/{id}`.
* If the bank says "Paid," the Reaper completes the order.
* If the bank says "Failed," the Reaper releases the inventory.

# 🚫 When NOT to Use Java Streams (Authoritative Guide)

---

## 1️⃣ Complex Control Flow (Multiple Breaks / Continues)

### ❌ Don’t use streams

```java
for (Order o : orders) {
    if (o.isFraud()) break;
    if (!o.isValid()) continue;
    if (retry(o)) return SUCCESS;
}
```

### ❌ Stream version is unreadable or impossible

Streams:

* No `break`
* No `continue`
* No early `return` (except terminal ops)

### ✅ Use loops

### Kill shot

> **Streams model pipelines, not decision trees.**

---

## 2️⃣ Heavy Stateful Logic

### ❌ Don’t use streams

```java
AtomicInteger counter = new AtomicInteger();

orders.stream()
      .forEach(o -> {
          if (o.amount > limit) {
              counter.incrementAndGet();
          }
      });
```

### Why it’s bad

* Side effects
* Race conditions in parallel streams
* Hard to reason about

### Kill shot

> **Streams assume stateless lambdas.**

---

## 3️⃣ Performance-Critical Hot Paths (µs Matter)

### ❌ Don’t use streams

```java
orders.stream()
      .filter(...)
      .map(...)
      .reduce(...)
```

### Why

* Lambda allocation
* Boxing
* Virtual calls

### ✅ Use loops

```java
for (int i = 0; i < orders.length; i++) { ... }
```

### Kill shot

> **Streams trade a bit of performance for clarity.**

---

## 4️⃣ I/O or Blocking Calls

### ❌ Don’t use streams

```java
orders.stream()
      .forEach(o -> paymentService.charge(o));
```

### Why

* Sequential stream = slow
* Parallel stream = thread starvation

### ✅ Use async pipelines / executors

### Kill shot

> **Streams don’t manage latency, only computation.**

---

## 5️⃣ Debugging-Heavy Code Paths

### ❌ Don’t use streams

```java
orders.stream()
      .filter(this::complexCheck)
      .map(this::transform)
      .toList();
```

### Why

* Hard to step through
* Stack traces are noisy
* Lambdas hide context

### Kill shot

> **Streams are hard to debug, easy to read.**

---

## 6️⃣ Business Transactions with Side Effects

### ❌ Don’t use streams

```java
@Transactional
orders.stream()
      .forEach(this::saveAndNotify);
```

### Why

* Partial failures
* No compensation
* Poor error handling

### Kill shot

> **Streams don’t express failure semantics.**

---

## 7️⃣ Error Handling & Retries

### ❌ Don’t use streams

```java
stream.map(o -> remoteCall(o))
      .toList();
```

### Why

* Checked exceptions don’t fit
* Retry logic gets ugly
* No circuit breakers

### Kill shot

> **Streams don’t compose well with failures.**

---

## 8️⃣ Large Datasets You Don’t Want in Memory

### ❌ Don’t use streams

```java
collect(Collectors.groupingBy(...));
```

### Why

* Eager materialization
* OOM risk

### ✅ Use:

* DB aggregation
* Chunked processing
* Cursor-based reads

### Kill shot

> **Collectors pull everything into memory.**

---

## 9️⃣ Highly Sequential Logic (Order Matters)

### ❌ Don’t use streams

```java
orders.stream()
      .forEachOrdered(this::process);
```

### Why

* Loses parallel benefit
* More overhead than loop

### Kill shot

> **If order matters deeply, streams add no value.**

---

## 🔟 Mutating Existing Objects as Primary Goal

### ❌ Don’t use streams

```java
orders.stream()
      .forEach(o -> o.setStatus(PROCESSED));
```

### Why

* Violates functional style
* Side effects dominate

### Kill shot

> **Streams transform data, they don’t manage state.**

---

## 1️⃣1️⃣ Nested Streams (Readability Collapse)

### ❌ Don’t do this

```java
users.stream()
     .flatMap(u -> u.orders.stream()
         .filter(o -> o.items.stream()
             .anyMatch(i -> i.price > 100)))
```

### Why

* Cognitive overload
* Debug nightmare

### Kill shot

> **If indentation keeps growing, streams are the wrong tool.**

---

## 1️⃣2️⃣ Very Small Collections (Noise > Value)

### ❌ Don’t use streams

```java
list.stream().filter(...).toList();
```

For 3 elements.

### Why

* Verbose
* Harder to read than loop

### Kill shot

> **Streams shine with scale, not triviality.**

---

# 🧠 Decision Rule (Memorize This)

> Use streams when your logic is:
>
> * Stateless
> * Declarative
> * Transformational
> * Read-heavy

> Avoid streams when your logic is:
>
> * Stateful
> * Imperative
> * Failure-prone
> * Latency-critical

* “Streams hide complexity — including bad complexity.”

# HLD vs Java LLD — What to Cover (Authoritative Guide)

---

## 🏗️ HIGH-LEVEL DESIGN (HLD)

### HLD answers **system questions**

> *“What exists, where it runs, and how it interacts.”*

### ❌ What HLD is NOT

* Java classes
* Frameworks
* Streams
* Annotations
* Method signatures

If you mention `@Service`, you’re already too low.

---

## ✅ What You MUST Cover in HLD

### 1️⃣ System Boundaries & Responsibilities

* What are the **major components**?
* What does each component own?

Example:

* API Gateway
* Order Service
* Payment Service
* Inventory Service

---

### 2️⃣ Data Flow (Happy Path + Failure Path)

* Request flow
* Response flow
* Retry flow
* Compensation flow

Example:

> “Order request → validate → reserve inventory → charge payment → persist order → emit event”

---

### 3️⃣ Scalability Strategy

* Horizontal vs vertical
* Stateless vs stateful
* Read vs write scaling

Example:

* Stateless services behind LB
* Partition by `orderId`

---

### 4️⃣ Consistency Model

* Strong vs eventual
* Where consistency is relaxed
* Where it is enforced

Example:

> “Inventory is strongly consistent; order history is eventually consistent.”

---

### 5️⃣ Data Storage Choices

* SQL vs NoSQL
* Why that choice
* Access patterns

Example:

* Orders → DynamoDB (high write TPS)
* Reports → OLAP

---

### 6️⃣ Reliability & Failure Handling

* Timeouts
* Retries
* Circuit breakers
* Dead letter queues

Example:

> “Payment failures trigger compensation via saga.”

---

### 7️⃣ Concurrency Model

* How contention is handled
* Locks vs optimistic concurrency
* Idempotency

---

### 8️⃣ Communication Style

* Sync vs async
* REST vs events
* Why

Example:

> “Payment is synchronous; fulfillment is async.”

---

### 9️⃣ Security & Compliance

* AuthN / AuthZ
* PCI, PII
* Data encryption

---

### 1️⃣0️⃣ Observability

* Logs
* Metrics
* Tracing

---

### 🎯 HLD Kill Shot

> **“HLD explains how the system survives load, failure, and change.”**

---

---

# 🧩 JAVA LOW-LEVEL DESIGN (LLD)

### Java LLD answers **code questions**

> *“How exactly do we implement this in Java?”*

---

## ✅ What You MUST Cover in Java LLD

---

### 1️⃣ Class Responsibilities

* Key classes
* Single Responsibility
* Ownership of logic

Example:

* `OrderService`
* `InventoryRepository`
* `PaymentClient`

---

### 2️⃣ Object Relationships

* Aggregates
* Composition vs inheritance
* Lifecycles

Example:

> “Order is the aggregate root.”

---

### 3️⃣ Design Patterns

* Service Layer
* Factory
* Strategy
* Adapter

Explain **why**, not just names.

---

### 4️⃣ Concurrency at Code Level

* Locks
* Atomic types
* Thread safety

Example:

* `@Version` for optimistic locking
* `synchronized` where unavoidable

---

### 5️⃣ Error Handling

* Checked vs unchecked
* Retryable vs fatal
* Mapping to HTTP errors

---

### 6️⃣ Transaction Boundaries

* `@Transactional`
* Isolation levels
* Rollback rules

---

### 7️⃣ Data Access Strategy

* JPA vs JDBC
* Batch size
* Lazy vs eager

---

### 8️⃣ Performance Considerations

* Streams vs loops
* Caching
* Connection pooling

---

### 9️⃣ API Contracts

* DTOs
* Validation
* Versioning

---

### 🔟 Testability

* Unit tests
* Mocking
* Test seams

---

### 🎯 Java LLD Kill Shot

> **“LLD turns architectural intent into maintainable, testable Java code.”**

# 🚫 Common Interview Mistakes

### ❌ Mixing Levels

HLD mistake:

> “We use Spring Boot with JPA repositories.”

LLD mistake:

> “The system scales horizontally with Kafka.”

---

### ❌ Over-specifying Java in HLD

Mentioning:

* Streams
* Parallelism
* Annotations

---

### ❌ Hand-waving in LLD

Saying:

> “This is handled by Spring”

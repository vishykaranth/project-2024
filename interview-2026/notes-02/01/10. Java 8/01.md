# STEP 0 — First, reset your mental model (IMPORTANT)

### ❌ Wrong mental model

> “Streams are collections with fancy methods.”

### ✅ Correct mental model

> **“A Stream is a pipeline for processing data, not a data structure.”**

* A `List` **stores** data
* A `Stream` **processes** data
* A Stream:

  * Has **no storage**
  * Is **lazy**
  * Is **one-time use**

Keep this in your head. Everything else clicks.

# STEP 1 — Why Streams exist (the pain they fix)

### Before Java 8 (imperative style)

```java
List<Integer> evens = new ArrayList<>();
for (int n : numbers) {
    if (n % 2 == 0) {
        evens.add(n * 2);
    }
}
```

Problems:

* Boilerplate
* Logic scattered
* Hard to compose
* Hard to parallelize safely

---

### With Streams (declarative style)

```java
List<Integer> evens =
    numbers.stream()
           .filter(n -> n % 2 == 0)
           .map(n -> n * 2)
           .toList();
```

Now you say **what**, not **how**.

---

# STEP 2 — The Stream Pipeline (THIS IS CORE)

Every stream has **three parts**:

```
Source → Intermediate Operations → Terminal Operation
```

Example:

```java
numbers.stream()          // Source
       .filter(n -> n > 10)   // Intermediate
       .map(n -> n * 2)       // Intermediate
       .toList();             // Terminal
```

### Rule (memorize this)

> **Nothing runs until a terminal operation is called.**

This is laziness.

---

# STEP 3 — Creating Streams (Sources)

Common sources:

```java
list.stream()
set.stream()
map.entrySet().stream()
```

From values:

```java
Stream.of(1, 2, 3)
```

From arrays:

```java
Arrays.stream(arr)
```

Infinite streams:

```java
Stream.iterate(1, n -> n + 1)
Stream.generate(Math::random)
```

⚠️ Infinite streams **must** be limited:

```java
Stream.iterate(1, n -> n + 1)
      .limit(10)
      .forEach(System.out::println);
```

---

# STEP 4 — Intermediate Operations (TRANSFORM, DON’T EXECUTE)

These **return a new Stream** and are **lazy**.

## 1️⃣ filter — keep or discard

```java
.filter(n -> n > 10)
```

Think:

> “Should this element continue in the pipeline?”

---

## 2️⃣ map — transform element

```java
.map(n -> n * 2)
```

Think:

> “Change shape, not count.”

---

## 3️⃣ flatMap — flatten nested structures (VERY IMPORTANT)

Problem:

```java
List<List<String>> data;
```

Solution:

```java
data.stream()
    .flatMap(List::stream)
    .toList();
```

Mental model:

> **map = 1 → 1**
> **flatMap = many → 1 stream**

This is a *huge interview favorite*.

---

## 4️⃣ distinct, sorted, limit, skip

```java
.distinct()
.sorted()
.limit(5)
.skip(2)
```

These are **stateful** operations — more expensive.

---

# STEP 5 — Terminal Operations (EXECUTE THE PIPELINE)

This is where things **actually run**.

## 1️⃣ forEach — do something (side effects)

```java
.forEach(System.out::println);
```

⚠️ Avoid business logic here.

---

## 2️⃣ collect — convert stream into something

```java
.collect(Collectors.toList())
.collect(Collectors.toSet())
.collect(Collectors.groupingBy(User::getRole))
```

This is **very important** in real systems.

---

## 3️⃣ reduce — combine into one value

```java
int sum =
    numbers.stream()
           .reduce(0, Integer::sum);
```

Mental model:

> “Fold the stream into one result.”

---

## 4️⃣ findFirst, anyMatch, allMatch, noneMatch

```java
.anyMatch(n -> n > 100)
.allMatch(n -> n > 0)
```

Short-circuiting → fast.

# STEP 6 — Laziness (THIS IS A TRAP QUESTION)

```java
Stream<Integer> s =
    numbers.stream()
           .filter(n -> {
               System.out.println("Filtering " + n);
               return n > 5;
           });

System.out.println("Nothing yet");
```

Nothing prints.

Only when:

```java
s.toList();
```

Now it runs.

### Kill-shot line (remember this)

> **Streams describe computation; terminal operations trigger it.**

---

# STEP 7 — Order of Operations (PERFORMANCE GOLD)

This:

```java
stream
  .filter(...)
  .map(...)
  .limit(5)
```

Is **much faster** than:

```java
stream
  .map(...)
  .filter(...)
  .limit(5)
```

Why?

* `filter` reduces data earlier
* `limit` short-circuits

### Rule

> **Reduce early, expand late.**

---

# STEP 8 — Parallel Streams (HANDLE WITH CARE)

```java
numbers.parallelStream()
       .map(...)
       .toList();
```

### When they help

* CPU-bound
* Stateless operations
* Large data sets

### When they hurt

* I/O
* Locks
* Shared mutable state

⚠️ Interview truth:

> **Parallel streams optimize throughput, not latency.**

---

# STEP 9 — Common Mistakes (VERY IMPORTANT)

❌ Mutating external state:

```java
List<Integer> result = new ArrayList<>();
stream.forEach(result::add); // NOT SAFE
```

❌ Using streams for simple loops

❌ Debugging with `forEach`

❌ Streams inside streams (nested pipelines)

---

# STEP 10 — Streams vs Loops (WHEN TO USE WHAT)

| Use Streams when    | Use Loops when                 |
| ------------------- | ------------------------------ |
| Transforming data   | Complex control flow           |
| Filtering/mapping   | Early exits                    |
| Aggregations        | Performance-critical hot paths |
| Readability matters | Debugging is key               |

### Principal-level truth

> **Streams are about clarity first, performance second.**

# STEP 11 — Real-World Example (End-to-End)

```java
Map<String, Long> countByRole =
    users.stream()
         .filter(User::isActive)
         .collect(Collectors.groupingBy(
             User::getRole,
             Collectors.counting()
         ));
```

Readable. Testable. Declarative.

* “flatMap flattens structure.”
* “Parallel streams trade latency for throughput.”
* “Avoid side effects in streams.”

> **A Stream is a lazy, one-time pipeline that transforms data from a source to a result.**

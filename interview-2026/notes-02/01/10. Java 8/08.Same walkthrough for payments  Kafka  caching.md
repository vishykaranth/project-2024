# 1️⃣ PAYMENTS — HLD → Java LLD

---

## 🏗️ PAYMENTS — HLD

### HLD Goal

> “Charge customers safely without double-charging, while surviving retries and failures.”

---

### HLD Components

```
Order Service
   |
Payment Service
   |
External Payment Gateway
```

---

### HLD Responsibilities

**Payment Service**

* Owns payment lifecycle
* Guarantees **idempotency**
* Handles retries safely
* Never double-charges

---

### HLD Constraints (VERY IMPORTANT)

* External gateway is **unreliable**
* Requests may be **replayed**
* Latency matters, but **correctness > speed**

---

### HLD Decisions

| Concern            | Decision           |
| ------------------ | ------------------ |
| Consistency        | Strong             |
| Communication      | Synchronous        |
| Duplicate requests | Idempotency        |
| Failures           | Retry with backoff |
| Scaling            | Stateless service  |

---

### 🎯 HLD Kill Shot

> **“Payments are about correctness first; throughput is secondary.”**

---

## 🔁 HLD → Java LLD Mapping (Payments)

| HLD Concept         | Java LLD       |
| ------------------- | -------------- |
| Idempotency         | DB/Redis key   |
| Strong consistency  | Transaction    |
| External dependency | Adapter        |
| Retry               | Explicit logic |
| Payment lifecycle   | State machine  |

---

## 🧩 Java LLD — Payments

### Entry

```java
@Service
public class PaymentService {

    @Transactional
    public PaymentResult charge(PaymentRequest req) {
        if (repo.exists(req.idempotencyKey())) {
            return repo.previousResult(req.idempotencyKey());
        }

        Payment payment = Payment.create(req);
        repo.save(payment);

        GatewayResponse resp = gateway.charge(req);

        payment.markCompleted(resp);
        return PaymentResult.success();
    }
}
```

---

### Key LLD Patterns

* **Idempotent Receiver**
* **Service Layer**
* **Adapter (Payment Gateway)**

---

### 🎯 Payment Kill Shot

> **“Idempotency is a business invariant, not an optimization.”**

# 2️⃣ KAFKA / EVENT STREAMING — HLD → Java LLD

---

## 🏗️ KAFKA — HLD

### HLD Goal

> “Decouple services and absorb traffic spikes without losing data.”

---

### HLD Components

```
Order Service → Kafka → Fulfillment / Analytics
```

---

### HLD Responsibilities

* Kafka buffers load
* Producers never block on consumers
* Consumers process independently

---

### HLD Constraints

* At-least-once delivery
* Ordering per key
* Eventual consistency

---

### HLD Decisions

| Concern          | Decision        |
| ---------------- | --------------- |
| Delivery         | At-least-once   |
| Ordering         | Per aggregate   |
| Failure handling | Retry + DLQ     |
| Scaling          | Consumer groups |

---

### 🎯 HLD Kill Shot

> **“Kafka trades immediacy for resilience.”**

---

## 🔁 HLD → Java LLD Mapping (Kafka)

| HLD Concept | Java LLD       |
| ----------- | -------------- |
| Producer    | EventPublisher |
| Consumer    | KafkaListener  |
| Ordering    | Partition key  |
| Retry       | Retry topic    |
| Failure     | DLQ            |

---

## 🧩 Java LLD — Kafka

### Producer

```java
events.publish(
    new OrderPlacedEvent(orderId),
    orderId // partition key
);
```

---

### Consumer

```java
@KafkaListener(topics = "order-placed")
public void handle(OrderPlacedEvent event) {
    fulfillmentService.ship(event.orderId());
}
```

---

### Failure Handling

```java
@Retryable(...)
@KafkaListener(...)
public void handle(...) { ... }
```

---

### 🎯 Kafka Kill Shot

> **“Kafka gives me temporal decoupling, not transactional consistency.”**

# 3️⃣ CACHING — HLD → Java LLD

---

## 🏗️ CACHING — HLD

### HLD Goal

> “Reduce latency and database load without breaking correctness.”

---

### HLD Components

```
Service → Cache → Database
```

---

### HLD Responsibilities

* Cache is **best-effort**
* DB is source of truth
* Cache failures must not break system

---

### HLD Decisions

| Concern     | Decision     |
| ----------- | ------------ |
| Cache type  | Read-through |
| Consistency | Eventual     |
| Failure     | Fail open    |
| Eviction    | TTL          |

---

### 🎯 HLD Kill Shot

> **“Caches optimize performance, not correctness.”**

---

## 🔁 HLD → Java LLD Mapping (Caching)

| HLD Concept  | Java LLD          |
| ------------ | ----------------- |
| Read-through | Cache abstraction |
| TTL          | Redis config      |
| Fail open    | Fallback logic    |
| Invalidation | Events            |

---

## 🧩 Java LLD — Caching

```java
@Cacheable(
    value = "product",
    key = "#productId",
    unless = "#result == null"
)
public Product getProduct(String productId) {
    return repo.findById(productId);
}
```

---

### Cache Invalidation

```java
@CacheEvict(value = "product", key = "#productId")
public void updateProduct(...) { ... }
```

---

### 🎯 Cache Kill Shot

> **“Invalidation is harder than caching — I design for it upfront.”**

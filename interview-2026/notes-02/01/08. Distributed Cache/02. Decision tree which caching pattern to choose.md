# Distributed Cache – Decision Tree

**“Which caching pattern should I choose, and why?”**

Think in **three axes** (say this out loud in interviews):

> **Read path · Write path · Failure behavior**

---

## 🌳 Step 1: Do you need a cache at all?

**Q:** Is the data:

* Expensive to compute?
* Read-heavy?
* Tolerant to staleness?

**If NO →** ❌ *Don’t cache*

> “Caching incorrect data is worse than not caching.”

**If YES →** Continue

---

## 🌳 Step 2: What happens if the cache goes down?

### Q: Can your system still work without cache?

### ✅ YES → Cache is **optional**

➡️ **Cache-Aside (Lazy Loading)**

**Why**

* Simplest
* Cache failure ≠ system failure
* Default choice in 80% systems

**One-liner**

> “Cache-aside is the safest default because the DB remains the source of truth.”

---

### ❌ NO → Cache is **mandatory**

➡️ Continue

---

## 🌳 Step 3: Who should load data on a cache miss?

### Q: Should the application control fallback logic?

### ✅ YES → **Cache-Aside**

* App handles miss
* Full control over retries, fallbacks, observability

### ❌ NO → **Read-Through Cache**

* Cache auto-loads data
* Cleaner code, less flexibility

**Principal signal**

> “Read-through trades control for simplicity.”

---

## 🌳 Step 4: How critical is write consistency?

### Q: After a write, must reads be immediately correct?

---

### 🔒 STRONG consistency required

➡️ **Write-Through Cache**

**Characteristics**

* Write cache + DB synchronously
* Higher latency
* Stronger correctness

**Use when**

* Financial limits
* Inventory
* Authorization data

**One-liner**

> “Write-through shifts latency to protect correctness.”

---

### ⚖️ Eventual consistency acceptable

➡️ Continue

---

## 🌳 Step 5: Is write latency critical?

### 🚀 YES → **Write-Behind (Write-Back) Cache**

**Characteristics**

* Write cache first
* Async DB persistence
* Risk of data loss

**Use when**

* Metrics
* Analytics
* Activity feeds

**Principal warning**

> “Write-behind is a durability decision, not just a performance one.”

---

### 🐢 NO → **Write-Through**

---

## 🌳 Step 6: How often does data change?

### 🔁 Frequently changing data

➡️ Prefer:

* Short TTL
* Explicit invalidation
* Event-driven cache updates

**Avoid**

* Long TTLs
* Write-behind

---

### 🧊 Mostly static data

➡️ Prefer:

* Long TTL
* Cache warming
* CDN + distributed cache combo

---

## 🌳 Step 7: Are cache misses expensive?

### ❗ YES (DB meltdown risk)

➡️ Add **Cache Stampede Protection**

* Mutex per key
* Request coalescing
* Early refresh
* TTL jitter

**One-liner**

> “Misses are fine; synchronized misses are not.”

---

## 🌳 Step 8: Are keys frequently missing (404-heavy)?

### ❗ YES → **Cache Nulls / Bloom Filter**

**Why**

* Prevents cache penetration
* Protects DB

---

## 🌳 Step 9: Is data personalized per user?

### 👤 YES

➡️ Use:

* Smaller TTLs
* Per-user keys
* Local + distributed hybrid cache

**Avoid**

* Over-sharing keys
* Large values

---

## 🧠 Quick Decision Table (Interview Gold)

| Scenario                | Best Pattern                 |
| ----------------------- | ---------------------------- |
| Default microservice    | Cache-aside                  |
| Critical correctness    | Write-through                |
| Ultra-low write latency | Write-behind                 |
| Library-managed cache   | Read-through                 |
| Read-heavy, static data | Cache-aside + long TTL       |
| DB protection required  | Cache-aside + rate limits    |
| Cache mandatory         | Read-through / write-through |

---

## 🎯 30-Second “Whiteboard Answer” (Memorize This)

> “I start with cache-aside because it fails open.
> If correctness is critical, I move to write-through.
> If write latency dominates and data loss is acceptable, I consider write-behind.
> Read-through is useful when I want to centralize loading logic, but I lose control.
> Beyond the pattern, the real risks are stampedes, invalidation, and failure behavior.”

---

## 🚨 Interview Traps to Avoid

❌ “Read-through is better than cache-aside”
✅ “It depends on operational control.”

❌ “Write-behind is faster”
✅ “Write-behind trades durability for latency.”

❌ “TTL solves consistency”
✅ “TTL only bounds staleness.”

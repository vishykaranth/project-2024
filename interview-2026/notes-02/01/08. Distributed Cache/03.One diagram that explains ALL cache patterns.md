## 🧠 THE ONE DIAGRAM (Master Cache Diagram)

```
                ┌──────────────┐
                │   Client /   │
                │ Application  │
                └──────┬───────┘
                       │
               (READ / WRITE)
                       │
               ┌───────▼────────┐
               │     CACHE      │
               │ (Distributed) │
               └───────┬────────┘
                       │
                 (MISS / FLUSH)
                       │
               ┌───────▼────────┐
               │   DATABASE     │
               │ (Source Truth) │
               └────────────────┘
```

👉 **Rule**:
**Every caching pattern is just a different choice of who controls the arrows and when.**

---

## 1️⃣ Cache-Aside (Lazy Loading) – DEFAULT

### Flow (READ)

```
App → Cache → (miss) → DB → App → Cache
```

### Flow (WRITE)

```
App → DB
App → Cache (invalidate or update)
```

### What to say

> “The application owns cache misses and invalidation. Cache fails open.”

### When to point at diagram

* App ↔ Cache ↔ DB
* App is in control

### Use when

* Most systems
* You want safety + flexibility

---

## 2️⃣ Read-Through Cache

### Flow (READ)

```
App → Cache → (miss) → Cache → DB → Cache → App
```

### Diagram explanation

* App **never talks to DB on reads**
* Cache becomes a *smart proxy*

### What to say

> “Read-through centralizes loading logic but reduces application control.”

### Interview signal

* Good for SDKs, platforms, shared infra

---

## 3️⃣ Write-Through Cache

### Flow (WRITE)

```
App → Cache → DB
```

### Reads

```
App → Cache → (hit)
```

### Diagram emphasis

* Writes go **through cache**
* DB write is synchronous

### What to say

> “Write-through shifts latency to guarantee read-after-write consistency.”

### Use when

* Financial limits
* Inventory
* Authorization data

---

## 4️⃣ Write-Behind (Write-Back) Cache

### Flow (WRITE)

```
App → Cache → (async) → DB
```

### Diagram emphasis

* Cache is ahead of DB
* DB write is delayed

### What to say (IMPORTANT)

> “Write-behind improves write latency but risks data loss.”

### Use when

* Metrics
* Feeds
* Analytics
* Never for money

---

## 5️⃣ Cache with TTL / Expiry

### Diagram annotation

```
CACHE
[ key → value | TTL ]
```

### Behavior

* Cache evicts automatically
* DB is eventually read again

### What to say

> “TTL bounds staleness; it does not guarantee consistency.”

---

## 6️⃣ Cache Invalidation (Event-Driven)

### Flow

```
Write → DB → Event → Cache Invalidate
```

### Diagram add-on

```
DB ──► Event Bus ──► Cache
```

### What to say

> “Events give faster correctness than TTL, but increase system coupling.”

---

## 7️⃣ Cache Stampede Protection (Overlay)

### Diagram overlay

```
App
 └─► Cache
      └─► [LOCK per key]
           └─► DB
```

### What to say

> “Misses are fine. Coordinated misses are not.”

---

## 🎯 The 30-Second Interview Script (Memorize)

> “I use one mental model: App, Cache, and DB.
> Cache-aside means the app controls misses.
> Read-through means cache controls misses.
> Write-through means writes synchronously update DB.
> Write-behind means cache is ahead of DB.
> TTL and invalidation control staleness, not correctness.
> The real design choice is failure behavior, not hit ratio.”

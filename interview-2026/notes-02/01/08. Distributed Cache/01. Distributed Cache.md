# Distributed Cache – Interview Pack

## 1️⃣ Warm-up (Signal maturity early)

### Q1. What is a distributed cache?

**Expected answer**

> A distributed cache is a shared, in-memory data store spread across multiple nodes to reduce latency and offload backend systems while providing horizontal scalability and fault tolerance.

**Green flags**

* Mentions *latency*, *backend offload*, *horizontal scale*
* Not just “Redis cluster”

**Red flags**

* “It’s Redis”
* No mention of failure or consistency

---

### Q2. Why do we need a distributed cache instead of local cache?

**One-liner**

> Local cache improves single-node latency; distributed cache improves system-wide scalability, consistency, and backend protection.

**Follow-up drill**

* Local cache → fast but inconsistent
* Distributed cache → slightly slower but shared + controllable

---

## 2️⃣ Core Concepts (Must-know)

### Q3. What problems does a distributed cache solve?

**Expected**

* Read latency reduction
* Database load shedding
* Traffic spikes smoothing
* Expensive computation caching
* Session/state sharing

**Principal signal**

> “Cache is a performance *and* resilience primitive.”

---

### Q4. What are common distributed cache systems?

**Expected**

* Redis (Cluster, Sentinel)
* Memcached
* Hazelcast / Ignite
* Aerospike
* Cloud: ElastiCache, Azure Cache, Memorystore

**Bonus**

* When *not* to use Redis (large values, heavy scans)

## 3️⃣ Caching Patterns (Very Important)

### Q5. Explain cache-aside (lazy loading)

**Golden answer**

> Application reads from cache first; on miss, loads from DB and populates cache.

**Pros**

* Simple
* Cache failures don’t break system

**Cons**

* Cache miss penalty
* Stale data risk

---

### Q6. Write-through vs write-behind

**Write-through**

* Write cache + DB synchronously
* Stronger consistency, higher latency

**Write-behind (write-back)**

* Write cache first, async DB write
* Faster writes, risk of data loss

**Principal signal**

> “Write-behind is a durability trade, not just a performance optimization.”

---

### Q7. Read-through cache

**Key insight**

> Cache itself loads data on miss (library / proxy pattern).

**Trap**

* Interviewer asks: *“Is read-through better than cache-aside?”*
  Correct answer: *“It depends on operational control and failure semantics.”*

---

## 4️⃣ Consistency & Correctness (Where seniors fail)

### Q8. How do you handle cache consistency?

**Expected approaches**

* TTL-based expiry
* Explicit invalidation
* Versioned keys
* Event-driven invalidation (CDC, Kafka)
* Cache warming

**Principal answer**

> “We don’t guarantee strong consistency; we guarantee bounded staleness.”

---

### Q9. How do you avoid stale reads?

**Good answers**

* Short TTL + async refresh
* Write-through for critical paths
* Cache stampede protection
* Read repair

**Red flag**

* “We just increase TTL”

---

## 5️⃣ Failure Modes (This decides senior vs principal)

### Q10. What happens if cache goes down?

**Expected**

* Cache is optional
* System falls back to DB
* DB protected via:

  * Rate limiting
  * Circuit breakers
  * Load shedding

**Kill shot**

> “Cache must fail open, but the database must not.”

---

### Q11. Explain cache stampede (thundering herd)

**Definition**

> Many clients recompute the same missing key simultaneously.

**Mitigations**

* Mutex / lock per key
* Request coalescing
* Early refresh
* Randomized TTL (jitter)

---

### Q12. What is cache penetration?

**Answer**

* Requests for non-existent keys hammer DB

**Fix**

* Cache nulls (with short TTL)
* Bloom filters

---

## 6️⃣ Scaling & Performance

### Q13. How does a distributed cache scale?

**Expected**

* Sharding (consistent hashing)
* Replication
* Client-side vs proxy routing

**Principal insight**

> “Scaling cache is mostly about key distribution, not memory.”

---

### Q14. How does Redis Cluster work at a high level?

**Expected**

* Hash slots (16384)
* Keys mapped to slots
* Slots mapped to nodes
* Replica for failover

**Trap**

* “Redis Cluster guarantees strong consistency” ❌

---

### Q15. Hot keys problem?

**Answer**

* Skewed access pattern
* Single shard overloaded

**Mitigation**

* Key splitting
* Local cache in front
* Replication of hot keys
* Request fan-out control

---

## 7️⃣ Eviction & Memory

### Q16. How does eviction work?

**Expected**

* LRU / LFU / TTL
* Eviction policies matter more than size

**Principal one-liner**

> “Eviction policy is a business decision disguised as a technical one.”

---

## 8️⃣ Security & Operations

### Q17. What about security?

**Expected**

* TLS in transit
* Auth (ACLs)
* Network isolation
* No PII caching (or encrypted values)

---

### Q18. What metrics do you monitor?

**Must mention**

* Hit ratio
* Evictions
* Latency
* Memory usage
* Connection count
* Command latency (Redis)

**Principal signal**

> “Hit ratio alone is meaningless without DB load correlation.”

---

## 9️⃣ Design Question (Always asked)

### Q19. Design a cache for a high-traffic product catalog

**Expected structure**

1. Cache-aside
2. TTL + versioned keys
3. Event-driven invalidation
4. Hot key mitigation
5. Graceful degradation

**Bad answer**

* “Use Redis with TTL”

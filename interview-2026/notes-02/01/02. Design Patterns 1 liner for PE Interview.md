# Principal Engineer – Design Patterns One‑Liners (Interview Signals)

> **How to use this**: Open with **one sharp line**, then expand only if asked. These are written to signal *judgment, trade‑offs, and scale* — not textbook recall.

---

## CREATIONAL PATTERNS

### Singleton

* **One‑liner**: *Singleton controls global access to a shared resource, but at scale it’s really about lifecycle, contention, and testability trade‑offs.*
* Alt: *I treat Singleton as a resource‑management decision, not a convenience pattern.*

### Factory Method

* **One‑liner**: *Factory Method defers object creation to subclasses so behavior varies without changing callers.*
* Alt: *It’s about pushing volatility in creation logic to the edges.*

### Abstract Factory

* **One‑liner**: *Abstract Factory creates families of related objects while keeping the client decoupled from concrete implementations.*
* Alt: *I use it when consistency across variants matters more than flexibility.*

### Builder

* **One‑liner**: *Builder separates object construction from representation, especially when invariants matter.*
* Alt: *It’s my go‑to when constructors start encoding business rules.*

### Prototype

* **One‑liner**: *Prototype creates objects by cloning, optimizing creation cost when setup is expensive.*
* Alt: *It trades construction clarity for performance and flexibility.*

---

## STRUCTURAL PATTERNS

### Adapter

* **One‑liner**: *Adapter translates one interface into another without changing either side.*
* Alt: *It’s a compatibility layer that protects core systems from external churn.*

### Bridge

* **One‑liner**: *Bridge decouples abstraction from implementation so both can evolve independently.*
* Alt: *I use it to avoid inheritance explosions across dimensions.*

### Composite

* **One‑liner**: *Composite lets clients treat individual objects and object trees uniformly.*
* Alt: *It models part‑whole hierarchies without special‑case logic.*

### Decorator

* **One‑liner**: *Decorator adds behavior dynamically without modifying the original class.*
* Alt: *It’s composition over inheritance, applied incrementally.*

### Facade

* **One‑liner**: *Facade provides a simplified interface over complex subsystems.*
* Alt: *It’s an API boundary pattern, not just a convenience wrapper.*

### Flyweight

* **One‑liner**: *Flyweight minimizes memory by sharing immutable state across many objects.*
* Alt: *I use it when scale turns object creation into a memory problem.*

### Proxy

* **One‑liner**: *Proxy controls access to an object, often adding latency, security, or lifecycle management.*
* Alt: *At scale, it’s where cross‑cutting concerns actually live.*

---

## BEHAVIORAL PATTERNS

### Strategy

* **One‑liner**: *Strategy encapsulates interchangeable algorithms behind a common interface.*
* Alt: *It isolates business volatility from stable orchestration logic.*

### State

* **One‑liner**: *State models behavior as a function of lifecycle, making transitions explicit and safe.*
* Alt: *It replaces conditionals with a governed state machine.*

### Command

* **One‑liner**: *Command encapsulates an action as an object, enabling queuing, retries, and undo.*
* Alt: *It’s how operations become first‑class citizens.*

### Observer

* **One‑liner**: *Observer enables event‑driven reactions without tight coupling.*
* Alt: *It’s the foundation of reactive and event‑driven systems.*

### Mediator

* **One‑liner**: *Mediator centralizes complex interactions to prevent object graphs from becoming tangled.*
* Alt: *It trades direct coupling for orchestration clarity.*

### Memento

* **One‑liner**: *Memento captures state for rollback without exposing internals.*
* Alt: *It’s controlled time travel, not persistence.*

### Iterator

* **One‑liner**: *Iterator abstracts traversal without exposing underlying structure.*
* Alt: *It separates data organization from access patterns.*

### Chain of Responsibility

* **One‑liner**: *Chain of Responsibility passes requests through handlers until one can process it.*
* Alt: *It’s a linearized decision pipeline.*

### Template Method

* **One‑liner**: *Template Method defines algorithm structure while allowing steps to vary.*
* Alt: *It enforces invariants while enabling controlled extensibility.*

### Visitor

* **One‑liner**: *Visitor separates algorithms from object structures when behavior changes more often than data.*
* Alt: *It optimizes for adding operations, not types.*

### Interpreter

* **One‑liner**: *Interpreter defines a grammar and evaluation model for domain‑specific expressions.*
* Alt: *It’s useful until performance or complexity demands a real parser.*

## CONCURRENCY & SYSTEM‑LEVEL PATTERNS (Principal Bonus)

### Immutable Object

* **One‑liner**: *Immutability simplifies concurrency by eliminating shared mutable state.*

### Object Pool

* **One‑liner**: *Object Pool manages expensive resources, but risks contention and hidden coupling.*

### Bulkhead

* **One‑liner**: *Bulkhead isolates failures to prevent systemic collapse.*

### Circuit Breaker

* **One‑liner**: *Circuit Breaker fails fast to protect systems under stress.*

### Saga

* **One‑liner**: *Saga coordinates distributed state transitions without global transactions.*

### Event Sourcing

* **One‑liner**: *Event Sourcing treats state as a projection of immutable events.*

### CQRS

* **One‑liner**: *CQRS separates read and write models to scale independently.*

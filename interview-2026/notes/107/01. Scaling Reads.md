**Scaling Reads** is a classic system-design pattern for handling **read-heavy workloads** without melting your primary database.

Here’s a clean, practical breakdown 👇

---

## 📈 The Problem

* Most real systems are **read-heavy** (90%+ reads).
* A single database (or primary node) becomes the bottleneck.
* Scaling the DB vertically gets expensive and hits limits.

---

## 🧩 Scaling Reads: Core Patterns

### 1️⃣ **Read Replicas (Leader–Follower)**

**Pattern:**

* One **primary (leader)** handles writes
* Multiple **read replicas (followers)** serve reads

**Flow:**

```
Write → Primary
Read  → Replica(s)
```

**Pros**

* Easy to add more replicas
* Linear read scaling
* Minimal app changes

**Cons**

* **Replication lag** → stale reads
* Read-after-write consistency issues

**Use when**

* Eventual consistency is acceptable
* Reads massively outnumber writes

---

### 2️⃣ **Caching (Most Important Pattern)**

**Pattern:**
Serve reads from **cache** before hitting the DB.

**Common caches**

* Redis
* Memcached
* In-memory (Caffeine, Guava)

**Flow:**

```
Read → Cache → DB (on miss)
```

**Pros**

* Orders of magnitude faster
* Offloads DB almost completely
* Cheap to scale

**Cons**

* Cache invalidation is hard
* Needs TTL / eviction strategy

**Use when**

* Data is frequently accessed
* Slight staleness is OK

💡 *Rule of thumb:*

> **Cache before you replicate**

---

### 3️⃣ **CDN (For Static / Semi-Static Reads)**

**Pattern:**
Move reads **closer to users**.

**Examples**

* Images
* Static files
* Public APIs with caching headers

**Pros**

* Near-zero latency
* Removes load from backend

**Cons**

* Not suitable for highly dynamic data

---

### 4️⃣ **Query Optimization & Denormalization**

**Pattern:**
Make reads cheaper instead of adding infra.

**Techniques**

* Proper indexing
* Covering indexes
* Pre-computed aggregates
* Read-optimized tables

**Pros**

* No extra infrastructure
* Immediate performance gain

**Cons**

* Write complexity increases

---

### 5️⃣ **CQRS (Command Query Responsibility Segregation)**

**Pattern:**
Separate **write model** and **read model**.

**Flow:**

```
Writes → Write DB
Events → Read DB(s)
Reads  → Read DB
```

**Pros**

* Independent scaling
* Read models optimized for queries

**Cons**

* Complex
* Eventual consistency by design

**Use when**

* Very complex queries
* Massive scale (think Uber, Amazon)

---

### 6️⃣ **Sharding for Reads (Advanced)**

**Pattern:**
Split data across nodes, then replicate per shard.

**Pros**

* Extreme scale

**Cons**

* High operational complexity
* Cross-shard queries are painful

---

## 🔁 Common Read Consistency Strategies

* **Read-your-writes** → route reads to primary
* **Session stickiness** → same replica per session
* **Versioned reads** → read only if version ≥ X

---

## 🏗️ Typical Real-World Stack

```
Client
  ↓
CDN
  ↓
Cache (Redis)
  ↓
Read Replica(s)
  ↓
Primary DB
```

---

## 🎯 Key Takeaways

* Start with **caching**
* Add **read replicas** next
* Optimize queries before adding complexity
* Accept **eventual consistency** where possible
* Scale infrastructure *only after* scaling design

Let’s walk through the **two most common (and painful) read-scaling failure modes**, how they show up, and how people actually fix them.

---

## 1️⃣ Cache Stampede (aka Thundering Herd)

### 🔥 What happens

* A hot key expires or gets evicted
* **Thousands of requests miss the cache at once**
* All of them hit the DB simultaneously
* DB falls over → cascading failures

**Timeline**

```
Cache entry expires at T
1000 concurrent requests
→ Cache MISS x1000
→ DB QPS spikes
→ Latency ↑↑↑
→ Timeouts / errors
```

---

### 🚨 Symptoms in production

* Sudden DB CPU spike
* Read latency jumps sharply
* Cache hit ratio drops briefly
* Happens at **TTL boundaries**

---

### 🧯 Mitigations (real-world)

#### ✅ 1. Request Coalescing (Single Flight)

Only **one request** recomputes the value.

```
First request → DB
Others wait → reuse result
```

* Redis: `SETNX` / locks
* Java: `Caffeine cache.get(key, loader)`
* Go: `singleflight`

👉 **Most effective fix**

---

#### ✅ 2. Soft TTL (Stale-While-Revalidate)

* Serve **stale data**
* Refresh cache **in background**

```
If expired:
  return old value
  async refresh
```

Used heavily by:

* CDNs
* High-scale APIs

---

#### ✅ 3. TTL Jitter

Randomize expiration time.

```
TTL = baseTTL ± random
```

Prevents mass expiration at the same second.

---

#### ✅ 4. Pre-warming / Scheduled Refresh

* Refresh hot keys before TTL expiry
* Cron / background workers

---

### 🧠 Interview one-liner

> “To avoid cache stampede, I use request coalescing and serve stale data while refreshing in the background.”

---

## 2️⃣ Replica Lag (Eventually Consistent Reads)

### 🔥 What happens

* Write goes to **primary**
* Read hits **replica**
* Replica hasn’t caught up yet
* Client sees **old data**

```
Write → Primary (OK)
Read → Replica (STALE)
```

---

### 🚨 Symptoms in production

* “I just updated this but can’t see it”
* Flaky tests
* Inconsistent UI behavior
* Hard-to-reproduce bugs

---

### 🧯 Mitigations (practical)

#### ✅ 1. Read-Your-Writes Routing

After a write:

* Route reads to **primary**
* Or sticky session

```
User writes → primary
User reads → primary (for N seconds)
```

Simple and effective.

---

#### ✅ 2. Session Stickiness

Same user → same replica.

* Reduces inconsistency
* Doesn’t fully eliminate lag issues

---

#### ✅ 3. Version / Timestamp-Based Reads

* Track version returned from write
* Only read from replica if version ≥ expected

Advanced, but clean.

---

#### ✅ 4. Bounded Staleness

Only read replicas with lag < X ms.

* Some DBs expose replica lag metrics
* App-side routing

---

### 🧠 Interview one-liner

> “To handle replica lag, I ensure read-your-writes by routing post-write reads to the primary.”

---

## 3️⃣ Bonus: Cache + Replica = Double Trouble 😈

### Worst-case scenario

* Cache expires
* Stampede hits
* Reads go to **lagging replica**
* Cache gets repopulated with **stale data**
* Stale data lives for full TTL 😬

---

### Fix

* On cache miss:

    * Read from **primary** for critical keys
    * Or validate version before caching
* Use **short TTL + background refresh**

---

| Cache stampede | Simultaneous cache misses | Single-flight, soft TTL   |
| Replica lag    | Async replication         | Read-your-writes          |
| Stale cache    | Cache + replica lag       | Versioning, primary reads |

## 1️⃣ Replica Lag ⇢ Consumer Lag

### What replica lag looks like

* Writes are ahead
* Reads see old data

### Kafka equivalent

* Producer is faster than consumer
* **Offsets fall behind**

```
Producer → Partition
           ↑
        Consumer (slow)
```

### Why it happens

* Slow processing logic
* Downstream dependency (DB, API) slowing consumer
* Rebalances
* GC pauses / crashes

### Impact

* Events processed **late**
* SLA violations
* Downstream systems drift out of sync

### Same fix philosophy

**Don’t “add more replicas/consumers blindly.”**

Instead:

* Speed up processing
* Reduce per-message cost
* Isolate slow paths
* Increase parallelism *only if partitioning allows it*

👉 Exactly like routing reads to primary for read-your-writes.

## 2️⃣ Hot Keys ⇢ Hot Partitions (Your Real Pain 😄)

You already described this perfectly in your earlier message.

### Read-scaling version

* One cache key / DB row gets 80% of traffic
* One replica melts
* Others idle

### Kafka version

* One partition gets most of the messages
* One consumer stuck
* Others idle

```
Partition 3 → 90% traffic → Consumer 3 (DYING)
Partitions 0–2 → chill
```

### Critical insight

> **Kafka parallelism is bounded by partition count**

Just like:

> **DB parallelism is bounded by shard / key distribution**

Adding consumers ≠ more throughput
Same as adding read replicas doesn’t help a hot key.

## 3️⃣ Cache Stampede ⇢ Consumer Restart Storm

This one’s subtle but deadly.

### Cache stampede

* Cache expires
* 1000 requests hit DB at once

### Kafka equivalent

* Consumer crashes / rebalance
* Offsets not committed properly
* Consumer restarts and **replays a large backlog**
* Downstream systems get flooded

```
Consumer crash
→ Rebalance
→ Replay from older offset
→ DB / API overload
```

### Symptoms

* Lag suddenly spikes
* DB gets hammered
* Duplicate work
* Cascading failures

## 4️⃣ Cache TTL Jitter ⇢ Offset Commit Strategy

### Cache lesson

* Don’t expire everything at once

### Kafka lesson

* Don’t:

    * Commit offsets too frequently (overhead)
    * Commit too late (large replays)

**Balanced strategy**

* Commit in batches
* Commit after durable side effects
* Use idempotency downstream

Same principle: **smooth load, avoid spikes**.

## 5️⃣ Stale Cache ⇢ Reprocessing Old Events

### Cache world

* Cache repopulated with stale replica data

### Kafka world

* Consumer restarts
* Reprocesses old events
* Writes outdated state

### Fix in both worlds

* **Versioning**
* **Idempotency**
* **Last-write-wins logic**

Benefits:

Downside:

Alright, let’s get very **concrete and battle-tested**.
Hot partitions are one of those problems where **the fix is architectural, not operational**.

Below are **exact migration strategies**, ordered from *least risky* to *most invasive*, with **when to use each** and **how to execute them safely in prod**.

---

## 0️⃣ First: Prove It’s a Hot Partition (Not a Slow Consumer)

Before migration, confirm:

* Lag is **isolated to 1–2 partitions**
* CPU / threads idle on other consumers
* Producer throughput skewed to same partition
* Key distribution is uneven

If yes → proceed.
If no → optimize consumer first.

---

## 1️⃣ Key Salting (Fastest, Most Common Fix)

### 🧠 Idea

Break one hot logical key into **multiple physical keys**.

```
orderId → orderId#0..N
userId  → userId#hash(userId)%N
```

---

### 🛠 How to migrate safely

#### Step 1: Increase partitions (if needed)

```bash
kafka-topics --alter --topic orders --partitions 24
```

#### Step 2: Producer change (dual-write aware)

```java
salt = hash(key) % N
newKey = key + "#" + salt
```

#### Step 3: Consumer change

* Strip salt
* Aggregate downstream if needed

#### Step 4: Gradual rollout

* 1% traffic
* Monitor lag
* Ramp up

---

### ✅ Pros

* Simple
* Massive throughput gain
* No Kafka magic needed

### ❌ Cons

* Ordering across salted keys is lost
* Consumers get more complex

### 🎯 Use when

* Ordering is **per logical key**, not global
* One key causes 80% traffic

---

## 2️⃣ Two-Topic Migration (Controlled & Clean)

### 🧠 Idea

Leave the hot partition alone.
Move **new traffic** to a better-partitioned topic.

---

### 🛠 Migration steps

```
old-topic (hot)
new-topic (better partitioning)
```

#### Step 1: Create new topic

```bash
orders-v2 (48 partitions)
```

#### Step 2: Dual-produce

```
Old messages → old-topic
New messages → new-topic
```

#### Step 3: Consumers read both

* Two consumer groups
* Or one service reading both topics

#### Step 4: Drain old topic

* Wait until lag = 0
* Kill old consumers
* Decommission topic

---

### ✅ Pros

* Zero ordering risk
* Clean design
* No reprocessing surprises

### ❌ Cons

* More infra
* Temporary complexity

### 🎯 Use when

* You need **strict ordering**
* You want clean partitioning going forward

---

## 3️⃣ Fan-Out Topic (Split Hot Path Only)

### 🧠 Idea

Split the **hot flow** from the normal flow.

```
Main topic → Router
             ├─ hot-topic
             └─ normal-topic
```

---

### 🛠 Execution

* Producer detects hot key
* Routes to `hot-topic`
* Other keys go to main topic
* Consumers scale independently

---

### ✅ Pros

* Surgical fix
* No impact to cold keys

### ❌ Cons

* Routing logic complexity
* Operational overhead

### 🎯 Use when

* 1–2 keys dominate traffic
* Hot keys are known

---

## 4️⃣ Repartition via Kafka Streams (Advanced)

### 🧠 Idea

Let Kafka re-partition for you.

```
Input topic
→ Streams repartition
→ Output topic
```

---

### 🛠 Example

```java
stream
  .selectKey((k, v) -> newBalancedKey(v))
  .to("repartitioned-topic");
```

---

### ✅ Pros

* No producer change
* Controlled transformation
* Exactly-once support

### ❌ Cons

* Streams operational complexity
* Extra hop & latency

### 🎯 Use when

* You don’t control producers
* You already run Kafka Streams

---

## 5️⃣ Stateful Consumer Sharding (Last Resort)

### 🧠 Idea

Ignore Kafka’s partitioning and shard **inside** the consumer.

```
Partition → Dispatcher → Worker pool
```

---

### ⚠️ Why it’s dangerous

* You fight Kafka’s model
* Hard ordering guarantees
* Complex failure handling

### 🎯 Use only when

* Partitioning **cannot change**
* Low ordering guarantees
* Temporary mitigation

---

## 6️⃣ Emergency Mitigations (While Migrating)

These **do not fix** hot partitions but buy time:

* Batch processing
* Async downstream calls
* Backpressure
* Pause/resume partitions
* Rate limit producer for hot keys

## 1️⃣ The Problem We’re Solving

Bad partitioning:

```text
partition = hash(userId) % N
```

If:

* One `userId` / `cityId` is hot
* Ordering is required per user (not globally)

👉 One partition melts.

---

## 2️⃣ Design Goals

We want:

* **Even distribution**
* **Per-logical-key ordering**
* **Hot-key isolation**
* **Backward compatibility**

Solution:

> **Logical key → multiple physical shards (salts)**
> but **sticky per logical key** during a window.

---

## 3️⃣ Custom Partitioner (Java)

### 🧠 Strategy

* For each logical key:

    * Assign a **salt** from a small range
    * Keep it **stable** for a time window
* Hash `(key + salt)` → partition

This spreads load *without destroying ordering completely*.

---

### 🧩 Code: Sticky-Salted Partitioner

```java
package com.example.kafka;

import org.apache.kafka.clients.producer.Partitioner;
import org.apache.kafka.common.Cluster;
import org.apache.kafka.common.utils.Utils;

import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.ThreadLocalRandom;

public class StickySaltedPartitioner implements Partitioner {

    // Number of shards per logical key
    private static final int SALT_RANGE = 8;

    // How long we keep a salt stable (ms)
    private static final long STICKY_WINDOW_MS = 60_000;

    private static class SaltState {
        final int salt;
        final long expiresAt;

        SaltState(int salt, long expiresAt) {
            this.salt = salt;
            this.expiresAt = expiresAt;
        }
    }

    private final Map<String, SaltState> saltCache = new ConcurrentHashMap<>();

    @Override
    public int partition(
            String topic,
            Object key,
            byte[] keyBytes,
            Object value,
            byte[] valueBytes,
            Cluster cluster
    ) {
        int numPartitions = cluster.partitionCountForTopic(topic);

        if (key == null) {
            // No key → random partition
            return ThreadLocalRandom.current().nextInt(numPartitions);
        }

        String logicalKey = key.toString();
        long now = System.currentTimeMillis();

        SaltState state = saltCache.compute(logicalKey, (k, existing) -> {
            if (existing == null || existing.expiresAt < now) {
                int newSalt = ThreadLocalRandom.current().nextInt(SALT_RANGE);
                return new SaltState(newSalt, now + STICKY_WINDOW_MS);
            }
            return existing;
        });

        String physicalKey = logicalKey + "#" + state.salt;
        return Utils.toPositive(Utils.murmur2(physicalKey.getBytes())) % numPartitions;
    }

    @Override
    public void configure(Map<String, ?> configs) {}

    @Override
    public void close() {}
}
```

---

## 4️⃣ How This Preserves Sanity

### ✅ Ordering

* Ordering is preserved **within the sticky window**
* Acceptable for:

    * Metrics
    * Aggregations
    * Session-based logic

### ✅ Load Distribution

* Hot key → spread across `SALT_RANGE` partitions
* Cold keys → unchanged

### ⚠️ Trade-off

* Ordering across salt rotations is **not guaranteed**
* This must be acceptable by design

---

## 🎯 What You’re Detecting (Be Precise)

You are NOT detecting:

* High traffic overall ❌
* High consumer lag overall ❌

You ARE detecting:

* **Skew** → one key / partition dominates others

> Hot key = **imbalance**, not just volume.

---

## 1️⃣ Partition Throughput Skew (Primary Signal)

### 📊 Metric

* Messages per second **per partition**

**From**

* Kafka broker metrics
* JMX: `MessagesInPerSec`

---

### 🚨 Detection rule

```text
max_partition_rate > 5 × median_partition_rate
```

Or:

```text
top_partition > 40% of total topic traffic
```

---

### 🧠 Why this works

* Even load → flat distribution
* Hot key → sharp spike on one partition

---

## 2️⃣ Consumer Lag Skew (Most Reliable Signal)

### 📊 Metric

* Consumer lag **per partition**

**From**

* `records-lag-max`
* Consumer group metrics

---

### 🚨 Detection rule

```text
lag(partition_i) > 10 × median_lag
AND
lag is monotonically increasing
```

---

### 🧠 Key insight

* High lag alone is noise
* High lag on ONE partition is fire

---

## 3️⃣ Processing Time Skew (Hidden Hot Key Detector)

### 📊 Metric

* Avg / p95 processing time **per partition**

---

### 🚨 Detection rule

```text
p95_processing_time(partition_i) >
  3 × median_processing_time
```

---

### 🧠 Why this matters

Sometimes:

* Message rate is fine
* Payload size or downstream calls differ
* Partition still becomes hot

---

## 4️⃣ Key Cardinality Collapse (Early Warning)

### 📊 Metric

* Unique keys per partition per time window

---

### 🚨 Detection rule

```text
unique_keys(partition_i) << expected_baseline
```

Example:

* Usually 50k keys/min
* Suddenly 500 keys/min

👉 One key dominating traffic.

---

## 5️⃣ Rebalance Frequency Spike (Secondary Signal)

### 📊 Metric

* Consumer group rebalances per hour

---

### 🚨 Detection rule

```text
rebalances > baseline + 2σ
```

---

### 🧠 Why it correlates

Hot partitions cause:

* Long processing
* Missed heartbeats
* Rebalances
* More lag
* Death spiral

---

## 6️⃣ End-to-End Lag vs Kafka Lag Gap

### 📊 Metric

* Event timestamp vs processing timestamp

---

### 🚨 Detection rule

```text
end_to_end_lag >> kafka_lag
```

---

### 🧠 Interpretation

* Consumer is alive
* Kafka isn’t the bottleneck
* One key causes slow downstream processing

---

## 7️⃣ Broker Resource Skew (Last Resort)

### 📊 Metrics

* Network IO per partition
* Disk IO per partition

---

### 🚨 Detection rule

```text
partition_io > 3 × broker average
```

---

## 🔁 Turning Metrics into Automation

### Simple Hot Partition Score

```text
hot_score =
  w1 * throughput_skew +
  w2 * lag_skew +
  w3 * processing_skew
```

Trigger when:

```text
hot_score > threshold for N minutes
```

---

## 🔔 Alerting Strategy (Avoid Noise)

### ❌ Don’t alert on

* Absolute lag
* Topic-level traffic

### ✅ Alert on

* Skew ratios
* Sustained imbalance (≥ 5 mins)

---

## 🛠️ What Happens After Detection

Once detected:

1. Flag key(s)
2. Enable salting / routing
3. Increase partitions if needed
4. Monitor convergence

Detection → mitigation → normalization

## I. JVM Internals & Performance

**1. Young vs Full GC**

> Young GC is allocation pressure; Full GC is *heap structure or promotion failure*. If I see frequent Full GCs, I assume sizing, tenure, or fragmentation—not allocation rate.

**2. GC pauses vs allocation rate**

> I optimize allocation rate first. GC pauses are a symptom; allocation behavior is the cause.

**3. G1 vs ZGC**

> G1 when I need predictability and maturity. ZGC when latency SLOs matter more than throughput and heap is large.

**4. Early memory pressure signals**

> Rising promotion rates, survivor overflow, allocation stalls, GC frequency—not just heap usage.

**5. Safepoints**

> They’re global coordination points. Too many means excessive deopts, biased lock revocation, or instrumentation.

**6. Object allocation**

> Bump-pointer allocation in Eden via TLABs. It’s fast—cleanup is the expensive part.

**7. GC thrashing**

> Allocation outpaces evacuation. Usually due to oversized object graphs or cache misuse.

**8. TLABs**

> They eliminate allocation contention. When disabled, allocation scalability collapses.

**9. High CPU, low throughput**

> GC, lock contention, or spinning. CPU ≠ progress.

**10. Boxing cost**

> Individually cheap, cumulatively deadly—especially in hot loops and collections.

**11. JIT behavior**

> Warm-up changes performance characteristics. Cold benchmarks lie.

**12. Flags never tuned blindly**

> GC, heap region sizing, compiler flags. Defaults encode years of scars.

**13. Heap vs off-heap**

> Heap gives safety; off-heap gives control. Off-heap shifts bugs to you.

**14. STW pauses**

> World stops to preserve invariants. Short pauses, but globally visible.

**15. When tuning backfires**

> When masking architectural problems with flags.

## II. Concurrency & Multithreading

**16. Java Memory Model**

> It defines visibility, not execution order. Assumptions cause bugs.

**17. `volatile`**

> Visibility + ordering, not atomicity.

**18. `synchronized` vs `ReentrantLock`**

> `synchronized` is simpler and often faster unless you need advanced control.

**19. CompletableFuture failures**

> Silent exceptions, thread pool misuse, lost backpressure.

**20. Starvation vs deadlock**

> Deadlock is mutual waiting. Starvation is unfair scheduling.

**21. Unbounded pools**

> Latency hides until memory explodes.

**22. Pool sizing**

> CPU = cores. IO = latency × throughput.

**23. False sharing**

> Cache line contention. Detected via perf counters, fixed via padding.

**24. Atomics bottleneck**

> High contention turns CAS into spin storms.

**25. Undetectable concurrency bugs**

> Timing-sensitive visibility issues.

**26. Backpressure**

> If queues grow, you don’t have backpressure—you have denial.

**27. Ignored interrupts**

> Cancellation breaks. Threads leak.

**28. Happens-before**

> If you didn’t establish it, the JVM owes you nothing.

**29. Async stack debugging**

> Logical flow ≠ call stack.

**30. Lock-free harm**

> When retries dominate useful work.

## III. Collections & Memory

**31. HashMap resize**

> Rehashing is CPU-heavy and latency-visible.

**32. ConcurrentHashMap**

> Striping + CAS avoids global locks.

**33. CopyOnWrite**

> Read-heavy, write-rare, latency-sensitive paths.

**34. Object graph cost**

> Traversal dominates memory, not object size.

**35. Cache GC pressure**

> Long-lived references prevent reclamation.

**36. LinkedList**

> Cache-unfriendly, pointer-heavy, almost never right.

**37. Weak/soft refs**

> GC heuristics ≠ cache policy.

**38. Cardinality explosion**

> Memory cliffs, GC collapse.

**39. Leak detection**

> Allocation trends + object lifetime analysis.

**40. LRU failure**

> Hot keys evict each other → churn.

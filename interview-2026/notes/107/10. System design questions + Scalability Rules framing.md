# 🧩 System Design Questions — *Scalability Rules Framing*

## How to Use This in Interviews

Before jumping to architecture, **say one sentence**:

> “I’ll frame this around bottlenecks, measurement, and change over time.”

That signals seniority immediately.

---

## 1. Design a URL Shortener

### Naive Approach (what juniors do)

* One DB
* One cache
* Done

### Scalability Rules Framing

**Start with**

> “The bottleneck will be writes and ID generation, not reads.”

**Key questions**

* Expected write QPS?
* Read/write ratio?
* Latency SLO?

### Design decisions

* Stateless API
* ID generation without coordination (Snowflake / hash-based)
* DB sharded by short-code prefix
* Cache in front for reads

### Scalability Rules Applied

* **Rule 3**: Remove centralized ID generator bottleneck
* **Rule 7**: Sharding allows change later

**Distinguished move**

> “If this grows 100×, where do we re-shard?”

## 2. Design a News Feed

### Core Insight

> Feeds fail due to **fan-out**, not storage.

### Two models

* Fan-out-on-write (push)
* Fan-out-on-read (pull)

### Scalability Rules Framing

**Say**

> “We’re trading write amplification for read latency.”

### Design

* Celebrities → fan-out-on-read
* Normal users → fan-out-on-write
* Hybrid model

### Bottlenecks

* Hot users
* Write storms
* Cache invalidation

### Rules Applied

* **Rule 3**: Bottlenecks move as usage changes
* **Rule 6**: Scale changes what “good” means

---

## 3. Design a Rate Limiter

### Key Insight

> Rate limiting is about *coordination cost*.

### Design choices

* Token bucket
* Sliding window
* Redis vs local counters

### Scalability Rules Framing

**Say**

> “Global consistency will cap throughput.”

### Design

* Local limits first
* Approximate global limits
* Periodic sync

### Rules Applied

* **Rule 3**: Avoid global locks
* **Rule 7**: Approximate consistency scales better

---

## 4. Design a Chat System

### Core Challenge

> Ordering + delivery + scale don’t coexist easily.

### Scalability Rules Framing

**Say**

> “We need to decide where ordering is required.”

### Design

* Per-conversation ordering
* Partition by conversation ID
* WebSocket fan-out
* Async persistence

### Failure handling

* Message replay
* Idempotent delivery

### Rules Applied

* **Rule 2**: Don’t overbuild global ordering
* **Rule 5**: Ownership by conversation

---

## 5. Design a File Storage System (Google Drive)

### Bottleneck

* Metadata, not blobs

### Scalability Rules Framing

**Say**

> “Storage systems usually fail on metadata operations.”

### Design

* Blob store (S3-like)
* Metadata DB sharded by user
* CDN for downloads

### Rules Applied

* **Rule 3**: Remove metadata hot spots
* **Rule 1**: Measure metadata QPS separately

---

## 6. Design a Ride-Sharing System (Uber-like)

### Core Challenge

> Real-time matching under skewed load.

### Scalability Rules Framing

**Say**

> “Hot regions dominate system behavior.”

### Design

* Geo-sharding
* Localized matching
* Event-driven updates

### Rules Applied

* **Rule 3**: Remove cross-region coordination
* **Rule 6**: Scale changes traffic shape

---

## 7. Design a Logging / Metrics System

### Key Insight

> Write-heavy, bursty traffic.

### Scalability Rules Framing

**Say**

> “Ingress must absorb spikes without blocking producers.”

### Design

* Kafka ingestion
* Tiered storage
* Sampling at edge

### Rules Applied

* **Rule 3**: Buffer before bottleneck
* **Rule 6**: Observability must scale with system

---

## 8. Design a Payment System

### Core Truth

> Correctness > availability.

### Scalability Rules Framing

**Say**

> “This is a business consistency decision.”

### Design

* Strong consistency on writes
* Idempotent APIs
* Async downstream processing

### Rules Applied

* **Rule 4**: Don’t scale prematurely
* **Rule 6**: Scale changes correctness trade-offs

---

## 9. Design a Recommendation System

### Bottleneck

* Data freshness vs compute cost

### Scalability Rules Framing

**Say**

> “We’re trading real-time accuracy for scalability.”

### Design

* Batch pre-compute
* Online re-ranking
* Cache results

### Rules Applied

* **Rule 6**: “Good” changes at scale
* **Rule 7**: Allow model evolution

---

## 10. Design Kafka at Scale (Bonus)

### Key Insight

> Partitions are scalability units.

### Scalability Rules Framing

**Say**

> “Partitioning is an irreversible decision.”

### Design

* Partition by balanced key
* Monitor lag per partition
* Repartition via migration topics

### Rules Applied

* **Rule 2**: Avoid early lock-in
* **Rule 3**: Remove hot partitions

* Trade-off articulation
* Change-over-time thinking
* Failure modes
* Measurement-first mindset

They do *not* care about:

* Exact technologies
* Perfect diagrams
* Buzzwords

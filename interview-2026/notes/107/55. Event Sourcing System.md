# 55️⃣ Event Sourcing System — Deep Dive (Principal+)

## What Event Sourcing *Actually* Is

Event sourcing means:

> **The source of truth is an append-only log of events, not current state.**

State is **derived**, **rebuildable**, and **secondary**.

This is not:

* “Just using Kafka”
* “Just audit logs”
* “Just CQRS”

Principal reframe:

> **“We’ve chosen time as our primary data model.”**

---

## Why Teams Choose Event Sourcing (Real Reasons)

### Legitimate Drivers

* Strong auditability requirements
* Need for temporal queries (“what did we know then?”)
* Complex domain workflows
* Replay for recovery or reprocessing

### Unspoken Drivers

* Fear of schema migrations
* Desire to avoid distributed transactions
* “Netflix does it”

Principal smell:

> **“We’re optimizing for elegance over operability.”**

---

## Core Architecture

### Event Store

* Append-only
* Ordered per aggregate
* Immutable

### Aggregates

* Logical entity boundaries
* Events scoped per aggregate ID
* Ordering matters **within**, not across aggregates

### Projections (Read Models)

* Built asynchronously
* Often multiple
* Eventually consistent

Principal translation:

> **“Writes are simple. Reads are everywhere.”**

---

## The Real Tradeoffs (Where Interviews Are Won)

### 1️⃣ Event Versioning (Forever Problem)

* Events cannot change
* Schemas evolve
* Consumers must handle all versions

Principal line:

> **“Event schemas are forever APIs.”**

Follow-up:

> **“Backward compatibility is not optional.”**

---

### 2️⃣ Replay Is an Outage

* Rebuilding projections means:

    * Heavy reads
    * Long-running jobs
    * Potential production impact

Principal line:

> **“Replay is a controlled outage.”**

Interview dagger:

> **“How do we replay without taking prod down?”**

---

### 3️⃣ Storage Growth Is Unbounded

* Every change is stored
* Compaction is semantic, not technical
* Deletion is hard (GDPR)

Principal line:

> **“Event stores only grow.”**

---

### 4️⃣ Debugging Is Time Travel

* Bugs may exist in projections, not events
* Fixing logic requires replay
* Observability spans time, not state

Principal line:

> **“Bugs hide in history.”**

---

## Consistency Model (Subtle but Critical)

* Strong consistency **within an aggregate**
* Eventual consistency **across aggregates**
* Read models lag writes

Principal framing:

> **“Consistency boundaries are explicit.”**

Red flag question:

> **“Which user actions tolerate stale reads?”**

---

## Failure Modes (This Is the Money Section)

### 1️⃣ Poison Events

* Bad event breaks all downstream consumers
* Replays fail repeatedly

Principal line:

> **“One bad event can brick the system.”**

Mitigation:

* Dead-letter events
* Versioned consumers
* Event validation on write

---

### 2️⃣ Projection Drift

* Different projections interpret events differently
* Logic divergence over time

Principal line:

> **“Projections are forks of reality.”**

---

### 3️⃣ Replay Cost Explosion

* New projections require full history
* Old data has different assumptions

Principal line:

> **“History gets more expensive over time.”**

---

### 4️⃣ Idempotency Bugs

* Reprocessing duplicates
* Exactly-once is an illusion

Principal line:

> **“At-least-once is the default.”**

---

## Operational Reality (What Docs Don’t Tell You)

### Tooling Gaps

* Schema registry becomes critical
* Versioning discipline required
* Consumer compatibility testing needed

### Human Cost

* New engineers struggle
* Debugging requires domain expertise
* On-call complexity increases

Principal line:

> **“Event sourcing raises the cognitive load floor.”**

---

## When Event Sourcing Is Worth It

### Good Fits

* Financial systems
* Ledger-style domains
* Compliance-heavy systems
* Long-lived business entities

### Bad Fits

* CRUD-heavy apps
* Simple workflows
* Teams without strong discipline
* Systems needing strong read consistency

Principal filter:

> **“If replay scares you, don’t event-source.”**

---

## Event Sourcing vs Traditional CRUD

| Dimension | CRUD        | Event Sourcing |
| --------- | ----------- | -------------- |
| Writes    | Complex     | Simple         |
| Reads     | Simple      | Complex        |
| Storage   | Bounded     | Unbounded      |
| Debugging | State-based | Time-based     |
| Recovery  | Hard        | Possible       |
| Ops       | Easier      | Harder         |

Principal summary:

> **“We moved complexity from writes to everything else.”**

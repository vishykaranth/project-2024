Below is a **Scalability Anti-Patterns Checklist** — concise, brutal, and practical. If you recognize even **3–4** of these in a design, you’re already in danger.

---

# 🚨 Scalability Anti-Patterns Checklist

Use this during **design reviews, incident RCA, interviews, and architecture docs**.

---

## I. Coordination & Control Plane Anti-Patterns

☐ **Centralized ID generator**
One service or DB sequence allocates IDs for all writes.

☐ **Global lock or mutex**
Anything that serializes progress across nodes.

☐ **Leader does all writes**
Leader-based systems without write offloading.

☐ **Strong consistency everywhere**
Using linearizability when eventual consistency is sufficient.

☐ **Single scheduler / cron master**
One node decides work for the entire system.

☐ **Synchronous fan-out**
Request waits on N downstream calls.

☐ **Distributed transactions by default**
2PC used on hot paths.

☐ **Global ordering requirement**
Ordering enforced when per-key ordering would suffice.

---

## II. Data & Storage Anti-Patterns

☐ **Auto-increment primary keys**
Creates write hotspots and index contention.

☐ **Monolithic database**
One DB for all workloads.

☐ **Shard-by-user-ID without skew analysis**
Power users destroy balance.

☐ **Early hard sharding**
Shard scheme that can’t evolve.

☐ **Re-sharding requires downtime**
No migration path designed.

☐ **Metadata and data mixed together**
Small hot rows block large cold blobs.

☐ **Indexes on everything**
Write amplification explosion.

☐ **Large transactional tables**
Slow scans, cache misses.

---

## III. Caching Anti-Patterns

☐ **Cache without eviction strategy**
Memory fills → crash.

☐ **Same TTL for everything**
Thundering herd guaranteed.

☐ **No cache warming**
Cold start outages.

☐ **Cache as source of truth**
Inconsistent state inevitable.

☐ **Write-through cache on hot path**
Adds latency and coupling.

☐ **Distributed cache for ultra-hot keys**
Network hop becomes bottleneck.

☐ **Cache invalidation via broadcast**
Storm under load.

☐ **No negative caching**
Repeated misses hammer backend.

---

## IV. Messaging & Streaming Anti-Patterns (Kafka-heavy)

☐ **Low partition count “for safety”**
Caps throughput forever.

☐ **Key choice without traffic analysis**
Hot partitions guaranteed.

☐ **Assuming consumers can scale infinitely**
Partition limit ignored.

☐ **Retrying consumers without backoff**
Reprocessing storms.

☐ **Exactly-once everywhere**
Throughput collapse.

☐ **Large messages in Kafka**
Broker memory pressure.

☐ **Synchronous processing inside consumer**
Lag amplification.

☐ **No lag-based alerting**
Failures detected too late.

---

## V. Traffic & Load Anti-Patterns

☐ **Assuming uniform traffic**
Real traffic is spiky and skewed.

☐ **No rate limiting**
One bad client takes down system.

☐ **Retries without budgets**
Self-inflicted DDoS.

☐ **No backpressure**
Queues grow until crash.

☐ **Autoscaling without warm-up**
Scale too late.

☐ **Peak traffic tested only once**
Seasonal failures.

☐ **Load testing without production data**
Invalid conclusions.

---

## VI. Failure Handling Anti-Patterns

☐ **No timeouts**
Threads pile up.

☐ **Same timeout everywhere**
Cascading failures.

☐ **Fail open by default**
Silent corruption.

☐ **Fail closed by default**
Total outage.

☐ **Health checks that don’t exercise dependencies**
False positives.

☐ **Shared thread pools**
One slow path blocks all.

☐ **No circuit breakers**
Meltdown under partial failure.

☐ **Cold failover paths**
Untested, broken when needed.

---

## VII. Observability Anti-Patterns

☐ **Only average latency tracked**
Tail pain invisible.

☐ **No saturation metrics**
Blind to capacity limits.

☐ **High-cardinality labels everywhere**
Metrics system overload.

☐ **Logs instead of metrics**
Cost explosion.

☐ **Dashboards without alerts**
Pretty but useless.

☐ **Alerts without ownership**
Ignored.

☐ **No per-partition metrics (Kafka)**
Hot spots invisible.

---

## VIII. Organizational / Process Anti-Patterns (Often Missed)

☐ **Architecture decided too early**
Locks future scale.

☐ **No migration playbooks**
Change becomes scary.

☐ **Optimizing for happy path only**
Failures dominate at scale.

☐ **Scaling decisions owned by one team**
Cross-org bottlenecks.

☐ **No load ownership**
Everyone assumes “someone else.”

☐ **Hero debugging culture**
Systems never improve.

---

## 🧠 Distinguished Engineer Rule of Thumb

If you see:

* Global coordination
* Irreversible partitioning
* No migration story
* Assumed uniform traffic

👉 **You’re not looking at a scalability problem yet — you’re looking at a future outage.**

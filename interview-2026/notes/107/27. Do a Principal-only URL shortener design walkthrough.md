# Principal-Only URL Shortener Design Walkthrough

---

## 0️⃣ First 2 Minutes: Frame the Problem (This Sets the Tone)

I start by saying:

> “Before we design anything, I want to understand what *must never break*.”

Then I extract **invariants**:

* Short URL must **always redirect correctly**
* Redirect must be **low latency (P99 matters)**
* Existing URLs must **never change**
* System must survive **viral spikes**
* Writes can fail; reads should not

If these aren’t agreed on, I stop.

---

## 1️⃣ Traffic Shape (This Drives Everything)

I state plainly:

> “This is a **read-dominated**, **hot-key–prone**, **tail-latency system**.”

Assumptions I validate:

* Reads ≫ writes (1000x+)
* Access pattern is extremely skewed
* Few URLs dominate traffic
* Writes are latency-insensitive
* Reads are user-facing and synchronous

Principal smell if someone skips this:

> “We’re designing for average load.”

---

## 2️⃣ ID Generation — Avoid Global Coordination

I immediately push back on:

* DB auto-increment
* Central ID service

I say:

> “This introduces global coordination and a write bottleneck.”

Acceptable options:

* Pre-allocated ID ranges per shard
* Snowflake-style IDs (with caution)
* Base62 encoding on monotonic IDs

Tradeoff I call out:

> “Random IDs reduce hot shards but hurt index locality.”

---

## 3️⃣ Storage Choice — Boring and Predictable

I recommend:

* Simple KV store
* Key: short_id
* Value: original URL + metadata

I explicitly **do not** optimize storage early.

I say:

> “Storage is not the bottleneck; access patterns are.”

---

## 4️⃣ Sharding Strategy — Hot Keys Are Inevitable

If someone says:

> “Hash by short URL”

I respond:

> “That gives uniform distribution until a link goes viral.”

I plan for:

* Hot shard overload
* Temporary skew
* No instant rebalancing

**Mitigation plan (must exist):**

* Aggressive caching
* Per-key rate limits
* Optional hot-key replication

Principal language:

> “We need a story for hot keys — not just hope.”

---

## 5️⃣ Cache Design — Where Systems Actually Fail

I say:

> “This system lives or dies by its cache.”

Design choices:

* Multi-layer cache (local + Redis)
* Cache-aside pattern
* Long TTLs for hot keys

Then I immediately ask:

> “What happens when the cache misses?”

### Failure modes I call out:

* Cache stampede on viral URL
* TTL expiry causing herd
* Cache outage melting DB

Mitigations:

* Request coalescing
* Soft TTL + background refresh
* Stale reads over failures

Principal smell:

> “Cache is treated as a performance optimization instead of a safety mechanism.”

---

## 6️⃣ Redirect Path — Optimize for Tail Latency

Redirect flow:

1. Lookup short_id
2. Return HTTP redirect
3. Async logging

Rules I enforce:

* No synchronous analytics
* No downstream fan-out
* Hard timeouts

I say:

> “Anything on the hot path must justify its existence.”

---

## 7️⃣ HTTP Semantics — Irreversible Decisions

I slow the room down here.

301 vs 302:

* 301 cached forever by browsers
* 302 increases traffic

I say:

> “This is an irreversible decision. We should default to 302 unless we’re very sure.”

Principal smell:

> “SEO decisions leaking into system design without rollback.”

---

## 8️⃣ Multi-Region Strategy — Keep It Simple

I prefer:

* Read from nearest region
* Write to one region
* Async replication

I explicitly avoid:

* Strong global consistency
* Cross-region transactions

I say:

> “Redirects don’t need strong consistency — correctness beats freshness.”

---

## 9️⃣ Abuse & Rate Limiting — Dominates at Scale

I raise this early:

> “At scale, abuse traffic will exceed real traffic.”

Controls:

* Per-IP rate limits
* Per-URL throttles
* Bot detection
* Safe-browsing integration (async)

Principal instinct:

> “Abuse is not an edge case — it’s the main case.”

---

## 🔟 Analytics — Decouple Aggressively

I insist:

* Fire-and-forget events
* Buffered async pipelines
* Data loss acceptable

I say:

> “We do not block redirects for analytics. Ever.”

---

## 1️⃣1️⃣ Observability — Detect Skew Early

Metrics I demand:

* Per-URL QPS distribution
* Cache hit ratio per key
* P99 redirect latency
* Top-N hot keys

Principal language:

> “If we can’t see skew, we can’t survive it.”

---

## 1️⃣2️⃣ Deletion & Expiry — Be Honest

I state clearly:

> “We cannot guarantee immediate deletion globally.”

Approach:

* Mark as deleted
* Stop serving from origin
* Accept cached remnants

Principal honesty:

> “Deletion is best-effort.”

---

## 1️⃣3️⃣ Migration Story — Required Before Launch

I ask:

* “Can we change ID length?”
* “Can we change encoding?”
* “Can we move storage backends?”

If answer is “no”:

> “Then we’re locking ourselves into a permanent mistake.”

---

## 1️⃣4️⃣ Failure Scenarios (I Always Walk These)

I explicitly walk through:

* Cache outage
* DB outage
* Region outage
* Hot key spike
* Retry storm

I ask:

> “Which failure takes us down fastest?”

That’s where we invest.

---

## 1️⃣5️⃣ Final Principal Summary (How I Close)

I close with:

> “This is a read-heavy, skew-dominated, tail-latency–sensitive system.
> The hardest problems are ID generation, hot keys, cache failure, and irreversible decisions.
> If we design for those explicitly, the rest is easy.”

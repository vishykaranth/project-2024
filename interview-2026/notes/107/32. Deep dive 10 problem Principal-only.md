# 1️⃣ URL Shortener (Principal-Only Deep Dive)

### What Juniors Say

* “Generate short IDs”
* “Store mapping in DB”
* “Cache with Redis”

### What Principals Worry About

#### Core Tensions

* **ID generation vs hot keys**
* **Read amplification vs write coordination**
* **Migration story**

#### Key Decisions

* ID strategy *defines* partitioning
* Sequential IDs → hot shard
* Random IDs → index + cache inefficiency

#### Failure Modes

* One short URL goes viral → single partition meltdown
* Cache miss storm during TTL expiry
* Re-keying URLs later is near-impossible

#### Principal Language

> “This ID scheme creates a hot partition under success.”
> “What’s our migration story if IDs become unbalanced?”
> “Cache helps reads, not viral write amplification.”

#### What Breaks First

* Cache eviction
* Primary shard CPU
* Kafka (if async logging)

# 2️⃣ Rate Limiter (Much Harder Than It Looks)

### Core Tensions

* **Accuracy vs coordination**
* **Fairness vs throughput**
* **Clock dependence**

#### Real Problem

You are building a **distributed counter with time semantics**.

#### Hard Questions

* Per-user? Per-IP? Per-token?
* Sliding window vs fixed window?
* Global enforcement or best-effort?

#### Failure Modes

* Clock skew → incorrect limits
* Bursts bypass enforcement
* Redis hot keys for popular users
* Retry storms amplify throttling

#### Principal Language

> “This design assumes clocks agree.”
> “Global rate limiting introduces coordination.”
> “We’re trading correctness for availability.”

#### Hidden Trap

> Retrying on 429 can *increase* load.

# 3️⃣ News Feed (Twitter-style)

### Core Tensions

* **Fan-out on write vs fan-out on read**
* **Celebrity amplification**
* **Consistency expectations**

#### Write Fan-Out

* Pros: fast reads
* Cons: massive write amplification

#### Read Fan-Out

* Pros: simpler writes
* Cons: tail latency explodes

#### Failure Modes

* Celebrity posts create write storms
* Backfill jobs overload storage
* Cache invalidation stampedes

#### Principal Language

> “This system fails under celebrity amplification.”
> “Fan-out is a scaling choice, not an implementation detail.”
> “Consistency here is a product decision.”

#### Real Trade

> Most systems use **hybrid fan-out** and accept inconsistency.

### Core Tensions

* **Freshness vs latency**
* **Recall vs tail latency**
* **Index size vs cost**

### Non-Obvious Complexity

* Queries fan out to many shards
* Tail latency is dictated by the slowest shard
* Ranking logic evolves faster than indexing logic

### Failure Modes

* One slow shard dominates P99
* Reindexing saturates IO and hurts live traffic
* Cache hides index skew until eviction
* Hot terms overload specific shards

### Principal Language

> “Search latency is dominated by the slowest shard.”
> “Freshness is a budget we spend carefully.”
> “Reindexing is a production event.”

### First Thing That Breaks

* Tail latency, not average latency

# I. FOUNDATIONS (1–40) — IN-DEPTH NOTES

---

## 1. Requirements vs Constraints

**Requirements** = what the system must do
**Constraints** = limits you cannot violate

Examples:

* Requirement: “Store user events”
* Constraint: “P99 < 200ms”, “Data must stay in EU”

🔴 **Failure mode**: Treating constraints as negotiable
🧠 **Principal instinct**: Clarify constraints *before* designing

---

## 2. Functional vs Non-Functional Requirements

* Functional: features (create order, read profile)
* Non-functional: how well (latency, availability, cost)

🔴 Most outages are **non-functional failures**
🧠 “System worked correctly — just too slowly”

---

## 3. Latency vs Throughput

* **Latency**: time per request
* **Throughput**: requests per second

You can increase one at the expense of the other.

🔴 Optimizing throughput can destroy P99 latency
🧠 Distinguished engineers ask: *“Which matters more here?”*

---

## 4. Availability vs Consistency

Core CAP tradeoff.

* Availability: system responds
* Consistency: system responds correctly

🔴 “Available but wrong” vs “Correct but unavailable”
🧠 Make this tradeoff **explicit**, never accidental

---

## 5. Durability Guarantees

What happens after ACK?

* In memory?
* On disk?
* Replicated?

🔴 ACK-before-durable = silent data loss
🧠 Always ask: *“What does ACK really mean?”*

---

## 6. Scalability Dimensions

Scaling is not one thing:

* users
* data volume
* request rate
* tenants
* regions

🔴 Scaling one dimension breaks another
🧠 Identify **which dimension grows fastest**

---

## 7. Horizontal vs Vertical Scaling

* Vertical: bigger machine
* Horizontal: more machines

🔴 Vertical scaling hides design flaws
🧠 Horizontal scaling forces good architecture

---

## 8. Elasticity

Ability to scale **up and down** automatically.

🔴 Systems that only scale up become cost bombs
🧠 Elasticity is an *operational* feature

---

## 9. Load Patterns

Load is rarely uniform:

* spikes
* bursts
* diurnal
* seasonal

🔴 Designing for average load = outage at peak
🧠 Design for **shape**, not size

---

## 10. Read-Heavy vs Write-Heavy

Reads and writes stress systems differently.

🔴 Read-optimized designs collapse under writes (and vice versa)
🧠 Identify dominant path early

---

## 11. Hot Paths

The most frequently executed code paths.

🔴 1% code paths cause 99% incidents
🧠 Optimize hot paths ruthlessly

---

## 12. Cold Paths

Rare but critical paths:

* migrations
* backfills
* restores

🔴 Cold paths fail when needed most
🧠 Test cold paths *deliberately*

---

## 13. Control Plane vs Data Plane

* Control plane: configs, metadata
* Data plane: actual traffic

🔴 Control-plane outages take down healthy data planes
🧠 Isolate aggressively

---

## 14. Steady-State vs Failure-State

Systems behave differently under failure.

🔴 “Works in steady-state” is meaningless
🧠 Design **failure behavior explicitly**

---

## 15. Partial Failure

Some components fail, others don’t.

🔴 Distributed systems fail partially by default
🧠 Assume partial failure always

---

## 16. Graceful Degradation

Reduced functionality > total failure.

Examples:

* serve stale data
* disable features

🔴 All-or-nothing systems are brittle
🧠 Decide what you can sacrifice

---

## 17. Backpressure

When downstream slows, upstream must slow.

🔴 Missing backpressure → cascading failure
🧠 Backpressure is a *design responsibility*

---

## 18. Timeouts

Waiting forever is worse than failing.

🔴 No timeouts = thread exhaustion
🧠 Every network call needs a timeout

---

## 19. Retries

Retries hide transient failure — or amplify it.

🔴 Retry storms kill recovering systems
🧠 Retries must be bounded and jittered

---

## 20. Retry Storms

When everyone retries simultaneously.

Classic pattern:

* dependency slows
* retries increase load
* system collapses

🧠 Always ask: *“What happens under retries?”*

---

## 21. Idempotency

Same request executed multiple times → same result.

🔴 Non-idempotent retries cause corruption
🧠 Idempotency is mandatory in distributed systems

---

## 22. Exactly-Once Illusion

True exactly-once is nearly impossible.

🔴 Most “exactly-once” systems are lying
🧠 Design **idempotent at-least-once**

---

## 23. At-Least-Once Semantics

May process duplicates.

🔴 Duplicate handling is mandatory
🧠 Embrace duplicates, don’t fight them

---

## 24. At-Most-Once Semantics

May lose messages, never duplicates.

🔴 Data loss is acceptable only sometimes
🧠 Use when loss < complexity

---

## 25. Eventual Consistency

Data converges over time.

🔴 User confusion if not communicated
🧠 Requires UX + system alignment

---

## 26. Strong Consistency

Reads always see latest writes.

🔴 Coordination kills scalability
🧠 Use sparingly, protect invariants only

---

## 27. Linearizability

Operations appear atomic and ordered.

🔴 Extremely expensive at scale
🧠 Only for correctness-critical paths

---

## 28. Serializability

Transactions behave as if sequential.

🔴 Poor performance under contention
🧠 Often unnecessary

---

## 29. CAP Theorem

You can’t have:

* Consistency
* Availability
* Partition tolerance

🔴 CAP tradeoffs are unavoidable
🧠 Choose **per operation**, not globally

---

## 30. PACELC Theorem

If partition:

* choose A or C
  Else:
* choose Latency or Consistency

🧠 Latency tradeoffs matter even without partitions

---

## 31. Fault Tolerance

System continues operating under failure.

🔴 Redundancy without isolation ≠ fault tolerance
🧠 Diversity matters (AZs, regions)

---

## 32. Fault Isolation

Failure doesn’t spread.

🔴 Shared dependencies defeat redundancy
🧠 Isolate aggressively

---

## 33. Blast Radius

Maximum damage from a failure.

🔴 Large blast radius = leadership failure
🧠 Limit scope by default

---

## 34. Error Budgets

Allowed failure rate.

🔴 Ignoring error budgets burns teams
🧠 Error budgets align product + reliability

---

## 35. SLOs

Targets for reliability.

🔴 Undefined SLOs = unbounded expectations
🧠 SLOs define “good enough”

---

## 36. SLIs

What you measure.

🔴 Measuring the wrong thing misleads everyone
🧠 SLIs must reflect user pain

---

## 37. SLAs

Contracts with penalties.

🔴 Internal SLAs often useless
🧠 External SLAs shape architecture

---

## 38. Brownouts

Intentional partial failure under load.

Examples:

* disable recommendations
* reduce personalization

🧠 Brownouts prevent blackouts

---

## 39. Load Shedding

Reject excess traffic intentionally.

🔴 Serving everyone poorly helps no one
🧠 Drop traffic early and explicitly

---

## 40. Graceful Failure

Fail predictably and visibly.

🔴 Silent failure is worst failure
🧠 Fail fast, fail loud, fail small

**Tail latency** is the latency experienced by the *slowest* requests — the ones at the end (the “tail”) of the latency distribution.

If you remember just one thing:

> **Tail latency is what users actually feel when systems are under stress.**

---

## The Simple Definition

* **Average latency (mean)**: how long requests take *on average*
* **Tail latency**: how long the *slowest* requests take
  (usually measured as **P95, P99, P99.9**)

Example:

* P50 = 50 ms
* P95 = 120 ms
* **P99 = 2,000 ms**

That P99 is **tail latency**.

---

## Real-World Analogy (Intuitive)

Imagine:

* 100 people in a grocery checkout line
* 99 finish in 30 seconds
* 1 person has a price check, coupon issue, card failure → 5 minutes

Average checkout time looks fine.
That one person is furious.

👉 **That’s tail latency.**

---

## Why Tail Latency Matters More Than Averages

### 1️⃣ Users Don’t Experience Averages

Users experience **their request**, not the average.

One slow request:

* feels like the system is broken
* causes retries
* causes abandonment

---

### 2️⃣ Distributed Systems Multiply Tail Latency

If a request fan-outs to **N services**:

```
End-to-end latency ≈ max(latency of all calls)
```

So:

* each service might have P99 = 200 ms
* combined system P99 can be seconds

👉 **Tail latency compounds.**

---

## Where Tail Latency Comes From

Common causes:

* GC pauses
* disk I/O stalls
* cache misses
* lock contention
* noisy neighbors
* retries
* cold starts
* leader elections
* hot partitions

> Tail latency is rarely caused by “slow code”.

---

## Classic Tail Latency Traps

### ❌ “Our average latency is fine”

This hides:

* skew
* hotspots
* partial failures

### ❌ “Just add more retries”

Retries push load to the tail and **make it worse**.

---

## How Distinguished Engineers Think About Tail Latency

They ask:

* “What’s the P99?”
* “What’s the slowest partition?”
* “What happens under retries?”
* “What happens during GC or failover?”

They treat tail latency as:

> **A correctness and reliability problem, not a performance problem.**

---

## Design Techniques to Reduce Tail Latency

* Timeouts everywhere
* Request hedging
* Load shedding
* Backpressure
* Circuit breakers
* Isolation (bulkheads)
* Async decoupling
* Limit fan-out
* Cache hot paths
* Remove coordination

---

> **“Tail latency is where distributed systems tell the truth.”**

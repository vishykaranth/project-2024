## Bandwidth Estimation in System Design

**Bandwidth estimation = calculating how much data per second your system must handle.**

It answers:

> “How much network throughput do we need so the system doesn’t choke under load?”

This is a **first-class design input**, not a side calculation.

---

# Why interviewers care

At Staff+/Principal level, this shows:

* You think in **numbers, not diagrams**
* You can **size infrastructure**
* You understand **cost, latency, and scaling limits**

Mic-drop framing:

> “Bandwidth defines whether your architecture is even feasible.”

---

# What counts as bandwidth?

Data moving between:

* Client ↔ backend
* Service ↔ service
* Cache ↔ DB
* Producer ↔ queue
* Cross-region replication

---

# 🧮 Core formula

```
Bandwidth = data per request × requests per second
```

---

# Example — Twitter home timeline

Let’s say:

### Assumptions

* 300M DAU
* 50% open timeline daily → 150M users
* Each opens app **5 times/day**

Requests per second:

```
150M × 5 / 86400 ≈ 8,700 req/sec
```

---

### Response payload

Each tweet:

* text + metadata ≈ 1 KB
* 50 tweets per timeline

```
50 KB per response
```

---

### Bandwidth

```
8,700 × 50 KB ≈ 435 MB/sec
```

That’s:

> ~3.5 Gbps outbound (egress) just for timeline reads

Now you can discuss:

* CDN?
* Edge caching?
* Compression?
* Fanout model?

This is why the calculation matters.

---

# 🏗️ Where bandwidth shapes architecture

## 1️⃣ Cache vs DB decision

If DB must serve:

```
435 MB/sec
```

It will die.

So you cache.

---

## 2️⃣ Fanout-on-write vs fanout-on-read

Celebrity with 100M followers:

```
100M × 1 KB = 100 GB per tweet
```

You cannot push that synchronously.

So:

* Hybrid fanout
* Async queues
* Pull for celebrities

---

## 3️⃣ Media strategy

Text is cheap. Video is not.

If a video = 5 MB and 1M views:

```
5 TB transfer
```

Now you must use:

* CDN
* Edge delivery

---

## 4️⃣ Cross-region replication cost

Bandwidth is:

* $$$ cloud cost
* replication latency

---

# 💰 Bandwidth = money

In cloud:

```
Data transfer OUT is expensive
```

So this becomes:

> A business decision, not just a tech one.

# 🎤 Scenario

A celebrity with **100M followers** posts **1 tweet**.

Assume:

* Fanout-on-write model
* Timeline entry size = **1 KB** (tweet id + author id + metadata pointer)

---

# 🧮 1️⃣ Data explosion calculation

```
100M followers × 1 KB = 100 GB
```

That is for **ONE tweet**.

If they tweet 5 times/day:

```
500 GB/day from a single user
```

This is the moment where a strong candidate pauses and says:

> “Fanout-on-write will not scale for high-fanout users.”

That’s the signal ✅

---

# ⛔ 2️⃣ Can we do this synchronously?

No — because:

### Write latency would become:

User tweet → system must:

* write tweet
* update 100M timelines

Even at:

```
1M writes/sec → 100 seconds
```

Tweet creation becomes minutes ❌

So:

> Fanout must be **asynchronous**

---

# 💥 3️⃣ What breaks first?

Not storage. Not CPU.

### The real bottlenecks:

#### 🔴 Queue throughput

You suddenly produce **100M events**

#### 🔴 Timeline write amplification

Massive random writes across shards

#### 🔴 Cache churn

Hot partitions

#### 🔴 Network bandwidth

Cross-node replication storm

Principal-level phrasing:

> “The system experiences write amplification and hot shard formation.”

---

# 🧠 4️⃣ Architectural strategies

## ✅ Hybrid fanout (industry standard)

### Normal users → fanout-on-write

### Celebrities → fanout-on-read

So:

* Store tweet once
* Followers pull it when they open timeline

This converts:

```
100 GB burst write
→ distributed read over time
```

---

## ✅ How do we detect celebrity?

At runtime:

```
follower_count > threshold
```

Dynamic strategy selection.

---

## ✅ Async fanout pipeline

Tweet write flow:

```
Tweet service
   → Kafka
      → Fanout workers
          → Timeline shards
```

With:

* backpressure
* rate limiting
* priority queues

---

## ✅ Priority delivery

Recent active followers → fast delivery
Inactive followers → lazy load

This protects:

> p99 timeline latency for active users.

---

# 🧊 5️⃣ Timeline read path for celebrity

When a user opens home timeline:

We:

* Fetch latest tweet IDs of followees
* Merge + rank

This is where:

* cache
* precomputed partial timelines
* Redis sorted sets

come in.

---

# 📊 6️⃣ Principal-level trade-off articulation

You say this explicitly:

| Fanout-on-write     | Fanout-on-read       |
| ------------------- | -------------------- |
| Fast reads          | Slow reads           |
| Heavy writes        | Light writes         |
| Bad for celebrities | Good for celebrities |

So we:

> Use a **hybrid model based on follower count**.

---

# 🎯 7️⃣ SLO thinking (this is Staff+ signal)

Your goal is NOT symmetry.

Your goal is:

> Protect timeline latency for the majority of users.

---

# 🎤 Interview-quality answer (what I expect you to say)

> “A celebrity with 100M followers creates a 100 GB write amplification if we use pure fanout-on-write. That makes tweet latency unacceptable and causes hot shards. So we switch to a hybrid model: normal users use fanout-on-write for fast reads, and high-fanout users use fanout-on-read. Fanout is handled asynchronously via a queue with backpressure and priority for active users to protect timeline SLOs.”

Designing a Home Timeline storage system is a classic "read-heavy" challenge. To hit a **< 200 ms** latency target at Twitter-like scale, you can't just query a database; you have to precompute the results.

Here is the design for the Home Timeline storage.

---

## 1. The Data Model

The storage is split into two layers: the **Permanent Store** (source of truth) and the **Timeline Cache** (high-speed retrieval).

### A. Permanent Storage (The "Tweet Store")

This is typically a distributed NoSQL database (like Cassandra or Manhattan) optimized for high-volume writes.

* **Tweet Table**: Stores the actual content.
* `tweet_id` (Primary Key), `author_id`, `content`, `timestamp`.


* **Social Graph**: Stores "Who follows whom."
* `user_id`, `follower_id`, `created_at`.



### B. Timeline Storage (The "Precomputed Cache")

The Home Timeline isn't a query; it’s a **list**. For every active user, we maintain a precomputed list of Tweet IDs in an in-memory store like **Redis**.

* **Key**: `home_timeline:{user_id}`
* **Value**: A sorted list (Redis ZSet) of `tweet_id`s, capped at ~800–1,000 entries.

---

## 2. Sharding Strategy

Since the goal is to serve a specific user's timeline, we shard by **User ID**.

* **Shard Key**: `user_id`
* **Why?**: This ensures that all the data required to build a single user's timeline (their "mailbox" of tweet IDs) lives on a single shard. This avoids "scatter-gather" queries where the system has to talk to 100 machines to build one page.
* **Consistent Hashing**: Use consistent hashing to map `user_id` to a specific Redis node. This allows you to add or remove cache nodes without reshuffling the entire dataset.

---

## 3. How to Fetch 50 Tweets in < 200 ms

To achieve sub-200ms performance, the system uses a **Hybrid Fan-out** approach.

### Step 1: Precomputation (Fan-out on Write)

When a "normal" user (low follower count) tweets:

1. The **Fan-out Service** looks up all their followers.
2. It pushes the `tweet_id` into the Redis ZSet of every follower.
3. **The Result**: When a follower logs in, their timeline is already "sitting there" in memory.

### Step 2: Handling Celebrities (The "Pull" Model)

If a celebrity (e.g., 50M followers) tweets, pushing to 50M Redis lists would take minutes and crash the system.

* **The Hybrid Fix**: We don't fan out celebrity tweets. Instead, when a user requests their timeline, the system fetches their precomputed Redis list **AND** pulls the latest tweets from any celebrities they follow.
* These are merged and sorted in milliseconds at read-time.

### Step 3: Fast Retrieval & Hydration

1. **Get IDs**: The Timeline Service hits the Redis shard for the `user_id` and grabs the top 50 `tweet_id`s. (**~5–10 ms**)
2. **Hydration**: The IDs are just numbers. We need the text/images. We perform a `multi-get` request to a separate **Tweet Object Cache** (Memcached) to fetch the actual content for those 50 IDs in parallel. (**~20–50 ms**)
3. **Filter/Rank**: Apply simple filters (e.g., hidden/blocked users). (**~10 ms**)

**Total Latency**: Usually **< 100 ms**, well within the 200 ms budget.

# Prime Video: Scaling Up and Reducing Costs by 90% - Summary

**Source**: [Prime Video Tech Blog - Scaling up the Prime Video audio/video monitoring service and reducing costs by 90%](https://web.archive.org/web/20230504060528/https://www.primevideotech.com/video-streaming/scaling-up-the-prime-video-audio-video-monitoring-service-and-reducing-costs-by-90)

**Author**: Marcin Kolny, Senior SDE – Prime Video  
**Date**: March 22, 2023

---

## Overview

Prime Video successfully migrated their audio/video monitoring service from a **distributed microservices architecture** to a **monolith application**, achieving:
- **90% cost reduction** in infrastructure costs
- **Significantly improved scaling capabilities** (from ~5% of expected load to thousands of concurrent streams)
- **Better resilience** and operational simplicity

**Key Lesson**: Microservices aren't always the right solution - the choice between microservices and monolith should be made on a case-by-case basis.

---

## The Problem

### Service Purpose

Prime Video needed to monitor **thousands of live streams** to:
- Automatically identify perceptual quality issues (block corruption, audio/video sync problems)
- Trigger automatic fixes when defects are detected
- Ensure seamless content delivery to customers

### Initial Architecture: Distributed Microservices

**Components**:
1. **Media Converter**: Converts audio/video streams to frames or decrypted audio buffers
2. **Defect Detectors**: Execute algorithms analyzing frames/buffers in real-time for defects
3. **Orchestration**: Controls the flow between components

**Technology Stack**:
- **AWS Step Functions**: Orchestration workflow
- **AWS Lambda**: Serverless execution for detectors
- **Amazon S3**: Intermediate storage for video frames/images
- **Serverless components**: Distributed architecture

**Architecture Diagram**:
```
Customer Request
    ↓
Lambda Function
    ↓
Step Functions (Orchestration)
    ↓
Media Conversion Service → S3 Bucket (frames)
    ↓
Detector Microservices (Lambda) → Download from S3 → Process
    ↓
Aggregated Results → S3 Bucket
```

### Scaling Bottlenecks

**Issue 1: Step Functions Limits**
- Performed **multiple state transitions per second** for every stream
- Quickly hit **AWS account limits** at ~5% of expected load
- **Cost**: Step Functions charges per state transition (expensive at scale)

**Issue 2: S3 Transfer Costs**
- Video frames (images) passed between components via S3
- High number of **Tier-1 S3 calls** (GET/PUT operations)
- **Expensive**: S3 data transfer costs accumulated quickly

**Issue 3: Overall Cost**
- Cost of all building blocks was **too high** for large-scale operation
- Could not achieve target scale (thousands of concurrent streams)

---

## The Solution: Migration to Monolith

### Architectural Change

**Decision**: Pack all components into a **single process** running on Amazon ECS/EC2.

**Key Changes**:
1. **Eliminated S3 intermediate storage**: Data transfer now happens **in-memory**
2. **Removed Step Functions**: Orchestration now **within single instance**
3. **Single ECS Task**: All components run in one container/process
4. **Reused code**: Same components, just repackaged

**New Architecture Diagram**:
```
Customer Request
    ↓
Lambda Function (lightweight routing)
    ↓
Single ECS Task (Monolith)
    ├── Media Converter (in-process)
    ├── Defector Detectors (in-process)
    └── Orchestration (in-process)
    ↓
Results → S3 Bucket (final storage only)
```

### Benefits of Monolith Approach

1. **In-Memory Data Transfer**:
   - No S3 intermediate storage needed
   - Frames passed directly in memory between components
   - **Eliminated S3 Tier-1 call costs**

2. **Simplified Orchestration**:
   - No Step Functions state transitions
   - Orchestration logic within single process
   - **Eliminated Step Functions costs**

3. **Better Resource Utilization**:
   - Single process uses resources more efficiently
   - No network overhead between components
   - **Reduced infrastructure costs**

### Handling Scale: Detector Cloning

**Challenge**: Detectors scale vertically (within single instance), but team regularly adds more detectors.

**Solution**: **Clone the service** multiple times with different detector subsets.

**Approach**:
```
Lambda Function (Router)
    ↓
    ├── ECS Task 1 (Detectors A, B, C)
    ├── ECS Task 2 (Detectors D, E, F)
    └── ECS Task 3 (Detectors G, H, I)
    ↓
Each task stores results separately in S3
```

**Lightweight Orchestration Layer**: Distributes customer requests to appropriate ECS tasks based on detector requirements.

## Key Insights and Takeaways

### 1. Microservices Aren't Always Better

**Lesson**: Distributed systems and serverless components work at high scale, but **whether to use them over monolith has to be made on a case-by-case basis**.

**When Microservices Make Sense**:
- Independent scaling requirements
- Different teams owning different services
- Need for technology diversity
- Clear service boundaries

**When Monolith Makes Sense**:
- Tightly coupled components
- High data transfer between components
- Cost of network/coordination overhead exceeds benefits
- Single team ownership

### 2. Cost of Data Transfer

**Key Finding**: Passing data between distributed components (especially via S3) was a **major cost driver**.

**Solution**: Keeping data transfer **in-memory** eliminated significant costs.

### 3. Orchestration Overhead

**Problem**: Step Functions state transitions became a **scaling bottleneck** and cost driver.

**Solution**: **In-process orchestration** eliminated both scaling limits and costs.

### 4. Counterintuitive Decisions

**Example**: Replicating computationally expensive media conversion process.

**Intuition**: Run conversion once and cache results (seems cheaper)

**Reality**: Replicating conversion closer to detectors was **more cost-effective** in their specific use case.

**Lesson**: **Measure and experiment** - assumptions may be wrong.

### 5. Architecture Evolution

**Approach**: 
- Started with distributed architecture (good for quick development)
- Identified bottlenecks and costs
- **Rearchitected** based on learnings
- Achieved better results with simpler architecture

**Process**: 
1. Build quickly with available tools
2. Measure and identify problems
3. Rearchitect based on data
4. Optimize for actual requirements

## Technical Details

### Original Architecture Components

1. **Media Converter Microservice**:
   - Converts streams to frames/audio buffers
   - Uploads to S3 for detector consumption

2. **Defect Detector Microservices** (Lambda):
   - Each detector as separate Lambda function
   - Downloads frames from S3
   - Processes concurrently
   - Sends notifications on defects

3. **Orchestration** (Step Functions):
   - Manages workflow between components
   - Handles state transitions
   - Coordinates execution

### New Architecture Components

1. **Single ECS Task**:
   - Media Converter (in-process)
   - All Defect Detectors (in-process)
   - Orchestration Logic (in-process)
   - Data transfer via memory

2. **Lightweight Router** (Lambda):
   - Routes requests to appropriate ECS task
   - Based on detector requirements

3. **S3 Bucket**:
   - Only for final results storage
   - No intermediate frame storage

### Scaling Strategy

**Vertical Scaling**: 
- Detectors scale within single instance
- Limited by instance capacity

**Horizontal Scaling**:
- Clone service with different detector subsets
- Multiple ECS tasks handle different detector combinations
- Router distributes requests

## Lessons for Other Teams

### Decision Framework

**Consider Microservices When**:
- ✅ Components have independent scaling needs
- ✅ Different teams own different services
- ✅ Clear service boundaries exist
- ✅ Network overhead is minimal

**Consider Monolith When**:
- ✅ Components are tightly coupled
- ✅ High data transfer between components
- ✅ Network/coordination costs exceed benefits
- ✅ Single team ownership
- ✅ Cost is a primary concern

### Cost Optimization Strategies

1. **Measure Everything**: Identify actual cost drivers
2. **Question Assumptions**: What seems cheaper may not be
3. **Consider Data Transfer Costs**: Network/storage costs can dominate
4. **Evaluate Orchestration Overhead**: Coordination costs add up
5. **Experiment**: Try different approaches and measure results

### Architecture Principles

1. **Start Simple**: Begin with simplest solution that works
2. **Measure First**: Understand actual bottlenecks before optimizing
3. **Question Patterns**: Microservices aren't always the answer
4. **Optimize for Reality**: Design for actual requirements, not theoretical benefits
5. **Be Pragmatic**: Choose architecture based on specific use case

## Conclusion

Prime Video's experience demonstrates that:

1. **Microservices aren't always better** - sometimes monolith is the right choice
2. **Cost optimization requires measurement** - identify actual cost drivers
3. **Data transfer costs matter** - in-memory can be much cheaper
4. **Orchestration overhead is real** - coordination costs add up
5. **Architecture should evolve** - start simple, measure, optimize

**Key Takeaway**: The choice between microservices and monolith should be **pragmatic and data-driven**, not ideological. Prime Video achieved 90% cost reduction and better scaling by choosing the simpler architecture that fit their specific use case.

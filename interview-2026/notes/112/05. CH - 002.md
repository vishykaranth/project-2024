## 1. The Core Concept (5 Minutes)

Standard hashing () fails in dynamic environments because if the number of nodes () changes, almost all keys must be remapped. Consistent Hashing minimizes this disruption.

* **The Hash Ring:** Both keys and nodes are hashed onto a circular space (typically  to ).
* **Assignment:** A key is assigned to the first node encountered moving clockwise on the ring.

## 2. Advanced Principal-Level Nuances (15 Minutes)

This is where you demonstrate seniority. Address the "real-world" problems of the basic algorithm.

### Virtual Nodes (Vnodes)

**Problem:** In a basic ring, nodes are rarely distributed uniformly, leading to "hot spots" where one node handles 5x the load of another.
**Solution:** Assign each physical node hundreds of "logical" locations (Vnodes) on the ring.

* **Benefit:** Better load balancing and heterogeneous support (give a 64GB RAM server more Vnodes than a 16GB RAM server).

### Replication & High Availability

In a distributed hash table (DHT) like Cassandra or Dynamo:

* **Replication Factor ():** A key isn't just stored on one node; it's stored on the first node found and the next  distinct physical nodes clockwise.
* **Sloppy Quorum:** Handling temporary failures by allowing "hinted handoff" to the next healthy node in the ring.

### Cascading Failures

Discuss the "Adjacent Node" problem. If Node A fails, all its load shifts to Node B. If Node B wasn't provisioned for 2x traffic, it fails too.

* **Mitigation:** Virtual nodes help here because the load of a failed physical node is distributed across *many* other physical nodes, not just one neighbor.

---

## 3. High-Level Trade-offs (10 Minutes)

A Principal Engineer should know when **not** to use Consistent Hashing.

| Strategy | Best For | Downside |
| --- | --- | --- |
| **Range-Based Sharding** | Range queries (e.g., `WHERE age > 20`) | Massive hot spots on certain ranges. |
| **Consistent Hashing** | Key-value lookups, distributed caching. | Complex implementation; range scans are expensive. |
| **Lookup Service** | Granular control (e.g., moving specific accounts). | The lookup table itself becomes a bottleneck/SPOF. |

---

## 4. Key Interviewer "Gotchas" to Prep For

* **"How do nodes know about the ring state?"** Talk about **Gossip Protocols** (decentralized) vs. an external **Control Plane** like ZooKeeper or Etcd (centralized).
* **"What happens during a network partition?"** Relate it to the **CAP Theorem**. Does your ring prioritize Availability (Dynamo/AP) or Consistency (HBase/CP)?
* **"How do you handle 'Celebrity' keys (Hot Keys)?"** Even with consistent hashing, a single key (e.g., a viral tweet) hits one node. Discuss **Local Caching** or **Sub-sharding** specific hot keys.

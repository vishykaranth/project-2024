## What is a Database Index?

A **database index** is a data structure that improves the speed of data retrieval operations on a database table at the cost of additional storage space and slower writes. Think of it as an index in a book - instead of reading every page to find a topic, you look it up in the index and go directly to the relevant page.

### Analogy: Book Index

```
Without Index (Full Table Scan):
- To find "Database Indexes" → Read entire book page by page
- Time: O(n) - linear search

With Index:
- Look up "Database Indexes" in index → Page 45
- Go directly to page 45
- Time: O(log n) - logarithmic search
```

### Formal Definition

An index is a **separate data structure** that:
- Contains a **copy of selected columns** from a table
- Maintains a **sorted or hash-based structure** for fast lookup
- Maps **index key values** to **row locations** (row IDs, pointers, or primary keys)
- Is automatically maintained by the database system

## How Indexes Work

### Basic Mechanism

```
┌─────────────────────────────────────────────────────────┐
│              Index Operation Flow                       │
└─────────────────────────────────────────────────────────┘

1. Query: SELECT * FROM users WHERE email = 'user@example.com'

2. Database Options:
   ├─► Option A: Full Table Scan
   │   └─► Read every row, check email column
   │   └─► Time: O(n) - scans all rows
   │
   └─► Option B: Use Index
       ├─► Look up 'user@example.com' in email index
       ├─► Index returns row location (e.g., row ID 12345)
       └─► Fetch row directly
       └─► Time: O(log n) - binary search in index
```

### Step-by-Step Process

#### 1. Index Creation
```sql
CREATE INDEX idx_email ON users(email);
```

**What Happens**:
- Database creates a new data structure
- Copies email column values
- Sorts or hashes them
- Creates mapping: email → row location

#### 2. Query Execution with Index

**Query**:
```sql
SELECT * FROM users WHERE email = 'user@example.com';
```

**Process**:
1. **Query Parser**: Identifies WHERE clause on `email`
2. **Query Optimizer**: Checks if index exists on `email`
3. **Index Lookup**: Searches index for `'user@example.com'`
4. **Row Retrieval**: Uses row location from index to fetch actual row
5. **Result Return**: Returns matching row(s)

#### 3. Index Structure Example

**Table**: `users`
```
Row ID | email              | name
-------|--------------------|----------
1      | alice@example.com | Alice
2      | bob@example.com   | Bob
3      | charlie@example.com| Charlie
```

**Index on `email`**:
```
Index Key (email)          | Row Location
---------------------------|-------------
alice@example.com         | → Row 1
bob@example.com           | → Row 2
charlie@example.com       | → Row 3
```

**Lookup**: `WHERE email = 'bob@example.com'`
- Search index → Find `bob@example.com` → Get Row 2
- Fetch Row 2 directly (no table scan needed)

## Types of Indexes

### 1. Primary Index (Clustered Index)

**Definition**: Index on primary key that determines physical storage order.

**Characteristics**:
- **One per table** (usually)
- **Physical ordering**: Rows stored in index order
- **Automatic**: Created automatically for primary key
- **Fastest**: No separate lookup needed

**Example**:
```sql
CREATE TABLE users (
    id INT PRIMARY KEY,  -- Automatically creates clustered index
    email VARCHAR(255)
);
```

**Storage**:
```
Physical Storage (matches index order):
Row 1: id=1, email='alice@example.com'
Row 2: id=2, email='bob@example.com'
Row 3: id=3, email='charlie@example.com'
```

### 2. Secondary Index (Non-Clustered Index)

**Definition**: Index on non-primary key columns.

**Characteristics**:
- **Multiple per table**: Can have many secondary indexes
- **Logical ordering**: Points to row locations
- **Separate structure**: Stored separately from table data
- **Two-step lookup**: Index → Row location → Actual row

**Example**:
```sql
CREATE INDEX idx_email ON users(email);  -- Secondary index
```

**Storage**:
```
Table (unordered):
Row 1: id=1, email='alice@example.com'
Row 3: id=3, email='charlie@example.com'
Row 2: id=2, email='bob@example.com'

Index (sorted):
email              | Row Location
-------------------|-------------
alice@example.com  | → Row 1
bob@example.com    | → Row 2
charlie@example.com| → Row 3
```

### 3. Unique Index

**Definition**: Index that enforces uniqueness constraint.

**Characteristics**:
- **Uniqueness**: Prevents duplicate values
- **Performance**: Faster than unique constraint without index
- **NULL handling**: Usually allows one NULL value

**Example**:
```sql
CREATE UNIQUE INDEX idx_unique_email ON users(email);
```

**Effect**:
- Prevents duplicate emails
- Speeds up email lookups
- Can be used for foreign key references

### 4. Composite Index (Multi-Column Index)

**Definition**: Index on multiple columns.

**Characteristics**:
- **Column order matters**: (col1, col2) ≠ (col2, col1)
- **Left-prefix rule**: Can use first N columns
- **Efficient**: For queries filtering on multiple columns

**Example**:
```sql
CREATE INDEX idx_name_email ON users(last_name, first_name, email);
```

**Query Matching**:
```sql
-- ✅ Uses index (left-prefix: last_name)
WHERE last_name = 'Smith'

-- ✅ Uses index (left-prefix: last_name, first_name)
WHERE last_name = 'Smith' AND first_name = 'John'

-- ✅ Uses index (all columns)
WHERE last_name = 'Smith' AND first_name = 'John' AND email = 'john@example.com'

-- ❌ Does NOT use index (skips last_name)
WHERE first_name = 'John'

-- ❌ Does NOT use index (wrong order)
WHERE email = 'john@example.com' AND last_name = 'Smith'
```

**Index Structure**:
```
(last_name, first_name, email)    | Row Location
----------------------------------|-------------
(Smith, John, john@example.com)  | → Row 1
(Smith, Jane, jane@example.com)  | → Row 2
(Williams, Bob, bob@example.com) | → Row 3
```

### 5. Partial Index

**Definition**: Index on subset of rows (filtered index).

**Characteristics**:
- **Smaller size**: Only indexes matching rows
- **Faster**: Smaller index = faster lookups
- **Conditional**: Uses WHERE clause

**Example**:
```sql
-- Index only active users
CREATE INDEX idx_active_email ON users(email) 
WHERE status = 'active';
```

**Use Case**: When most queries filter by a condition.

### 6. Full-Text Index

**Definition**: Index for text search operations.

**Characteristics**:
- **Text search**: For LIKE, CONTAINS, MATCH operations
- **Language-aware**: Handles stemming, synonyms
- **Specialized**: Different from regular indexes

**Example**:
```sql
CREATE FULLTEXT INDEX idx_content ON articles(content);
```

**Query**:
```sql
SELECT * FROM articles 
WHERE MATCH(content) AGAINST('database index' IN NATURAL LANGUAGE MODE);
```

### 7. Hash Index

**Definition**: Index using hash table structure.

**Characteristics**:
- **O(1) lookup**: Constant time for exact matches
- **No range queries**: Only equality comparisons
- **Memory-based**: Usually in-memory only

**Example** (PostgreSQL):
```sql
CREATE INDEX idx_email_hash ON users USING HASH(email);
```

**Use Case**: Exact match lookups only.

### 8. Bitmap Index

**Definition**: Index using bitmap for each distinct value.

**Characteristics**:
- **Low cardinality**: Good for columns with few distinct values
- **Space efficient**: For boolean, status, category columns
- **Fast AND/OR**: Efficient for multiple conditions

**Example**:
```sql
-- Status: 'active', 'inactive', 'pending' (3 values)
CREATE BITMAP INDEX idx_status ON users(status);
```

**Structure**:
```
Status    | Bitmap (1 = row has this status)
----------|----------------------------------
active    | 1010101... (rows 1,3,5,7,...)
inactive  | 0101010... (rows 2,4,6,8,...)
pending   | 0000000... (no rows)
```

## Index Data Structures

### 1. B-Tree (Balanced Tree)

**Most Common**: Used by PostgreSQL, MySQL (InnoDB), SQL Server, Oracle.

**Structure**:
```
                    [50]
                   /    \
              [25]        [75]
             /   \       /    \
        [10,20] [30,40] [60,70] [80,90]
```

**Characteristics**:
- **Balanced**: All leaves at same depth
- **Sorted**: Keys in sorted order
- **Range queries**: Efficient for range scans
- **O(log n)**: Logarithmic search time

**Operations**:
- **Search**: O(log n)
- **Insert**: O(log n)
- **Delete**: O(log n)
- **Range Scan**: O(log n + k) where k = results

### 2. B+ Tree

**Enhanced B-Tree**: Used by MySQL (InnoDB), PostgreSQL.

**Differences from B-Tree**:
- **Data in leaves only**: Internal nodes store keys only
- **Linked leaves**: Leaves linked for range scans
- **More efficient**: Better for range queries

**Structure**:
```
                    [50]
                   /    \
              [25]        [75]
             /   \       /    \
        [10,20] [30,40] [60,70] [80,90]
         ↓        ↓       ↓       ↓
    [data]   [data]  [data]  [data]
     ↓        ↓       ↓       ↓
   [next] → [next] → [next] → [next]
```

### 3. Hash Table

**Structure**: Key-value pairs with hash function.

**Characteristics**:
- **O(1) average**: Constant time lookup
- **No ordering**: Keys not sorted
- **Collision handling**: Chaining or open addressing

**Example**:
```
Hash(email) → Bucket → Linked List of Row Locations

hash('alice@example.com') = 5 → Bucket 5 → [Row 1, Row 10]
hash('bob@example.com') = 2 → Bucket 2 → [Row 2]
```

### 4. LSM-Tree (Log-Structured Merge Tree)

**Used By**: Cassandra, RocksDB, LevelDB.

**Characteristics**:
- **Write-optimized**: Appends to log
- **Read-optimized**: Merges sorted files
- **No in-place updates**: Immutable segments

## Benefits of Indexes

### 1. Faster Queries

**Performance Improvement**:

| Operation | Without Index | With Index | Improvement |
|-----------|---------------|------------|-------------|
| Find one row | O(n) - Full scan | O(log n) - Index lookup | 100-1000x faster |
| Find range | O(n) - Full scan | O(log n + k) - Index scan | 10-100x faster |
| Sort | O(n log n) - Sort all | O(k) - Already sorted | Significant |

**Example**:
```sql
-- Table with 1 million rows
SELECT * FROM users WHERE email = 'user@example.com';

-- Without index: ~100ms (scans 1M rows)
-- With index: ~1ms (index lookup)
-- Improvement: 100x faster
```

### 2. Faster Joins

**Join Performance**:
```sql
SELECT u.*, o.* 
FROM users u
JOIN orders o ON u.id = o.user_id;
```

**Without Index on `o.user_id`**:
- Nested loop: For each user, scan all orders
- Time: O(n × m) - very slow

**With Index on `o.user_id`**:
- For each user, lookup orders via index
- Time: O(n × log m) - much faster

### 3. Faster Sorting

**ORDER BY Performance**:
```sql
SELECT * FROM users ORDER BY email;
```

**Without Index**:
- Must sort all rows: O(n log n)

**With Index**:
- Already sorted: O(n) - just traverse index

### 4. Enforce Uniqueness

**Unique Constraint**:
```sql
CREATE UNIQUE INDEX idx_email ON users(email);
```

**Benefits**:
- Prevents duplicates
- Fast duplicate checking
- Database-level enforcement

### 5. Foreign Key Performance

**Foreign Key Lookups**:
```sql
-- Check if user exists before inserting order
INSERT INTO orders (user_id, ...) VALUES (123, ...);
```

**With Index on `users.id`**:
- Fast foreign key validation
- Efficient referential integrity checks

## Costs and Trade-offs

### 1. Storage Space

**Overhead**:
- **Index size**: Typically 10-30% of table size
- **Composite indexes**: Larger (multiple columns)
- **Multiple indexes**: Additive overhead

**Example**:
```
Table: users (1GB)
├─ Primary key index: ~100MB
├─ Email index: ~150MB
├─ Name index: ~200MB
└─ Composite index: ~300MB

Total: 1GB + 750MB = 1.75GB
```

### 2. Write Performance

**Impact on INSERT/UPDATE/DELETE**:

**Without Index**:
- INSERT: O(1) - append to table
- UPDATE: O(1) - modify row
- DELETE: O(1) - remove row

**With Index**:
- INSERT: O(log n) - update all indexes
- UPDATE: O(log n) - update indexes if indexed columns change
- DELETE: O(log n) - remove from all indexes

**Example**:
```sql
-- Table with 5 indexes
INSERT INTO users (email, name, ...) VALUES (...);

-- Must update:
-- 1. Table row
-- 2. Primary key index
-- 3. Email index
-- 4. Name index
-- 5. Composite index 1
-- 6. Composite index 2

-- Each index update: O(log n)
-- Total: 6 × O(log n) = O(log n) but with higher constant
```

**Rule of Thumb**:
- **Read-heavy**: More indexes = better
- **Write-heavy**: Fewer indexes = better

### 3. Maintenance Overhead

**Index Maintenance**:
- **Rebuilding**: Periodic rebuilds for optimization
- **Statistics**: Database updates index statistics
- **Fragmentation**: Indexes can become fragmented

**Costs**:
- **CPU**: Index operations consume CPU
- **I/O**: Index updates require disk I/O
- **Lock contention**: Index updates may lock pages

### 4. Query Optimizer Complexity

**Optimizer Decisions**:
- **Index selection**: Must choose best index
- **Index merge**: May use multiple indexes
- **Covering index**: May use index-only scan

**Potential Issues**:
- **Wrong index**: Optimizer may choose suboptimal index
- **Index hints**: May need manual hints
- **Statistics**: Outdated statistics = poor decisions

## When to Use Indexes

### 1. Frequently Queried Columns

**Criteria**:
- Column appears in WHERE clause often
- Column used in JOIN conditions
- Column used in ORDER BY

**Example**:
```sql
-- Frequently queried
SELECT * FROM users WHERE email = ?;  -- Index on email
SELECT * FROM orders WHERE user_id = ?;  -- Index on user_id
SELECT * FROM products ORDER BY price;  -- Index on price
```

### 2. Foreign Key Columns

**Always Index**:
```sql
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- Always create index on foreign key
CREATE INDEX idx_user_id ON orders(user_id);
```

**Reason**: Foreign key lookups are frequent.

### 3. Columns with High Selectivity

**Selectivity**: Ratio of distinct values to total rows.

**High Selectivity** (Good for Index):
- Email: 1M distinct values / 1M rows = 100% selectivity
- User ID: 1M distinct values / 1M rows = 100% selectivity

**Low Selectivity** (Poor for Index):
- Status: 3 distinct values / 1M rows = 0.0003% selectivity
- Gender: 2 distinct values / 1M rows = 0.0002% selectivity

**Rule**: Higher selectivity = better index candidate.

### 4. Range Queries

**Range Operations**:
```sql
-- Good for B-Tree index
WHERE price BETWEEN 100 AND 200
WHERE created_at > '2024-01-01'
WHERE age >= 18 AND age <= 65
```

**Index Benefits**:
- Efficient range scans
- Sorted data enables binary search

### 5. Sorting Operations

**ORDER BY**:
```sql
-- Index on price enables fast sorting
SELECT * FROM products ORDER BY price;
```

**Benefits**:
- Pre-sorted data
- No sort operation needed

### 6. Composite Queries

**Multiple Conditions**:
```sql
-- Composite index on (status, created_at)
WHERE status = 'active' AND created_at > '2024-01-01'
```

**Benefits**:
- Single index covers multiple conditions
- More efficient than multiple single-column indexes

## When NOT to Use Indexes

### 1. Small Tables

**Rule**: Tables with < 1000 rows usually don't benefit.

**Reason**:
- Full table scan is fast enough
- Index overhead > benefit

**Example**:
```sql
-- Configuration table with 50 rows
-- Index not needed
CREATE TABLE config (
    key VARCHAR(50),
    value TEXT
);
```

### 2. Frequently Updated Columns

**Problem**: Every update requires index maintenance.

**Example**:
```sql
-- Counter updated frequently
UPDATE users SET login_count = login_count + 1 WHERE id = ?;

-- Index on login_count would slow down updates
-- Better: No index, or update in application
```

### 3. Low Selectivity Columns

**Problem**: Index doesn't filter much data.

**Example**:
```sql
-- Status: 'active' (90%), 'inactive' (10%)
-- Index returns 90% of rows anyway
-- Full table scan may be faster
WHERE status = 'active'  -- Returns 900K of 1M rows
```

**Exception**: Bitmap indexes can help for low selectivity.

### 4. Columns Rarely Used in Queries

**Problem**: Index overhead without benefit.

**Example**:
```sql
-- Column queried once per year
-- Index maintenance cost > query benefit
CREATE INDEX idx_rare_column ON users(rare_column);
```

### 5. Text Columns (Long)

**Problem**: Large index size, limited benefit.

**Example**:
```sql
-- Long text column
CREATE TABLE articles (
    id INT PRIMARY KEY,
    content TEXT  -- 10KB average
);

-- Index on content would be huge
-- Better: Full-text index if needed
```

### 6. Computed/Expression Columns

**Problem**: Index on expression requires computation.

**Example**:
```sql
-- Expression index
CREATE INDEX idx_full_name ON users(first_name || ' ' || last_name);

-- Every lookup must compute expression
-- Consider: Materialized column instead
```

## Index Best Practices

### 1. Index Primary Keys

**Always**: Primary keys are automatically indexed.

```sql
CREATE TABLE users (
    id INT PRIMARY KEY,  -- Automatically indexed
    email VARCHAR(255)
);
```

### 2. Index Foreign Keys

**Always**: Foreign keys should be indexed.

```sql
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- Always create index
CREATE INDEX idx_user_id ON orders(user_id);
```

### 3. Index Frequently Filtered Columns

**Identify**: Columns in WHERE clauses.

```sql
-- Common query pattern
SELECT * FROM users WHERE email = ?;
-- → Index on email

SELECT * FROM orders WHERE status = ? AND created_at > ?;
-- → Composite index on (status, created_at)
```

### 4. Consider Composite Indexes

**Strategy**: Single composite index vs multiple single-column indexes.

**Example**:
```sql
-- Query pattern
WHERE status = 'active' AND created_at > '2024-01-01'

-- Option 1: Two indexes (less efficient)
CREATE INDEX idx_status ON orders(status);
CREATE INDEX idx_created_at ON orders(created_at);

-- Option 2: One composite index (more efficient)
CREATE INDEX idx_status_created ON orders(status, created_at);
```

**Rule**: Match index to query pattern.

### 5. Column Order Matters

**Left-Prefix Rule**: Order columns by selectivity (most selective first).

```sql
-- Good: High selectivity first
CREATE INDEX idx_email_status ON users(email, status);
-- email: 100% selectivity
-- status: 10% selectivity

-- Can use for:
-- WHERE email = ?  ✅
-- WHERE email = ? AND status = ?  ✅
```

### 6. Monitor Index Usage

**Check**: Which indexes are actually used.

**PostgreSQL**:
```sql
SELECT 
    schemaname,
    tablename,
    indexname,
    idx_scan  -- Number of times index used
FROM pg_stat_user_indexes
ORDER BY idx_scan;
```

**MySQL**:
```sql
SHOW INDEX FROM users;
-- Check Cardinality and usage stats
```

### 7. Avoid Over-Indexing

**Problem**: Too many indexes slow down writes.

**Guideline**:
- **Read-heavy**: 5-10 indexes per table
- **Write-heavy**: 2-5 indexes per table
- **Balanced**: 3-7 indexes per table

### 8. Use Partial Indexes

**When**: Most queries filter by condition.

```sql
-- Most queries filter active users
CREATE INDEX idx_active_email ON users(email) 
WHERE status = 'active';

-- Smaller index, faster lookups
```

### 9. Consider Covering Indexes

**Covering Index**: Index contains all columns needed by query.

```sql
-- Query
SELECT id, email, name FROM users WHERE email = ?;

-- Covering index (includes all columns)
CREATE INDEX idx_email_covering ON users(email, id, name);

-- Benefit: Index-only scan (no table lookup)
```

### 10. Regular Maintenance

**Tasks**:
- **Rebuild**: Periodically rebuild fragmented indexes
- **Statistics**: Update index statistics
- **Monitor**: Track index usage and performance

## Index Maintenance

### 1. Rebuilding Indexes

**When**: Index becomes fragmented or statistics outdated.

**PostgreSQL**:
```sql
REINDEX INDEX idx_email;
REINDEX TABLE users;
```

**MySQL**:
```sql
ALTER TABLE users DROP INDEX idx_email;
CREATE INDEX idx_email ON users(email);
```

**SQL Server**:
```sql
ALTER INDEX idx_email ON users REBUILD;
```

### 2. Updating Statistics

**Purpose**: Help query optimizer make better decisions.

**PostgreSQL**:
```sql
ANALYZE users;
```

**MySQL**:
```sql
ANALYZE TABLE users;
```

**SQL Server**:
```sql
UPDATE STATISTICS users;
```

### 3. Monitoring Index Fragmentation

**PostgreSQL**:
```sql
SELECT 
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes;
```

**SQL Server**:
```sql
SELECT 
    OBJECT_NAME(object_id) AS table_name,
    name AS index_name,
    avg_fragmentation_in_percent
FROM sys.dm_db_index_physical_stats(
    DB_ID(), NULL, NULL, NULL, 'DETAILED'
)
WHERE avg_fragmentation_in_percent > 30;
```

### 4. Dropping Unused Indexes

**Identify Unused**:
```sql
-- PostgreSQL
SELECT 
    schemaname,
    tablename,
    indexname,
    idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0;  -- Never used
```

**Drop**:
```sql
DROP INDEX idx_unused;
```

## Examples and Use Cases

### Example 1: E-Commerce Orders

**Table**:
```sql
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    status VARCHAR(20),
    created_at TIMESTAMP,
    total_amount DECIMAL(10,2)
);
```

**Common Queries**:
```sql
-- 1. Find user's orders
SELECT * FROM orders WHERE user_id = ?;
-- → Index on user_id

-- 2. Find orders by status and date
SELECT * FROM orders 
WHERE status = 'pending' AND created_at > '2024-01-01';
-- → Composite index on (status, created_at)

-- 3. Find recent orders
SELECT * FROM orders ORDER BY created_at DESC LIMIT 10;
-- → Index on created_at
```

**Indexes**:
```sql
CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_status_created ON orders(status, created_at);
CREATE INDEX idx_created_at ON orders(created_at);
```

### Example 2: User Authentication

**Table**:
```sql
CREATE TABLE users (
    id INT PRIMARY KEY,
    email VARCHAR(255) UNIQUE,
    username VARCHAR(50) UNIQUE,
    password_hash VARCHAR(255),
    last_login TIMESTAMP
);
```

**Common Queries**:
```sql
-- 1. Login by email
SELECT * FROM users WHERE email = ?;
-- → Unique index on email (automatic)

-- 2. Login by username
SELECT * FROM users WHERE username = ?;
-- → Unique index on username (automatic)

-- 3. Find inactive users
SELECT * FROM users 
WHERE last_login < '2024-01-01';
-- → Index on last_login
```

**Indexes**:
```sql
-- Automatic from UNIQUE constraints
-- CREATE UNIQUE INDEX idx_email ON users(email);
-- CREATE UNIQUE INDEX idx_username ON users(username);

CREATE INDEX idx_last_login ON users(last_login);
```

### Example 3: Search Functionality

**Table**:
```sql
CREATE TABLE products (
    id INT PRIMARY KEY,
    name VARCHAR(255),
    category VARCHAR(50),
    price DECIMAL(10,2),
    description TEXT
);
```

**Common Queries**:
```sql
-- 1. Search by category and price range
SELECT * FROM products 
WHERE category = 'electronics' 
  AND price BETWEEN 100 AND 500;

-- 2. Sort by price
SELECT * FROM products ORDER BY price;

-- 3. Full-text search
SELECT * FROM products 
WHERE MATCH(name, description) AGAINST('laptop' IN NATURAL LANGUAGE MODE);
```

**Indexes**:
```sql
CREATE INDEX idx_category_price ON products(category, price);
CREATE INDEX idx_price ON products(price);
CREATE FULLTEXT INDEX idx_fulltext ON products(name, description);
```

### Example 4: Time-Series Data

**Table**:
```sql
CREATE TABLE sensor_readings (
    id BIGINT PRIMARY KEY,
    sensor_id INT,
    timestamp TIMESTAMP,
    value DECIMAL(10,2)
);
```

**Common Queries**:
```sql
-- 1. Recent readings for sensor
SELECT * FROM sensor_readings 
WHERE sensor_id = ? 
  AND timestamp > NOW() - INTERVAL '1 hour'
ORDER BY timestamp DESC;

-- 2. Aggregate by time range
SELECT DATE_TRUNC('hour', timestamp), AVG(value)
FROM sensor_readings
WHERE timestamp BETWEEN ? AND ?
GROUP BY DATE_TRUNC('hour', timestamp);
```

**Indexes**:
```sql
CREATE INDEX idx_sensor_time ON sensor_readings(sensor_id, timestamp);
CREATE INDEX idx_timestamp ON sensor_readings(timestamp);
```

## Database-Specific Considerations

### PostgreSQL

**Index Types**:
- **B-Tree**: Default, most common
- **Hash**: Exact matches only
- **GIN**: Full-text, arrays
- **GiST**: Geometric data, full-text
- **BRIN**: Block range indexes (large tables)

**Example**:
```sql
-- B-Tree (default)
CREATE INDEX idx_email ON users(email);

-- Hash
CREATE INDEX idx_email_hash ON users USING HASH(email);

-- GIN (full-text)
CREATE INDEX idx_content_gin ON articles USING GIN(to_tsvector('english', content));
```

**Partial Index**:
```sql
CREATE INDEX idx_active_email ON users(email) 
WHERE status = 'active';
```

### MySQL

**Storage Engines**:
- **InnoDB**: B+ Tree indexes
- **MyISAM**: B-Tree indexes
- **Memory**: Hash indexes

**Index Types**:
```sql
-- B-Tree (default)
CREATE INDEX idx_email ON users(email);

CREATE FULLTEXT INDEX idx_content ON articles(content);
```

### SQL Server

**Index Types**:
- **Clustered**: One per table
- **Non-clustered**: Multiple per table
- **Filtered**: Partial indexes

## Summary

### Key Points

1. **Indexes speed up reads** but slow down writes
2. **Choose indexes based on query patterns**
3. **Monitor index usage** and remove unused indexes
4. **Composite indexes** can be more efficient than multiple single-column indexes
5. **Column order matters** in composite indexes
6. **Balance**: More indexes = faster reads, slower writes
7. **Maintain**: Rebuild and update statistics regularly

### Decision Matrix

| Scenario | Action |
|----------|--------|
| Column in WHERE clause frequently | ✅ Create index |
| Foreign key column | ✅ Always index |
| High selectivity column | ✅ Good candidate |
| Low selectivity column | ❌ Usually skip |
| Small table (< 1000 rows) | ❌ Skip index |
| Frequently updated column | ⚠️ Consider carefully |
| Range queries | ✅ B-Tree index |
| Exact match only | ✅ Hash index |
| Text search | ✅ Full-text index |

### Best Practices Checklist

- [ ] Index primary keys (automatic)
- [ ] Index foreign keys
- [ ] Index frequently queried columns
- [ ] Use composite indexes for multi-column queries
- [ ] Order composite index columns by selectivity
- [ ] Consider partial indexes for filtered queries
- [ ] Monitor index usage
- [ ] Remove unused indexes
- [ ] Rebuild fragmented indexes
- [ ] Update statistics regularly

## Question

**How can we ensure short URLs are unique in a URL shortening system?**

At scale, this is really a question about **ID generation under concurrency**.

---

## Solution 1: Centralized ID Generator (Auto-increment / Sequence)

### How it works

* A single service (or DB sequence) generates monotonically increasing IDs
* IDs are Base62-encoded to produce short URLs

```
ID = next_sequence()
shortUrl = base62(ID)
```

### Why it works

* Strong uniqueness guarantee
* Simple mental model
* Naturally ordered

### Where it breaks

* Central point of coordination
* Throughput bottleneck
* High availability becomes hard

---

## Solution 2: Pre-generated Random IDs (Collision-Checked)

### How it works

* Generate random strings (e.g., 6–8 chars Base62)
* Check datastore for existence
* Retry on collision

```
key = randomBase62(7)
if exists(key): retry
```

### Why it works

* No coordination
* Horizontally scalable
* Simple to deploy

### Where it breaks

* Collision probability grows with scale
* Requires read-before-write
* Retry storms possible under high load

---

## Solution 3: Distributed ID Generation (Snowflake-style)

### How it works

* Each node generates IDs using:

    * timestamp
    * node ID
    * sequence counter

```
ID = timestamp | nodeId | sequence
shortUrl = base62(ID)
```

### Why it works

* No central bottleneck
* High throughput
* Globally unique by construction

### Where it breaks

* Clock synchronization issues
* Ordering is approximate, not strict
* More complex to operate

---

## Solution 4: Hashing the Original URL (with Salt)

### How it works

* Hash the long URL (e.g., SHA-256)
* Encode part of the hash as short URL
* Optionally add salt to handle collisions

```
hash = sha256(longUrl + salt)
shortUrl = base62(hash[0..N])
```

### Why it works

* Same URL → same short URL (idempotent)
* No coordination required
* Useful for deduplication

### Where it breaks

* Collisions still possible
* Difficult to change length later
* Reveals structure if not salted

---

## Tradeoff Summary (This Is the Real Answer)

| Approach             | Coordination | Ordering | Collision Risk | Ops Complexity      | Scalability |
| -------------------- | ------------ | -------- | -------------- | ------------------- | ----------- |
| Centralized Sequence | High         | Strong   | None           | Low → High at scale | Limited     |
| Random IDs           | None         | None     | Low → Medium   | Low                 | High        |
| Snowflake-style      | Low          | Approx   | None           | Medium              | Very High   |
| Hash-based           | None         | None     | Low            | Low                 | High        |

---

## Principal-Level Conclusion

> **There’s no perfect way to generate short URLs — only consciously chosen failure modes.**

* If you want **simplicity** → centralized sequence
* If you want **scale with minimal ops** → random IDs
* If you want **high throughput + no bottleneck** → Snowflake-style
* If you want **idempotency** → hashing

The *real* design question is:

> **Which pain can you afford — coordination, collisions, or operational complexity?**

### **Snowflake-style Distributed ID Generation**

## Why This Is the Right Default Choice

For a production URL shortener expected to scale beyond a single region, **uniqueness must not depend on coordination**.

A Snowflake-style ID generator gives us:

* **Guaranteed uniqueness**
* **No central bottleneck**
* **Predictable performance at high QPS**
* **Operationally controllable failure modes**

That combination is hard to beat.

---

## How It Works (Briefly)

Each short URL ID is composed of:

* **Timestamp** (for rough ordering)
* **Node ID** (to avoid cross-node collisions)
* **Sequence counter** (to handle bursts)

```
| timestamp | nodeId | sequence |
```

The resulting number is then **Base62 encoded** to produce a compact short URL.

---

## Why This Beats the Other Options

### vs Centralized Sequence

* Avoids a single point of failure
* Eliminates write contention
* No leader election or failover complexity

> A central ID generator is fine at low scale, but it becomes the first thing you have to redesign.

---

### vs Random IDs

* No collision checks
* No retries
* No read-before-write latency

> Random IDs push the collision problem into runtime and hope probability saves you.

---

### vs Hash-Based IDs

* Supports multiple short URLs for the same long URL if desired
* Easier to change length or format later
* No leakage of URL structure

> Hashing couples identity to input forever — that’s an irreversible decision.

---

## Failure Modes (Consciously Chosen)

This approach **does fail**, just predictably:

1. **Clock drift**

    * Mitigated by monotonic clocks or blocking on clock rollback

2. **Node ID misconfiguration**

    * Prevented by static assignment or coordination at startup

3. **Sequence exhaustion within a millisecond**

    * Handled by waiting for next timestamp

> These failures degrade throughput, not correctness.

# 🌍 Multi-Region URL Shortener (Snowflake IDs)

## Core Goal

> Generate **globally unique short URLs** across regions **without cross-region coordination**, while keeping **writes fast and availability high**.

---

## 1. ID Structure (Region-Aware Snowflake)

We extend the Snowflake ID to encode **region identity**.

```
| timestamp | regionId | nodeId | sequence |
```

Example (64-bit):

* 41 bits → timestamp
* 5 bits  → regionId (32 regions)
* 8 bits  → nodeId (256 nodes / region)
* 10 bits → sequence (1024 IDs/ms/node)

This gives:

* Global uniqueness
* Region-level isolation
* Predictable capacity

> **Key insight:** region identity is part of uniqueness, not metadata.

---

## 2. ID Generation (No Cross-Region Calls)

Each region:

* Generates IDs independently
* Uses its own regionId namespace
* Does **not** call a global service

This ensures:

* No latency tax
* No global bottleneck
* Region failure does not block others

> **If ID generation needs a network call, it will fail at scale.**

---

## 3. Write Path (Region-Local First)

### Flow

1. Client hits nearest region
2. Short URL is generated locally
3. Mapping stored in **region-local DB**
4. Asynchronous replication to other regions

This prioritizes:

* Write availability
* Low latency

> **We choose availability over immediate global consistency.**

---

## 4. Read Path (Global Resolution)

### Key requirement

> A short URL must resolve from *any* region.

### Strategy

* Global read replicas
* Or multi-region KV store (e.g., Dynamo-style)
* Or CDN-backed cache for hot URLs

Flow:

```
shortUrl → local cache → local store → fallback region
```

> **Reads are globally consistent eventually; users tolerate milliseconds, not failures.**

---

## 5. Handling Region Failure

### If a region goes down:

* Other regions continue generating IDs
* No collision risk (regionId partitioning)
* Reads fall back to replicas or origin region

> **Failure domain is the region, not the system.**

---

## 6. Clock Drift in Multi-Region

### Why it’s dangerous

* Timestamps overlap across regions

### Why it’s safe here

* regionId guarantees uniqueness even if clocks skew

### Still required

* Detect clock rollback per node
* Block ID generation until clock recovers

> **Clock drift hurts ordering, not correctness.**

---

## 7. Data Replication Strategy

We do **async replication**, because:

* Strong consistency adds cross-region latency
* Users don’t need read-after-write across continents

Replication options:

* CDC stream (Kafka)
* Native DB replication
* Event-driven sync

> **Short URLs are immutable — replication is simple.**

---

## 8. Caching Strategy (Critical)

* CDN caches shortUrl → longUrl mapping
* TTL tuned to traffic pattern
* Cache misses fall back to origin store

This:

* Shields databases
* Handles viral traffic spikes
* Reduces cross-region reads

> **Cache absorbs read amplification, Kafka absorbs write bursts.**

---

## 9. Hot URL Problem (Global)

### Problem

* One short URL goes viral globally

### Mitigation

* CDN edge caching
* Regional cache warming
* Rate limiting at edges

> **Hot keys move from DB to cache — intentionally.**

---

## 10. Failure Modes (Chosen, Not Accidental)

| Failure         | Behavior                          |
| --------------- | --------------------------------- |
| Region outage   | Writes succeed elsewhere          |
| Replication lag | Temporary 404s in distant regions |
| Clock skew      | Ordering affected, not uniqueness |
| Node exhaustion | Throughput degrades locally       |
| Cache eviction  | DB load increases gracefully      |

> **Every failure degrades performance, not correctness.**

---

## 11. Why This Design Works

This design:

* Avoids global coordination
* Limits blast radius
* Preserves uniqueness
* Keeps latency low
* Has reversible decisions

> **Multi-region success comes from isolating failure domains, not eliminating failure.**

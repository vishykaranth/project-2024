# ⭐ Principal-Level Whiteboard Answers — Spring Boot (1–100)

---

## 🔹 Architecture & Design

### 1️⃣ What Spring Boot really provides

> *“Spring Boot doesn’t add new capabilities — it reduces configuration friction. The real value is opinionated defaults and auto-configuration, which trade flexibility for speed.”*

---

### 2️⃣ When NOT to use Spring Boot

> *“Spring Boot is not ideal for ultra-low-latency or extremely memory-constrained environments. The convenience comes with startup and footprint costs.”*

---

### 3️⃣ Fat JAR vs thin JAR

> *“Fat JARs simplify deployment but increase startup time and image size. Thin JARs improve reuse and caching but complicate deployment.”*

---

### 4️⃣ Monolith vs microservice

> *“Spring Boot makes microservices easy to start, but distributed systems hard to operate. Architectural decisions must precede framework choice.”*

---

### 5️⃣ Layered architecture

> *“Controller → Service → Repository is a convention, not a law. What matters is dependency direction and testability, not layer count.”*

---

## 🔹 Startup & Performance

### 6️⃣ Slow startup issue

> *“Slow startup is usually classpath scanning and auto-configuration. Excluding unused starters and lazy initialization gives immediate wins.”*

---

### 7️⃣ Cold start in Kubernetes

> *“Cold starts combine JVM startup, Spring context creation, and dependency initialization. Container orchestration magnifies these costs.”*

---

### 8️⃣ Spring context bloat

> *“Large contexts increase startup time and memory. Every bean is a long-lived object — unused beans are a production liability.”*

---

### 9️⃣ Lazy initialization trade-off

> *“Lazy init improves startup but defers failures to runtime. It’s acceptable for non-critical beans, risky for core infrastructure.”*

---

### 🔟 Classpath scanning cost

> *“Classpath scanning scales with dependency count. Reducing starters often yields more benefit than JVM tuning.”*

---

## 🔹 Dependency Injection & Beans

### 1️⃣1️⃣ Constructor vs field injection

> *“Constructor injection enforces immutability and testability. Field injection hides dependencies and complicates testing.”*

---

### 1️⃣2️⃣ Circular dependencies

> *“Circular dependencies indicate poor design. Spring can mask them, but production failures become harder to reason about.”*

---

### 1️⃣3️⃣ Bean scope misuse

> *“Most beans should be singleton. Prototype beans increase memory churn and lifecycle complexity.”*

---

### 1️⃣4️⃣ @Configuration vs @Component

> *“@Configuration enables proxying and guarantees singleton semantics. Using @Component for config can cause subtle bugs.”*

---

### 1️⃣5️⃣ Conditional beans

> *“Conditional beans are powerful but can obscure runtime behavior. Observability is critical when conditions control wiring.”*

---

## 🔹 Web & REST

### 1️⃣6️⃣ Blocking vs non-blocking

> *“Spring MVC is blocking by default. Using it for IO-heavy workloads without tuning causes thread starvation.”*

---

### 1️⃣7️⃣ WebFlux decision

> *“WebFlux improves concurrency under IO wait but increases cognitive load. It’s not a universal performance upgrade.”*

---

### 1️⃣8️⃣ Thread pool exhaustion

> *“Servlet thread exhaustion manifests as hanging requests without errors. Timeouts and bulkheads are mandatory.”*

---

### 1️⃣9️⃣ Controller logic smell

> *“Business logic in controllers reduces testability and reuse. Controllers should translate protocols, not implement rules.”*

---

### 🔟 Error handling strategy

> *“Centralized exception handling ensures consistency. Leaking framework exceptions to clients couples APIs to implementation.”*

---

## 🔹 Data & Transactions

### 2️⃣1️⃣ Transaction boundaries

> *“Transactions should be scoped to business operations, not HTTP requests. Long transactions increase lock contention.”*

---

### 2️⃣2️⃣ @Transactional misuse

> *“@Transactional on private or self-invoked methods silently fails. This is a common production bug.”*

---

### 2️⃣3️⃣ Read-only transactions

> *“Read-only transactions optimize database behavior and prevent accidental writes. They are underused.”*

---

### 2️⃣4️⃣ N+1 query issue

> *“N+1 issues hide in happy-path testing and explode under load. Query plans reveal them faster than logs.”*

---

### 2️⃣5️⃣ Connection pool exhaustion

> *“Connection pool exhaustion causes cascading failures. Pool size must reflect downstream capacity, not CPU count.”*

---

## 🔹 Configuration & Profiles

### 2️⃣6️⃣ Externalized config

> *“Configuration must be environment-specific and runtime-injected. Rebuilding artifacts for config changes is an anti-pattern.”*

---

### 2️⃣7️⃣ Profile misuse

> *“Too many profiles create combinatorial complexity. Profiles should model environments, not features.”*

---

### 2️⃣8️⃣ Config drift

> *“Config drift is harder to detect than code drift. Centralized config with auditability is essential.”*

---

### 2️⃣9️⃣ Secrets management

> *“Secrets should never live in properties files. Integration with secret stores is a baseline requirement.”*

---

### 🔟 Refresh scope risk

> *“Runtime refresh increases flexibility but risks partial reconfiguration. It must be applied carefully.”*

---

## 🔹 Observability

### 3️⃣1️⃣ Logging pitfalls

> *“Synchronous logging can block request threads. Logging strategy affects latency and throughput.”*

---

### 3️⃣2️⃣ Log levels in prod

> *“Debug logging in production increases allocation rate and CPU. Logging is a performance decision.”*

---

### 3️⃣3️⃣ Metrics first

> *“Metrics reveal failure modes before logs do. Logs explain failures; metrics detect them.”*

---

### 3️⃣4️⃣ Tracing overhead

> *“Distributed tracing adds latency and storage cost. Sampling must be deliberate.”*

---

### 3️⃣5️⃣ Health checks misuse

> *“Health endpoints should reflect service health, not dependency health. Otherwise restarts amplify outages.”*

---

## 🔹 Resilience & Reliability

### 3️⃣6️⃣ Retry storms

> *“Retries without backoff amplify failures. Spring Retry must be bounded and observable.”*

---

### 3️⃣7️⃣ Circuit breakers

> *“Circuit breakers protect systems, not services. Incorrect thresholds cause self-inflicted outages.”*

---

### 3️⃣8️⃣ Timeout discipline

> *“Timeouts are mandatory at every boundary. A missing timeout is a latent outage.”*

---

### 3️⃣9️⃣ Bulkheads

> *“Bulkheads isolate failure domains. Shared thread pools defeat resilience.”*

---

### 🔟 Graceful shutdown

> *“Graceful shutdown prevents dropped requests and duplicate processing. It’s essential in Kubernetes.”*

---

## 🔹 Testing

### 4️⃣1️⃣ @SpringBootTest misuse

> *“@SpringBootTest loads the entire context and slows tests. Slice tests provide faster feedback.”*

---

### 4️⃣2️⃣ Mocking strategy

> *“Mock external boundaries, not internal collaborators. Over-mocking hides integration issues.”*

---

### 4️⃣3️⃣ Testcontainers value

> *“Testcontainers reduce environment drift and uncover real integration failures early.”*

---

### 4️⃣4️⃣ Contract testing

> *“Contract tests prevent breaking changes across teams. They shift failures left.”*

---

### 4️⃣5️⃣ Flaky tests

> *“Flaky tests indicate hidden coupling or timing assumptions. They erode trust in CI.”*

---

## 🔹 Security

### 4️⃣6️⃣ Spring Security complexity

> *“Spring Security is powerful but easy to misconfigure. Defaults are safer than custom filters.”*

---

### 4️⃣7️⃣ Filter ordering

> *“Incorrect filter ordering leads to silent security gaps. Security must be observable.”*

---

### 4️⃣8️⃣ Method-level security

> *“Method-level security enforces business rules closer to data. It complements, not replaces, endpoint security.”*

---

### 4️⃣9️⃣ Auth vs authz

> *“Authentication proves identity; authorization enforces access. Mixing them creates brittle security.”*

---

### 🔟 Security testing

> *“Security failures rarely appear in happy-path tests. Negative testing is essential.”*

---

## 🔹 Deployment & Ops

### 5️⃣1️⃣ Docker image size

> *“Large images increase cold start and network cost. Layering and dependency trimming matter.”*

---

### 5️⃣2️⃣ JVM flags in containers

> *“JVM defaults assume bare metal. Container-aware memory settings are mandatory.”*

---

### 5️⃣3️⃣ Readiness vs liveness

> *“Readiness protects users; liveness protects the platform. Mixing them causes restart storms.”*

---

### 5️⃣4️⃣ Blue-green vs rolling

> *“Rolling deploys reduce risk but complicate backward compatibility. Schema evolution must align.”*

---

### 5️⃣5️⃣ Feature flags

> *“Feature flags decouple deployment from release. Poor flag hygiene creates permanent complexity.”*

---

## 🔹 Scalability & Failure Modes

### 5️⃣6️⃣ Horizontal scaling limits

> *“Stateful components limit horizontal scaling. Externalizing state is required for elasticity.”*

---

### 5️⃣7️⃣ Cache misuse

> *“Caches hide latency but increase inconsistency risk. Cache invalidation is a correctness problem.”*

---

### 5️⃣8️⃣ Memory leaks

> *“Memory leaks often come from static references, listeners, or caches. Heap dumps reveal them.”*

---

### 5️⃣9️⃣ CPU spikes

> *“CPU spikes often correlate with allocation pressure or contention, not business logic.”*

---

### 🔟 Thread starvation

> *“Thread starvation manifests as latency, not errors. Monitoring queue depth is critical.”*

---

## 🔹 Governance & Ownership

### 6️⃣1️⃣ Framework upgrades

> *“Spring upgrades should be treated as migrations. Behavior changes matter more than API changes.”*

---

### 6️⃣2️⃣ Dependency explosion

> *“Every dependency expands the attack and failure surface. Dependency hygiene is architectural work.”*

---

### 6️⃣3️⃣ Platform vs product code

> *“Mixing platform concerns into product code slows teams. Separation enables scale.”*

---

### 6️⃣4️⃣ Opinionated defaults

> *“Defaults accelerate teams but must be documented. Undocumented conventions become tribal knowledge.”*

---

### 6️⃣5️⃣ Principal takeaway

> *“Spring Boot accelerates delivery, but reliability depends on discipline, not annotations.”*

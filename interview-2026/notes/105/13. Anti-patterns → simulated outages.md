## 1️⃣ Hot Kafka Partition → Downstream Meltdown

**Anti-pattern:**

> Assuming consumer scaling fixes throughput

**Scenario:**

* 12 partitions, 12 consumers
* One partition lag grows to millions
* DB write latency spikes
* Upstream retries increase

**Your Principal Moves:**

1. Identify hot key via partition metrics
2. Pause traffic for that key range
3. Migrate keys to new partitions
4. Add future hash-based key strategy

**What You Say:**

> “Adding consumers won’t help. Partitions define concurrency.”

---

## 2️⃣ Auto-Commit Offset → Silent Data Loss

**Anti-pattern:**

> Trusting Kafka auto-commit

**Scenario:**

* Consumer crashes mid-processing
* Offset already committed
* Missing DB records reported days later

**Your Principal Moves:**

1. Disable auto-commit
2. Commit offset **after** durable DB write
3. Add idempotent writes
4. Add replay guardrails

**What You Say:**

> “Offset commit is not processing completion.”

---

## 3️⃣ Retry Storm → DB Collapse

**Anti-pattern:**

> Retry without backoff

**Scenario:**

* Downstream DB slows
* All services retry immediately
* Connection pool exhaustion
* System fully down

**Your Principal Moves:**

1. Stop retries temporarily
2. Enable exponential backoff + jitter
3. Add circuit breakers
4. Cap retry budget

**What You Say:**

> “Retries amplify failure when the dependency is unhealthy.”

---

## 4️⃣ Liveness Misuse → Kubernetes Restart Loop

**Anti-pattern:**

> Liveness tied to dependency health

**Scenario:**

* Kafka broker blips
* All pods restart
* Cold start overloads cluster
* Total outage

**Your Principal Moves:**

1. Fix probes: liveness ≠ readiness
2. Make app degrade gracefully
3. Add startup probes
4. Rate-limit restarts

**What You Say:**

> “We killed healthy processes during a dependency outage.”

---

## 5️⃣ Logging Flood → Latency Spike

**Anti-pattern:**

> Logging as debugging tool

**Scenario:**

* Error rate increases
* Log volume explodes
* Disk IO saturated
* Latency jumps 10×

**Your Principal Moves:**

1. Reduce log level immediately
2. Sample logs
3. Move to structured logs
4. Add metrics instead of logs

**What You Say:**

> “Logs are a performance decision.”

---

## 6️⃣ Flyway at Startup → Failed Rollout

**Anti-pattern:**

> Schema migration on app start

**Scenario:**

* New column added
* Old code still running
* Startup blocks
* Rollback impossible

**Your Principal Moves:**

1. Roll back app
2. Apply backward-compatible schema first
3. Decouple migrations from deploy
4. Use expand-contract pattern

**What You Say:**

> “Schema changes are the hardest deploy.”

---

## 7️⃣ Thread Pool Starvation → 503 Storm

**Anti-pattern:**

> Blocking calls everywhere

**Scenario:**

* Spring MVC app
* External API slows
* All request threads blocked
* Service appears “up” but unusable

**Your Principal Moves:**

1. Identify blocking dependency
2. Add timeouts
3. Isolate thread pools
4. Move async/reactive where needed

**What You Say:**

> “Throughput collapsed due to thread starvation, not CPU.”

---

## 8️⃣ DLQ as Graveyard → Infinite Data Loss

**Anti-pattern:**

> DLQ without ownership

**Scenario:**

* Messages routed to DLQ
* No alerts
* Business reports missing data weeks later

**Your Principal Moves:**

1. Alert on DLQ volume
2. Add reprocessing pipeline
3. Assign ownership
4. Track DLQ SLO

**What You Say:**

> “DLQs are a process, not a feature.”

---

## 9️⃣ Shared DB → Cascading Failure

**Anti-pattern:**

> Shared resources for unrelated services

**Scenario:**

* Reporting job spikes load
* Core API slows
* Kafka consumers back up
* Full system degradation

**Your Principal Moves:**

1. Throttle batch jobs
2. Separate DBs or replicas
3. Enforce query budgets
4. Isolate workloads

**What You Say:**

> “Shared resources erase failure boundaries.”

---

## 🔟 Replay Without Guards → Double Writes

**Anti-pattern:**

> Blind Kafka replays

**Scenario:**

* Replay initiated
* Downstream writes duplicated
* Financial impact

**Your Principal Moves:**

1. Halt replay
2. Add idempotency keys
3. Add replay mode flag
4. Reconcile data

**What You Say:**

> “Replays are outages unless designed safely.”

# 🧩 Kafka System Design (Whiteboard Style)

## Problem statement (typical)

> “Design an event-driven system using Kafka for order processing.”

---

## 1️⃣ Clarify requirements (say this out loud)

**Functional**

* Capture order lifecycle events
* Guarantee ordering per order
* Support retries & reprocessing
* Multiple downstream consumers

**Non-functional**

* High throughput (10k–100k events/sec)
* No data loss
* Scalable consumers
* Observability

👉 Interviewer wants to see you think **before drawing**.

---

## 2️⃣ High-level architecture (draw this)

```
[Order Service]
      |
      v
  Kafka Topic (orders)
      |
  +---+----+------+
  |        |      |
[Billing] [Shipping] [Analytics]
```

**Say:**

> “Kafka decouples producers from consumers and allows independent scaling.”

---

## 3️⃣ Topic & partition design (key part)

### Topic

```
orders.events
```

### Partitioning

* Key = `orderId`
* Guarantees ordering per order

**Draw:**

```
orders.events
P0  P1  P2  P3
 |   |   |   |
```

**Say:**

> “Ordering is guaranteed per order because all events for an order go to the same partition.”

---

## 4️⃣ Producer design (important signals)

**Config highlights**

* `acks=all`
* `retries=∞`
* `enable.idempotence=true`
* Compression = `lz4` or `snappy`

**Say:**

> “Idempotent producers prevent duplicates during retries.”

---

## 5️⃣ Consumer group design

Each service = **its own consumer group**.

```
orders.events
   |
+--+--+--+
|     |     |
Billing  Shipping  Analytics
(CG1)    (CG2)      (CG3)
```

**Say:**

> “Each consumer group gets all events independently.”

---

## 6️⃣ Offset management (this is where seniors shine)

**Strategy**

* Manual commits
* Commit **after** successful processing

**Say:**

> “This gives at-least-once semantics; consumers must be idempotent.”

---

## 7️⃣ Failure handling & retries

### Transient failures

* Consumer retries with backoff

### Poison messages

* Dead Letter Topic (DLT)

```
orders.events  -->  orders.dlt
```

**Say:**

> “We isolate bad events to avoid blocking the partition.”

---

## 8️⃣ Exactly-once (if asked)

If Kafka → Kafka:

* Use **transactions**

If Kafka → DB:

* Use **idempotent writes** or **outbox pattern**

**Say:**

> “Kafka EOS doesn’t automatically cover external systems.”

---

## 9️⃣ Retention & replay

**Config**

* Retention = 7 days
* Enable replay for recovery or backfill

**Say:**

> “Kafka acts as both a transport and a short-term system of record.”

---

## 🔟 Observability & ops (often skipped, big bonus)

* Consumer lag monitoring
* Broker disk usage
* Under-replicated partitions
* Alert on ISR shrink

**Say:**

> “Most Kafka issues show up first as consumer lag.”

---

## 🧠 Final whiteboard summary (what you point to)

```
Producer → Topic (partitioned by orderId)
        → Multiple consumer groups
        → Idempotent processing
        → DLT for failures
        → Replay via retention
```

# ❌ Common Kafka Design Mistakes (That Fail Interviews)

---

## 1️⃣ Treating Kafka like a queue

**❌ Mistake**

> “Once a consumer reads a message, it’s gone.”

**Why it fails**

* Kafka is a **log**, not a queue
* Multiple consumers can read the same data

**✅ Say instead**

> “Kafka stores events durably and consumers track their own offsets.”

---

## 2️⃣ Ignoring partitioning strategy

**❌ Mistake**

> “We’ll just let Kafka distribute messages evenly.”

**Why it fails**

* Breaks ordering
* Causes hot partitions

**✅ Say instead**

> “We partition by business key to guarantee ordering and avoid skew.”

---

## 3️⃣ Claiming global ordering

**❌ Mistake**

> “Kafka guarantees message ordering.”

**Why it fails**

* Only **per partition**, not per topic

**✅ Say instead**

> “Ordering is guaranteed only within a partition.”

---

## 4️⃣ Using auto-commit offsets blindly

**❌ Mistake**

> “Kafka handles offsets automatically, so we’re safe.”

**Why it fails**

* Causes message loss on crashes

**✅ Say instead**

> “We commit offsets after successful processing.”

---

## 5️⃣ Saying “Kafka is exactly-once everywhere”

**❌ Mistake**

> “Kafka guarantees exactly-once delivery.”

**Why it fails**

* Only within Kafka
* External systems break this

**✅ Say instead**

> “Exactly-once applies within Kafka; external systems need idempotency.”

---

## 6️⃣ No plan for poison messages

**❌ Mistake**

> “We’ll just retry until it works.”

**Why it fails**

* One bad message can block a partition forever

**✅ Say instead**

> “We use retries with backoff and a dead-letter topic.”

---

## 7️⃣ Forgetting consumer lag & backpressure

**❌ Mistake**

> “Kafka can handle high throughput, so we’re fine.”

**Why it fails**

* Consumers are often the bottleneck

**✅ Say instead**

> “We monitor consumer lag and scale consumers or partitions accordingly.”

---

## 8️⃣ No failure story

**❌ Mistake**

> *Blank stare* when asked about broker failure

**Why it fails**

* Distributed systems **fail constantly**

**✅ Say instead**

> “On broker failure, leaders are re-elected from ISR and consumers recover automatically.”

---

## 9️⃣ No replay or recovery strategy

**❌ Mistake**

> “If a consumer fails, we redeploy it.”

**Why it fails**

* No recovery from bad code or data bugs

**✅ Say instead**

> “Kafka retention allows replay by resetting offsets.”

---

## 🔟 Overengineering or underengineering

**❌ Mistake**

> Using Kafka for simple CRUD or request-response

**Why it fails**

* Shows poor tool judgment

**✅ Say instead**

> “Kafka is best for event streaming, not low-latency synchronous workflows.”

---

## 🚨 Interviewer red flags (instant fail vibes)

* “Kafka guarantees no duplicates”
* “Ordering is global”
* “Offsets are stored in ZooKeeper”
* “Kafka replaces databases”
* “We don’t need monitoring”

---

## 🎯 How to recover if you slip

If you say something wrong, **self-correct fast**:

> “Let me rephrase that—Kafka guarantees ordering per partition, not globally.”

Interviewers LOVE this.

---

## TL;DR

Kafka interview failures usually come from:

* Wrong mental model
* Ignoring failures
* Overclaiming guarantees

Avoid these, and you’ll stand out.

# 🧠 Model Answers — Top 25 Hardest Spring Boot Postmortems

---

## 1️⃣ Thread Pool Exhaustion (Blocking Calls)

**Postmortem Answer**

> *“The service didn’t crash — it stalled. Servlet threads were blocked on downstream calls without timeouts, exhausting the pool and causing request pileup. Kafka, DB, and HTTP calls all competed for the same threads. Immediate fix was enforcing timeouts and isolating thread pools. Long term, we added bulkheads and removed blocking calls from request threads.”*

---

## 2️⃣ Connection Pool Exhaustion

> *“The database wasn’t slow — we were holding connections too long due to missing transaction boundaries. Once the pool saturated, the service deadlocked itself. We reduced transaction scope immediately and capped pool size. Long-term, we added connection leak detection and aligned pool size with DB capacity.”*

---

## 3️⃣ @Transactional Self-Invocation Bug

> *“The transaction annotation never applied because the method was invoked internally. Failures were silently committed. We verified via logs and AOP proxy inspection. Immediate fix was refactoring the call path. Long-term, we banned self-invoked transactional methods via static analysis.”*

---

## 4️⃣ Retry Storm Causing Cascade Failure

> *“Retries amplified a downstream outage instead of masking it. All services retried simultaneously with no backoff, overwhelming dependencies. We disabled retries immediately. Long-term, we enforced exponential backoff, jitter, and circuit breakers at every boundary.”*

---

## 5️⃣ Health Check Restart Storm

> *“Health checks reflected dependency health instead of service health. When one dependency flapped, Kubernetes restarted healthy pods, worsening the outage. Immediate fix was decoupling liveness from readiness. Long-term, we standardized health semantics platform-wide.”*

---

## 6️⃣ JVM OOM in Containers

> *“The JVM assumed host memory, not container limits, and exceeded the pod memory. The fix was container-aware heap sizing. Long-term, we templated JVM flags and enforced memory budgets in CI.”*

---

## 7️⃣ Lazy Loading Outside Transaction

> *“Hibernate lazy loading triggered DB access after the transaction closed, causing runtime failures. We mitigated by eager-fetching critical paths. Long-term, we enforced DTO mapping inside service layers.”*

---

## 8️⃣ N+1 Query Explosion

> *“N+1 queries passed tests but collapsed under scale. DB CPU spiked while app metrics looked healthy. We fixed it with fetch joins and batch sizing. Long-term, we added query count alerts and load tests.”*

---

## 9️⃣ Blocking WebFlux Application

> *“The app was reactive in name only — JDBC calls blocked event-loop threads. Latency skyrocketed under load. Immediate fix was moving blocking calls to bounded elastic schedulers. Long-term, we enforced non-blocking contracts in reactive services.”*

---

## 🔟 Logging-Induced Latency

> *“Synchronous logging blocked request threads during peak load. Log volume increased CPU and IO pressure. Immediate fix was async logging and log level reduction. Long-term, we treated logging as a performance decision.”*

---

## 1️⃣1️⃣ Schema Migration at Startup

> *“Schema migration ran on startup, blocking pods during deploy. Traffic shifted to fewer pods, causing overload. We decoupled migrations immediately. Long-term, schema changes became explicit, versioned pipelines.”*

---

## 1️⃣2️⃣ Circuit Breaker Misconfiguration

> *“The breaker tripped too aggressively and caused self-inflicted outages. Metrics showed healthy downstreams. We tuned thresholds immediately. Long-term, we aligned breaker configs with real SLAs.”*

---

## 1️⃣3️⃣ GC Pressure Latency Spike

> *“Latency was driven by allocation pressure, not CPU. Short-lived objects overwhelmed GC. We reduced allocations and tuned GC. Long-term, we added allocation profiling to performance testing.”*

---

## 1️⃣4️⃣ Config Drift Across Environments

> *“Prod behaved differently because config wasn’t versioned or audited. We diffed configs to identify drift. Long-term, we centralized config with audit trails.”*

---

## 1️⃣5️⃣ Read Replica Overload

> *“Reads were routed to replicas without backpressure, overwhelming them. We throttled reads immediately. Long-term, we separated read traffic and enforced query budgets.”*

---

## 1️⃣6️⃣ Graceful Shutdown Missing

> *“Pods terminated while processing requests, causing partial writes and retries. Immediate fix was enabling graceful shutdown. Long-term, it became a deployment requirement.”*

---

## 1️⃣7️⃣ ThreadLocal Memory Leak

> *“ThreadLocal values weren’t cleared in pooled threads, causing memory growth. Heap dumps confirmed retention. We fixed cleanup and banned ThreadLocal usage without lifecycle guarantees.”*

---

## 1️⃣8️⃣ Actuator Endpoint Exposure

> *“Sensitive actuator endpoints were publicly accessible. We restricted access immediately. Long-term, actuator exposure became security-reviewed configuration.”*

---

## 1️⃣9️⃣ Idempotency Missing

> *“Retries caused duplicate writes because operations weren’t idempotent. We patched deduplication immediately. Long-term, idempotency keys became mandatory for write APIs.”*

---

## 2️⃣0️⃣ Servlet Timeout Mismatch

> *“Ingress timed out before the app, leaving hanging threads. We aligned timeouts immediately. Long-term, timeout budgets were documented end-to-end.”*

---

## 2️⃣1️⃣ Cache Invalidation Bug

> *“Cache hid DB writes inconsistently, causing stale reads. We disabled cache temporarily. Long-term, we formalized cache ownership and invalidation rules.”*

---

## 2️⃣2️⃣ Feature Flag Left Enabled

> *“A temporary flag became permanent technical debt and failed under scale. We rolled it back immediately. Long-term, flags got owners and expiry dates.”*

---

## 2️⃣3️⃣ Dependency Auto-Config Surprise

> *“Adding a starter enabled unexpected auto-configuration in prod. We excluded it immediately. Long-term, dependency changes required architecture review.”*

---

## 2️⃣4️⃣ Thread Pool Sharing

> *“Critical and non-critical workloads shared the same executor, allowing low-priority work to starve high-priority traffic. We isolated pools immediately. Long-term, thread pools were explicitly defined.”*

---

## 2️⃣5️⃣ Silent Partial Failures

> *“Failures were swallowed and logged but not surfaced. The system degraded quietly. We added metrics and alerts. Long-term, errors had to be observable or fail fast.”*

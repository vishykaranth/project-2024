## 🧨 1️⃣ ISR Shrink → Unclean Leader Election → Data Loss

### What happened

* One broker went down
* ISR for some partitions shrank to 1
* Second broker failed
* Kafka elected an **out-of-sync replica** as leader

### Root cause

* `unclean.leader.election.enable=true`

### Impact

* Silent data loss
* No alerts triggered
* Corrupted downstream state

### Lesson (say this in interviews)

> “We now prefer availability loss over silent data loss and disable unclean leader election.”

🔥 **Staff signal:** correctness > uptime.

---

## 🧨 2️⃣ Consumer Group Rebalance Storm

### What happened

* Autoscaling consumers aggressively
* Frequent joins/leaves
* Continuous rebalances

### Root cause

* Eager rebalancing
* Short session timeouts

### Impact

* Consumption paused repeatedly
* Lag spiked
* Downstream SLAs violated

### Fix

* Cooperative rebalancing
* Sticky assignor
* Reduced autoscaling aggressiveness

### Lesson

> “Rebalances are a latency event — not a free operation.”

---

## 🧨 3️⃣ Hot Partition Took Down the System

### What happened

* Partition key = `country`
* One country produced 80% of traffic

### Root cause

* Poor partition key choice
* No monitoring on per-partition lag

### Impact

* One partition maxed CPU
* Others idle
* Consumers appeared “healthy” globally

### Fix

* Repartitioned using `orderId`
* Added per-partition lag alerts

### Lesson

> “Even distribution matters more than semantic convenience.”

---

## 🧨 4️⃣ Disk Filled → Brokers Started Dropping Writes

### What happened

* Retention misconfigured
* No disk usage alerts
* Brokers hit disk full

### Root cause

* Infinite retention on high-volume topics

### Impact

* Producers blocked
* Backpressure cascaded across services

### Fix

* Enforced retention policies
* Added disk utilization SLOs

### Lesson

> “Kafka failures often start as storage problems.”

---

## 🧨 5️⃣ Offset Committed Too Early → Data Loss

### What happened

* Auto-commit enabled
* Consumer crashed mid-processing

### Root cause

* Offsets committed before business logic completed

### Impact

* Messages skipped
* Inconsistent state

### Fix

* Manual offset commits
* Idempotent consumers

### Lesson

> “Offset management is application logic, not Kafka magic.”

---

## 🧨 6️⃣ Schema Evolution Broke All Consumers

### What happened

* Producer removed a field
* Consumers expected it
* Deserialization failed at scale

### Root cause

* No schema compatibility checks

### Impact

* Entire pipeline stalled
* Emergency rollback

### Fix

* Schema Registry
* Backward compatibility enforcement

### Lesson

> “Schemas are contracts, not suggestions.”

---

## 🧨 7️⃣ MirrorMaker Lag Caused Cross-Region Inconsistency

### What happened

* Region A → Region B replication lagged
* Consumers in Region B processed stale data

### Root cause

* Assumed MirrorMaker was real-time
* No replication lag monitoring

### Impact

* Inconsistent user experiences

### Fix

* Treated replication as async
* Added region-awareness to consumers

### Lesson

> “Multi-region Kafka is eventually consistent by design.”

---

## 🧨 8️⃣ Zookeeper / Controller Flapping

### What happened

* Zookeeper GC pauses
* Controller leadership kept changing

### Root cause

* Undersized Zookeeper cluster

### Impact

* Topic creation failures
* Metadata timeouts

### Fix

* Dedicated Zookeeper nodes
* Later migrated to KRaft

### Lesson

> “Metadata stability is as critical as data throughput.”

## 🧠 How to talk about outages in interviews

Use this format:

> **“We saw X. Root cause was Y. We fixed it by Z. The long-term change was W.”**

Example:

> “We saw consumer lag spikes. Root cause was rebalance storms. We fixed it with cooperative rebalancing and changed our autoscaling strategy.”

That sounds like **real ownership**.

---

## 🚨 Instant staff-level red flags

* “Kafka lost messages randomly”
* “We just increased partitions”
* “We restarted the cluster”
* “It never happened again”

Interviewers want **learning**, not heroics.

---

## 🎯 TL;DR

Most Kafka outages are caused by:

* Misunderstood guarantees
* Poor defaults
* Missing observability
* Human assumptions

Staff/principal engineers **design for failure first**.

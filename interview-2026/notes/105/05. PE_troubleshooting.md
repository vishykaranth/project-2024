# 🧠 Model Answers — Top 20 Hardest Postmortem Questions

I’ll keep a consistent structure:

> **Signal → Root cause → Fix → Long-term change**

---

## 1️⃣ JVM CPU at 100% — how did you debug it?

**Model answer:**

> We correlated CPU spikes with increased allocation rate using JFR. Thread dumps showed excessive contention caused by synchronized logging. We fixed it by removing blocking log appenders and reduced allocation pressure. Long term, we added CPU profiling to our on-call playbook.

---

## 2️⃣ OutOfMemoryError but heap wasn’t full — explain.

**Model answer:**

> It was native memory exhaustion due to direct buffers and thread stacks. Heap metrics looked fine, but RSS kept growing. We fixed it by tuning direct memory limits and reducing thread count. Long term, we monitor native memory, not just heap.

---

## 3️⃣ Thread pool exhaustion — how did it manifest?

**Model answer:**

> Requests stalled without errors while CPU remained low. Thread dumps showed all request threads blocked on downstream calls. We added timeouts and bulkheads. Long term, we standardized thread pool sizing and alert on queue depth.

---

## 4️⃣ Blocking JDBC call inside async flow — impact?

**Model answer:**

> Blocking calls consumed async executor threads, causing starvation and request pile-up. We fixed it by isolating blocking IO to a dedicated pool. Long term, we enforced non-blocking checks in code reviews.

---

## 5️⃣ Consumer lag kept increasing — root cause?

**Model answer:**

> Lag increased because a hot partition concentrated traffic on a single consumer thread. We detected it via per-partition lag metrics. We repartitioned using a better key and added skew detection alerts.

---

## 6️⃣ Offset committed too early — how was data lost?

**Model answer:**

> Offsets were auto-committed before processing completed, so crashes caused messages to be skipped. We moved to manual commits after successful processing and made consumers idempotent. Long term, offset handling became part of our design checklist.

---

## 7️⃣ Exactly-once didn’t work — why?

**Model answer:**

> Exactly-once applied only within Kafka, but our database writes were not idempotent. We fixed it using the outbox pattern. Long term, we stopped claiming EOS across system boundaries.

---

## 8️⃣ Kafka disk filled — what broke first?

**Model answer:**

> Producers started blocking due to log append failures, which cascaded into upstream service timeouts. We fixed retention misconfiguration and added disk usage alerts. Long term, storage SLOs became mandatory for Kafka clusters.

---

## 9️⃣ Schema evolution broke consumers — why?

**Model answer:**

> A required field was removed without backward compatibility checks. Consumers failed deserialization. We enforced schema registry compatibility rules and added CI checks. Long term, schemas became versioned contracts.

---

## 🔟 Container OOMKilled but memory looked fine — why?

**Model answer:**

> The container exceeded its memory limit due to native memory usage, not heap. Kubernetes killed it even though JVM heap looked healthy. We fixed it by aligning JVM memory flags with container limits. Long term, we standardized container-aware JVM configs.

---

## 1️⃣1️⃣ Container CPU throttling — symptoms?

**Model answer:**

> Latency increased while CPU metrics looked capped. The container was being throttled due to low CPU limits. We adjusted requests and limits and added throttling metrics. Long term, we stopped under-provisioning CPU for JVM services.

---

## 1️⃣2️⃣ Pod evictions caused outage — why?

**Model answer:**

> Nodes experienced memory pressure and evicted pods without sufficient requests. We fixed resource requests and added pod priority classes. Long term, we treated resource requests as reliability guarantees, not suggestions.

---

## 1️⃣3️⃣ Liveness probe killed healthy service — explain.

**Model answer:**

> The liveness probe was tied to downstream dependencies, so temporary slowness caused restarts. We decoupled liveness from readiness. Long term, probes were reviewed as part of production readiness.

---

## 1️⃣4️⃣ HPA scaled incorrectly — why?

**Model answer:**

> HPA used CPU metrics, but the bottleneck was IO wait. Scaling added pods without improving throughput. We fixed it by using custom metrics based on request latency. Long term, scaling signals were aligned with real bottlenecks.

---

## 1️⃣5️⃣ DB CPU at 100% — root cause?

**Model answer:**

> A missing index caused full table scans after a data growth threshold. We identified it via query plans. We added the index and query guards. Long term, we added query plan regression checks.

---

## 1️⃣6️⃣ Connection pool exhaustion — explain.

**Model answer:**

> Slow queries held connections longer, exhausting the pool. Requests piled up without errors. We fixed slow queries and capped max pool size. Long term, pool metrics became first-class alerts.

---

## 1️⃣7️⃣ Autovacuum caused latency spikes — why?

**Model answer:**

> Long-running transactions prevented vacuum cleanup, leading to aggressive vacuum runs. We fixed transaction boundaries and tuned vacuum settings. Long term, we monitored transaction age, not just query time.

---

## 1️⃣8️⃣ Cascading failure across services — how did it start?

**Model answer:**

> A single downstream timeout caused retries that amplified load across services. We fixed it with timeouts, circuit breakers, and backpressure. Long term, retries were treated as a reliability risk.

---

## 1️⃣9️⃣ “Everything looked green” but users were down — explain.

**Model answer:**

> Infrastructure metrics were healthy, but a functional dependency failed silently. We added synthetic user journeys and end-to-end SLIs. Long term, user-centric monitoring replaced component-only dashboards.

---

## 2️⃣0️⃣ Idempotency missing — consequence?

**Model answer:**

> Retries caused duplicate side effects, corrupting business state. We fixed it with idempotent writes and deduplication keys. Long term, idempotency became mandatory for all externally visible operations.

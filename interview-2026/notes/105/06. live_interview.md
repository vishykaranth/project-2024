## ✅ Model Answer / Walkthrough

### 1️⃣ Identify the root cause

* The consumer **crashed after reading messages but before completing processing**.
* If **auto-commit offsets** is enabled or offsets are committed **before processing**, Kafka thinks those messages were already processed.
* On restart, consumer resumes from the committed offset → **messages skipped**.
* This is **classic at-least-once vs exactly-once semantics** confusion.

**Diagram (mental whiteboard):**

```
Consumer reads: M1, M2, M3
Process M1 -> OK
Commit offset -> done
Process M2 -> CRASH
Restart consumer -> starts at offset after M2
=> M2 skipped
```

---

### 2️⃣ How to verify

* Check consumer configuration: `enable.auto.commit=true`?
* Inspect **committed offsets** in Kafka (`kafka-consumer-groups.sh`)
* Correlate consumer logs with offsets to see which messages were skipped
* Review transaction / write logs in Postgres

---

### 3️⃣ Immediate mitigation

* Switch to **manual offset commits**: commit only **after successful processing**
* Make consumer **idempotent** to handle potential retries safely
* Restart consumer carefully; optionally **replay skipped offsets** using `--reset-offsets`

---

### 4️⃣ Long-term solution

* Enforce **at-least-once semantics with idempotent processing**
* Use **transactions** if Kafka → Kafka → DB path is needed (exactly-once)
* Add **consumer lag and offset mismatch monitoring**
* Add **synthetic verification**: ensure all messages are processed exactly once
* Code review and CI to ensure all new consumers follow **manual commit + idempotency pattern**

1. **Can you achieve exactly-once with DB writes?**

    * Yes, using **Kafka transactions or outbox pattern**, but only if DB writes are idempotent or transactional.

2. **What happens if a consumer fails mid-batch?**

    * With manual commit, unprocessed messages are **reprocessed** safely.

3. **How do you monitor offset correctness in prod?**

    * Use **consumer lag per partition** + **offset difference alerts** + end-to-end SLOs.

## ✅ Model Principal-Level Answer

### 1️⃣ What is happening (root cause)

* Kafka guarantees **ordering per partition**
* One partition is **hot**
* That partition is handled by **exactly one consumer thread**
* That consumer becomes CPU-bound
* Lag increases **only for that partition**

Key insight:

> **Kafka parallelism = number of partitions, not consumers**

---

### 2️⃣ Why adding consumers didn’t help

* Consumers > partitions ⇒ **idle consumers**
* Hot partition **cannot be split**
* One consumer is the bottleneck regardless of scale

Interview phrase to use:

> “This is a data-skew problem, not a scaling problem.”

---

### 3️⃣ Why cascading failures occur

#### Upstream

* Lag increases
* Producers block due to:

    * `acks=all`
    * `max.in.flight` exhaustion
* Producer latency spikes → timeouts

#### Downstream

* Consumer thread slows
* DB writes pile up
* Connection pool saturates
* Requests block → thread pool exhaustion

**Whiteboard cascade:**

```
Hot partition
   ↓
Slow consumer
   ↓
Lag ↑
   ↓
Producer backpressure
   ↓
DB pool exhaustion
   ↓
System-wide latency spike
```

---

### 4️⃣ Immediate mitigation (on-call actions)

In order:

1. **Rate-limit or throttle hot keys at producer**
2. **Pause consumption** for that partition temporarily
3. **Spin up parallel topic** for hot traffic (if possible)
4. **Increase DB pool temporarily** (last resort)
5. **Alert stakeholders** — this will not auto-heal

Key phrase:

> “This is not something autoscaling will fix.”

#### Partitioning

* Redesign partition key to avoid skew
* Use composite keys (`userId + shard`)
* Increase partition count **before** traffic grows

* Detect hot partitions via metrics

* Backpressure instead of retries

* Split hot paths into **separate topics**

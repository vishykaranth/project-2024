## 🧠 What actually affects throughput?

### 1️⃣ Message size (huge factor)

* **1 KB messages** → millions/sec
* **100 KB messages** → tens of thousands/sec
* **1 MB messages** → much lower

Kafka is optimized for **many small messages**, not giant blobs.

---

### 2️⃣ Partitions

Throughput scales with partitions.

Rule of thumb:

> **1 partition ≈ 10–100 MB/sec**, depending on hardware

So:

* 10 partitions → ~100–1000 MB/sec
* 100 partitions → very high parallelism

---

### 3️⃣ Replication factor

* Replication = 3 → ~⅓ write throughput vs no replication
* Still worth it for durability

---

### 4️⃣ Acks & durability

* `acks=0` → 🔥 fastest, risky
* `acks=1` → common
* `acks=all` → slower, safest

## 🧩 High-level difference

**Kafka = event streaming & messaging**
**Temporal = workflow orchestration & durable execution**

---

## 🔁 Kafka (Apache Kafka)

### What it is

A **distributed event streaming platform**.

### What it’s great at

* Moving **huge volumes of events** reliably
* Decoupling producers and consumers
* Event-driven architectures

### Core ideas

* **Topics** → append-only logs
* **Producers** write events
* **Consumers** read events
* Kafka stores events for a configurable time

### Typical use cases

* Event streaming (user clicks, logs, metrics)
* Data pipelines (CDC, ETL)
* Pub/sub between services
* Replaying past events

### What Kafka does *not* do well

* Business workflows
* Retries with complex logic
* Long-running processes
* State transitions with guarantees

Kafka just delivers messages. **What happens next is your problem.**

---

## ⏳ Temporal

### What it is

A **workflow orchestration engine** for **reliable, long-running business logic**.

### What it’s great at

* Orchestrating **multi-step workflows**
* Handling **retries, timeouts, backoff**
* Surviving crashes, restarts, deployments
* Managing **state** automatically

### Core ideas

* **Workflows** = code that describes steps
* **Activities** = units of work (API calls, DB ops)
* Temporal **persists workflow state**
* Deterministic execution

### Typical use cases

* Order fulfillment workflows
* Payment + refund flows
* User onboarding journeys
* Sagas / distributed transactions
* Human-in-the-loop processes (wait days/weeks)

### What Temporal does *not* do well

* High-throughput event streaming
* Analytics pipelines
* Broadcasting millions of events

## 🤝 How they often work together

A very common pattern:

1. **Kafka event arrives** (e.g., `OrderCreated`)
2. Kafka consumer **starts a Temporal workflow**
3. Temporal handles:

    * Calling services
    * Waiting
    * Retrying
    * Compensating on failure
4. Workflow emits completion events back to Kafka

Kafka = **event highway**
Temporal = **traffic controller**

## ⚠️ Common mistake

> “Can Kafka replace Temporal?”

Usually **no**.

You *can* build workflows on Kafka with:

* State machines
* Retry topics
* Dead-letter queues
* Timers
* Idempotency logic

…but you’ll basically be **rebuilding Temporal badly** 😅

## 📁 Kafka log segment directory

Each **partition** on disk looks like this:

```
00000000000000000000.log
00000000000000000000.index
00000000000000000000.timeindex
```

All three files together represent **one log segment**.

---

## 1️⃣ `*.log` — the actual data

### What it is

* The **real Kafka data**
* Append-only file
* Contains serialized records (messages)

### What’s inside

```
[offset][timestamp][key][value][headers]
```

### Key points

* Sequential writes → very fast
* Never modified (only appended)
* Deleted or compacted during retention

👉 If this file is gone, **your data is gone**.

---

## 2️⃣ `*.index` — offset index

### What it is

* Maps **logical offsets → physical file positions** in the `.log` file

### Why it exists

Kafka doesn’t scan the `.log` file linearly for every fetch.

Instead:

```
offset 10500 → byte position 982,312 in .log
```

### Important details

* **Sparse index** (not every offset)
* Binary file
* Rebuilt automatically if deleted

👉 Think of it as the **table of contents by offset**.

---

## 3️⃣ `*.timeindex` — timestamp index

### What it is

* Maps **timestamps → offsets**

### Why it exists

Supports:

* Fetch by timestamp
* `kafka-consumer-groups.sh --reset-offsets --to-datetime`
* Time-based retention

Example:

```
timestamp 1700000000000 → offset 10500
```

### Important details

* Also sparse
* One entry per N messages
* Can be rebuilt

👉 Think of it as **table of contents by time**.

---

## 🔗 How they work together (real flow)

### Consumer asks:

> “Give me messages from offset 10500”

Kafka:

1. Looks up `10500` in `.index`
2. Gets byte position in `.log`
3. Reads sequentially from there

---

### Consumer asks:

> “Start from messages at 10:30 AM”

Kafka:

1. Looks up timestamp in `.timeindex`
2. Gets closest offset
3. Uses `.index` to jump into `.log`

---

## 🧠 Why not just one index?

Because:

* Offset-based reads are most common
* Time-based seeks are occasional
* Separate indexes = smaller, faster, cache-friendly

---

## ⚠️ What if index files are deleted?

* Kafka **rebuilds them** from `.log`
* Slower startup for that partition
* No data loss

But:

* Deleting `.log` = 💀

---

## 🧪 Quick analogy

| Kafka file   | Analogy                    |
| ------------ | -------------------------- |
| `.log`       | Book content               |
| `.index`     | Page number index          |
| `.timeindex` | Timeline / chapter by date |

* `*.timeindex` → timestamp → offset
* Index files are **optimizations**, `.log` is **truth**

## 1️⃣ Kafka storage internals (log segments & indexes)

**Model answer:**

> Kafka stores data in append-only log segments per partition. Each segment has a `.log` file with the actual records, a sparse `.index` mapping offsets to file positions, and a `.timeindex` mapping timestamps to offsets. Segments are rolled based on size or time. Index files are optimizations and can be rebuilt; the `.log` file is the source of truth.

---

## 2️⃣ Partitioning strategy & ordering guarantees

**Model answer:**

> Kafka guarantees ordering only within a partition. To guarantee ordering for a specific entity like a user or account, all related events must go to the same partition using a consistent key. Poor partitioning can cause hot partitions and limit throughput.

---

## 3️⃣ Consumer groups & rebalancing

**Model answer:**

> A consumer group allows parallel consumption where each partition is assigned to only one consumer. Rebalancing occurs when consumers join, leave, or fail, temporarily stopping consumption. Cooperative rebalancing minimizes disruption by incrementally reassigning partitions instead of stopping the entire group.

---

## 4️⃣ Offset management & delivery semantics

**Model answer:**

> Kafka supports at-most-once, at-least-once, and exactly-once semantics. Offsets are committed to Kafka’s internal `__consumer_offsets` topic. At-least-once is achieved by committing offsets after processing. Consumers must be idempotent to handle reprocessing.

---

## 5️⃣ Exactly-once semantics (EOS)

**Model answer:**

> Exactly-once in Kafka means a record is processed once within Kafka and written once to an output topic. It’s achieved using idempotent producers, transactions, producer IDs, and sequence numbers. This does not automatically guarantee exactly-once when interacting with external systems like databases.

---

## 6️⃣ Replication, ISR & leader election

**Model answer:**

> Each partition has a leader and followers. The ISR is the set of replicas that are fully caught up. Writes with `acks=all` are only successful if acknowledged by all ISR replicas. If the leader fails, a new leader is elected from the ISR to prevent data loss.

---

## 7️⃣ Performance tuning & throughput optimization

**Model answer:**

> Kafka achieves high throughput through batching, compression, sequential disk writes, and zero-copy transfers. Throughput can be improved by tuning `batch.size`, `linger.ms`, and compression, increasing partitions, and ensuring consumers are not the bottleneck.

---

## 8️⃣ Retention, compaction & cleanup policies

**Model answer:**

> Kafka supports time-based and size-based retention. Log compaction retains only the latest record per key, enabling Kafka to behave like a durable key-value store. Tombstones are used to represent deletions during compaction.

---

## 9️⃣ Failure scenarios & recovery

**Model answer:**

> Kafka is resilient to broker failures through replication and leader election. If a broker fails, partitions are reassigned and leaders are elected from the ISR. Data loss can occur if replication is misconfigured or unclean leader election is enabled.

---

## 🔟 Kafka architecture & ecosystem

**Model answer:**

> Kafka consists of brokers, a controller, and metadata management via ZooKeeper or KRaft. The ecosystem includes Kafka Connect for integrations, Kafka Streams for stream processing, and Schema Registry for schema evolution. Kafka should not be used for low-latency request-response workloads.

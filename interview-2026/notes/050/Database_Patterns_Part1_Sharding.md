```
â”Œâ”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”
â”‚              Shard Key Selection Criteria                    â”‚
â””â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”˜

1. High Cardinality
   - Many unique values
   - Avoids hot spots
   Example: User ID (good) vs Status (bad - only few values)

2. Even Distribution
   - Data spread evenly across shards
   - No skew
   Example: Hash of User ID (good) vs Timestamp (bad - time-based skew)

3. Query Patterns
   - Most queries include shard key
   - Minimizes cross-shard queries
   Example: User ID in user queries (good)

4. Immutability
   - Shard key doesn't change
   - Avoids data migration
   Example: User ID (good) vs Email (bad - can change)

5. Low Correlation
   - Independent of other attributes
   - Prevents related data in same shard
   Example: Random UUID (good) vs Sequential ID (bad)
```

## 9. Best Practices

### Sharding Best Practices
```
â”Œâ”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”
â”‚              Sharding Best Practices                        â”‚
â””â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”˜

1. Start Simple
   - Begin with single database
   - Monitor performance metrics
   - Shard only when necessary

2. Choose Right Shard Key
   - High cardinality
   - Even distribution
   - Matches query patterns

3. Plan for Growth
   - Design for 2-3x current size
   - Plan rebalancing strategy
   - Consider future requirements

4. Monitor Distribution
   - Track shard sizes
   - Monitor query patterns
   - Detect hot spots early

5. Handle Failures
   - Replicate each shard
   - Implement failover
   - Backup strategies

6. Minimize Cross-Shard Queries
   - Design queries with shard key
   - Denormalize when needed
   - Use materialized views

7. Test Thoroughly
   - Load testing
   - Failure scenarios
   - Rebalancing procedures
```

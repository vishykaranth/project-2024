# 🐦 Practical Exercise: Design Twitter/X at Scale

**Time Estimate:** 4-5 hours  
**Difficulty:** Principal Engineer Level

---

## 📋 Problem Statement

Design a global social media platform similar to Twitter/X that supports:
- **300M+ daily active users**
- **500M tweets per day**
- **Real-time timeline updates**
- **Search across billions of tweets**
- **Notifications and trending topics**

---

## 🎯 Functional Requirements

### Core Features
1. **Post tweets** (280 characters, with media)
2. **Follow/unfollow users**
3. **View personalized timeline** (home feed)
4. **View user profile** (tweets, followers, following)
5. **Search tweets** (keywords, hashtags, users)
6. **Like, retweet, reply** to tweets
7. **Real-time notifications**
8. **Trending topics/hashtags**
9. **Direct messaging** (optional for this exercise)

### User Actions per Day (Estimates)
- Average user follows: 200 people
- Average user posts: 2 tweets/day
- Average user reads: 100 tweets/day
- Read:Write ratio: ~100:1

## 📊 Non-Functional Requirements

### Scale
- **DAU:** 300M users
- **Tweets:** 500M/day = ~6,000 tweets/sec average, **12,000 tweets/sec** peak
- **Timeline reads:** 30B/day = ~350,000 reads/sec average, **700,000 reads/sec** peak
- **Search queries:** 500M/day = ~6,000 queries/sec

### Performance
- **Tweet posting:** < 200ms p99
- **Timeline loading:** < 500ms p99
- **Search results:** < 1s p99
- **Notifications:** < 3 seconds delivery

### Availability
- **99.99% uptime** (52 minutes downtime/year)
- Multi-region deployment
- Graceful degradation under high load

### Data
- **Tweet size:** ~500 bytes average (with metadata)
- **Media:** 30% of tweets have images/videos
- **Data retention:** Indefinite (all tweets searchable)
- **Storage growth:** ~90TB/year (tweets only)

### 1. Tweet Ingestion Service

**Responsibilities:**
- Validate and sanitize tweet content
- Generate unique tweet ID (Snowflake algorithm)
- Store tweet in database
- Publish event to message queue
- Handle media uploads

**API Design:**
```
POST /api/v1/tweets
{
  "content": "Hello World! #FirstTweet",
  "media_ids": ["media_123", "media_456"],
  "reply_to": null
}

Response:
{
  "tweet_id": "1234567890",
  "created_at": "2024-02-10T10:30:00Z",
  "user_id": "user_123",
  "content": "Hello World! #FirstTweet",
  "media_urls": [...],
  "metrics": {
    "likes": 0,
    "retweets": 0,
    "replies": 0
  }
}
```

**Database Schema (Tweet DB - Cassandra/DynamoDB):**
```sql
tweets {
  tweet_id: BIGINT (PRIMARY KEY, Snowflake ID)
  user_id: BIGINT (INDEXED)
  content: TEXT
  created_at: TIMESTAMP
  media_ids: LIST<STRING>
  reply_to: BIGINT (nullable)
  retweet_of: BIGINT (nullable)
  hashtags: SET<STRING>
  mentions: SET<BIGINT>
  metrics: {
    likes: COUNTER
    retweets: COUNTER
    replies: COUNTER
    views: COUNTER
  }
}

// Partition by user_id for user timeline queries
user_tweets {
  user_id: BIGINT (PARTITION KEY)
  created_at: TIMESTAMP (CLUSTERING KEY, DESC)
  tweet_id: BIGINT
}
```

**Optimizations:**
- **Snowflake ID:** Distributed ID generation (timestamp + machine ID + sequence)
- **Write-heavy optimization:** Use Cassandra for high write throughput
- **Media handling:** Offload to object storage (S3), return presigned URLs
- **Async processing:** Queue tweet for fanout after quick DB write

### 2. Timeline Generation Service

This is the **most complex** part! Two main approaches:

#### **Approach A: Fanout-on-Write (Push Model)**

**How it works:**
1. When user posts tweet, push it to all followers' timelines
2. Pre-compute timelines and store in cache
3. Timeline read is just a cache lookup

**Pros:**
- Fast timeline reads (just cache lookup)
- Predictable read performance

**Cons:**
- Expensive for users with millions of followers (celebrities)
- High write amplification (1 tweet → millions of writes)
- Hotspot problems

**Implementation:**
```python
# When tweet is posted
def fanout_tweet(tweet):
    followers = get_followers(tweet.user_id)
    
    # For each follower, insert tweet into their timeline cache
    for follower_id in followers:
        timeline_cache.zadd(
            f"timeline:{follower_id}",
            {tweet.tweet_id: tweet.timestamp}
        )
        # Keep only top 1000 tweets
        timeline_cache.zremrangebyrank(
            f"timeline:{follower_id}", 0, -1001
        )
```

**Data Structure (Redis Sorted Set):**
```
Key: "timeline:user_123"
Score: timestamp (for ordering)
Value: tweet_id

ZADD timeline:user_123 1707562800 tweet_456
ZADD timeline:user_123 1707562900 tweet_789
ZREVRANGE timeline:user_123 0 99  // Get top 100 tweets
```

#### **Approach B: Fanout-on-Read (Pull Model)**

**How it works:**
1. Store tweets only in tweet DB
2. When user requests timeline, fetch latest tweets from all followed users
3. Merge and rank in real-time

**Pros:**
- No write amplification
- Works well for celebrity accounts
- Always fresh data

**Cons:**
- Slow timeline reads (query N users' tweets)
- High read load on database

**Implementation:**
```python
def get_timeline(user_id, page_size=100):
    following = get_following(user_id)  # e.g., 200 users
    
    # Fetch recent tweets from each followed user
    recent_tweets = []
    for followed_id in following:
        tweets = db.query(
            "SELECT * FROM user_tweets 
             WHERE user_id = ? 
             ORDER BY created_at DESC 
             LIMIT 20",
            followed_id
        )
        recent_tweets.extend(tweets)
    
    # Merge and sort by timestamp
    timeline = sorted(recent_tweets, 
                     key=lambda t: t.created_at, 
                     reverse=True)
    
    return timeline[:page_size]
```

#### **Hybrid Approach (RECOMMENDED for Twitter scale)**

Combine both strategies:

1. **Fanout-on-write** for normal users (< 10K followers)
2. **Fanout-on-read** for celebrities (> 10K followers)
3. Merge both at read time

**Implementation:**
```python
def get_timeline_hybrid(user_id, page_size=100):
    # Get pre-computed timeline from cache (fanout-on-write)
    cached_tweets = timeline_cache.zrevrange(
        f"timeline:{user_id}", 0, page_size
    )
    
    # Get tweets from celebrities user follows (fanout-on-read)
    celebrity_ids = get_celebrity_following(user_id)
    celebrity_tweets = fetch_recent_tweets(celebrity_ids, limit=50)
    
    # Merge and rank
    all_tweets = merge_sorted([cached_tweets, celebrity_tweets])
    
    return all_tweets[:page_size]
```

**Timeline Cache Schema:**
```
Redis Sorted Set per user:
- Key: "timeline:{user_id}"
- Score: tweet timestamp
- Value: tweet_id
- TTL: 7 days
- Size limit: 1000 most recent tweets

For each tweet_id, store full tweet data:
- Key: "tweet:{tweet_id}"
- Value: JSON blob with all tweet data
- TTL: 7 days
```

### 3. Search Service

**Requirements:**
- Search across billions of tweets
- Support keyword, hashtag, user search
- Real-time indexing (new tweets searchable within seconds)
- Ranking by relevance and recency

**Technology Choice: Elasticsearch**

**Index Schema:**
```json
{
  "mappings": {
    "properties": {
      "tweet_id": { "type": "keyword" },
      "user_id": { "type": "keyword" },
      "username": { "type": "keyword" },
      "content": { 
        "type": "text", 
        "analyzer": "standard",
        "fields": {
          "keyword": { "type": "keyword" }
        }
      },
      "hashtags": { "type": "keyword" },
      "mentions": { "type": "keyword" },
      "created_at": { "type": "date" },
      "metrics": {
        "properties": {
          "likes": { "type": "integer" },
          "retweets": { "type": "integer" },
          "replies": { "type": "integer" }
        }
      },
      "location": { "type": "geo_point" }
    }
  }
}
```

**Search Query Example:**
```json
GET /tweets/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "content": "system design" } }
      ],
      "filter": [
        { "range": { "created_at": { "gte": "now-7d" } } },
        { "term": { "hashtags": "engineering" } }
      ]
    }
  },
  "sort": [
    { "_score": { "order": "desc" } },
    { "created_at": { "order": "desc" } }
  ],
  "size": 20
}
```

**Indexing Pipeline:**
```
Tweet Published → Kafka → Search Indexer → Elasticsearch

Indexer:
1. Consume from Kafka topic
2. Extract searchable fields
3. Bulk index to Elasticsearch (batch 1000s)
4. Handle failures with retry queue
```

**Optimizations:**
- **Sharding:** Partition by date (monthly indices)
- **Replicas:** 3 replicas for read scalability
- **Hot-Warm-Cold architecture:**
  - Hot: Last 7 days (SSD, 3 replicas)
  - Warm: 8-30 days (SSD, 2 replicas)
  - Cold: 30+ days (HDD, 1 replica)
- **Index aliases** for seamless shard rotation

### 5. Notification Service

**Types:**
- New follower
- Tweet liked/retweeted
- Reply to your tweet
- Mention in tweet
- Direct message

**Architecture:**
```
Event (like/retweet/mention) 
    → Kafka Topic 
    → Notification Service
    → Check user preferences
    → Store in Notification DB
    → Push to device (FCM/APNS)
    → WebSocket for web clients
```

**Database (Cassandra):**
```sql
notifications {
  user_id: BIGINT (PARTITION KEY)
  created_at: TIMESTAMP (CLUSTERING KEY, DESC)
  notification_id: UUID
  type: VARCHAR  // 'like', 'retweet', 'follow', 'mention'
  actor_user_id: BIGINT
  tweet_id: BIGINT (nullable)
  read: BOOLEAN
  metadata: JSON
}
```

**Push Notifications:**
- **Batch processing:** Group notifications (10 likes → "10 people liked your tweet")
- **Rate limiting:** Don't spam users (max 5 push/hour)
- **Priority queue:** High priority for mentions, low for likes
- **Fallback:** If push fails, show in-app notification

**Real-time Delivery:**
- **WebSocket connections** for active users
- Presence service tracks online users
- Fan-out notification to all user's devices

### 6. Analytics & Trending Service

**Trending Hashtags:**

**Approach: Count-Min Sketch + Sliding Window**

```python
# Real-time hashtag counter
class TrendingTracker:
    def __init__(self):
        self.windows = {
            '1h': CountMinSketch(),
            '6h': CountMinSketch(),
            '24h': CountMinSketch()
        }
    
    def process_tweet(self, tweet):
        for hashtag in tweet.hashtags:
            for window in self.windows.values():
                window.increment(hashtag)
    
    def get_trending(self, window='1h', top_n=10):
        return self.windows[window].top_k(top_n)
```

**Batch Job (Hourly):**
```
1. Aggregate hashtag counts from last hour
2. Score by: (count / expected_count) * recency_factor
3. Store top 100 in Redis
4. Serve from cache
```

**Tweet Analytics:**
- **Real-time:** Kafka Streams for view counts, impressions
- **Batch:** Spark jobs for historical analytics
- **Storage:** Time-series DB (InfluxDB) for metrics

```

   - Validate content
   - Generate Snowflake ID
   - Write to Cassandra
   - Upload media to S3

   - Timeline Service → Fanout to followers' timelines

```
1. Client → API Gateway → Timeline Service

   - Get cached timeline from Redis (fanout-on-write)

## ⚡ Performance Optimizations

### Caching Strategy

**Multi-layer cache:**
```
1. Browser cache (static assets)
2. CDN (images, videos, profile pics)
3. Redis (timelines, user objects, trending)
4. Application cache (hot users, frequent queries)
```

**Cache invalidation:**
- Tweet updates: Rare (only metrics change) → Cache for 5 mins
- User profiles: Invalidate on update
- Timelines: TTL 7 days, LRU eviction
- Trending: Refresh every 15 minutes

### Database Optimizations

**Cassandra:**
- Wide partition design for user timelines
- Compaction strategy for write-heavy workload
- Bloom filters for faster lookups

**Redis:**
- Redis Cluster for horizontal scaling
- Separate clusters for different use cases
- Pipelining for bulk operations

**PostgreSQL:**
- Read replicas for user reads
- Connection pooling
- Prepared statements

### API Optimizations

- **Pagination:** Cursor-based (not offset)
- **Field selection:** `?fields=id,content,user` (partial responses)
- **Batch endpoints:** Get multiple tweets in one call
- **HTTP/2:** Multiplexing for parallel requests
- **gRPC:** For internal service communication

**Regions:** US-East, US-West, EU-West, Asia-Pacific

```

    │          │          │
    └──────────┴──────────┘
         Kafka Replication
```

**Data Locality:**
- User profiles: Replicated globally
- Tweets: Written to nearest region, replicated async
- Timeline cache: Regional
- Media: CDN edge locations

**Conflict Resolution:**
- Last-write-wins (LWW) for tweets (unlikely conflicts)
- CRDTs for counters (likes, retweets)
- Vector clocks for user updates
